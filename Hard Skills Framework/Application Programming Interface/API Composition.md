## Что такое паттерн композиции API?

Шаблон композиции API — это подход к проектированию в [архитектуре микросервисов](https://www.geeksforgeeks.org/system-design/microservices/) который позволяет разработчикам объединять ответы от нескольких микросервисов в единую конечную точку API. В типичной конфигурации микросервисов каждый сервис отвечает за определенную бизнес-возможность и предоставляет собственный API. Однако для выполнения запроса клиентам часто требуются данные от нескольких сервисов, что приводит к усложнению системы и потенциальным проблемам с производительностью при выполнении нескольких сетевых вызовов.

- Шаблон композиции API решает эту проблему, выступая в качестве промежуточного уровня, который организует вызовы соответствующих микросервисов, собирает их ответы и предоставляет клиенту единый ответ.
- Этот шаблон можно реализовать с помощью специального шлюза API или сервиса композиции, который асинхронно извлекает данные, гарантируя, что клиент получит всю необходимую информацию за один вызов.

## Ключевые понятия шаблона композиции API

Вот ключевые понятия шаблона композиции API в микросервисах:

1. ****Агрегация:**** Основная функция шаблона композиции API — агрегирование ответов от нескольких микросервисов. Вместо того чтобы клиент делал несколько запросов к отдельным сервисам, уровень композиции объединяет данные в один ответ.
2. ****Оркестрация:**** Уровень композиции управляет вызовами базовых микросервисов. Он регулирует последовательность вызовов сервисов, обрабатывает все зависимости и определяет, как объединять данные, полученные от каждого сервиса.
3. ****Единая точка входа:**** клиенты взаимодействуют с одной конечной точкой API, а не с несколькими. Это упрощает логику на стороне клиента, поскольку ему нужно обработать только один вызов API.
4. ****Разделение:**** Шаблон композиции API отделяет клиента от архитектуры микросервисов. Клиенты не взаимодействуют напрямую с отдельными сервисами, а взаимодействуют только с уровнем композиции, который может развиваться независимо.
5. ****Преобразование данных:**** В процессе агрегации уровню композиции может потребоваться преобразовать данные из различных микросервисов в единый формат, подходящий для клиента. Это может включать в себя фильтрацию, сопоставление или объединение полей данных.
6. ****Обработка ошибок:**** Композиционный уровень отвечает за обработку ошибок, которые могут возникать при вызове отдельных сервисов. Он может реализовывать такие стратегии, как резервные механизмы, повторные попытки или возврат частичных результатов в случае сбоя некоторых сервисов.
7. [****Кэширование****](https://www.geeksforgeeks.org/system-design/caching-system-design-concept-for-beginners/)****:**** Для повышения производительности и снижения нагрузки на отдельные микросервисы композиционный уровень может реализовывать стратегии кэширования. Ответы могут кэшироваться на композиционном уровне для быстрого выполнения последующих запросов.

## Когда следует использовать паттерн композиции API?

Паттерн композиции API особенно полезен в некоторых сценариях использования микросервисной архитектуры. Вот основные ситуации, в которых стоит рассмотреть возможность применения этого паттерна:

- ****Потребности конкретного клиента:**** Когда разным клиентам требуются различные комбинации данных из нескольких микросервисов, шаблон композиции API позволяет формировать индивидуальные ответы в соответствии с конкретными требованиями клиента, не заставляя его обрабатывать несколько вызовов сервисов.
- ****Сложное извлечение данных:****В случаях, когда для получения данных требуется взаимодействие с несколькими микросервисами, например когда для одного клиентского запроса нужны данные из нескольких сервисов (например, профили пользователей, заказы и платежные сервисы), использование композиции API упрощает процесс, объединяя данные в один ответ.
- ****Снижение сложности клиентского приложения:**** Если клиентское приложение становится слишком сложным из-за большого количества вызовов сервисов, внедрение уровня композиции API может упростить взаимодействие за счет предоставления унифицированного интерфейса.
- ****Оптимизация производительности:**** Если производительность критически важна, а минимизация сетевых задержек является приоритетом, то шаблон композиции API сокращает количество сетевых запросов за счет объединения нескольких вызовов сервисов в один, что улучшает общее время отклика для клиентов.
- ****Эволюция микросервисов:**** Если предполагается, что ваша архитектура микросервисов будет часто меняться, то с помощью шаблона композиции API можно отделить клиента от отдельных сервисов. Это позволит вносить изменения, заменять или масштабировать внутренние сервисы независимо друг от друга, не затрагивая клиентский интерфейс.

## Как работает композиция API?

Шаблон композиции API в микросервисах работает как промежуточный уровень, который собирает данные из нескольких микросервисов и объединяет их в единый ответ для клиента. Вот как это обычно работает:

- ****Шаг 1. Запрос от клиента:**** Клиент отправляет запрос на единую конечную точку API, предоставляемую уровнем композиции. В запросе могут указываться необходимые данные, в том числе различные идентификаторы ресурсов.
- ****Шаг 2. Анализ запроса:**** Уровень композиции получает запрос и анализирует его, чтобы понять, с какими микросервисами нужно связаться и какие данные требуются от каждого из них.
- ****Шаг 3. Вызовы сервисов:**** Уровень композиции отправляет запросы соответствующим микросервисам. Эти вызовы могут выполняться параллельно или последовательно, в зависимости от зависимостей между службами и дизайна композиционного уровня. Если некоторые службы зависят от результатов других, уровень композиции может дождаться этих ответов, прежде чем выполнять дальнейшие вызовы.
- ****Шаг 4. Агрегирование данных:**** По мере того, как композиционный уровень получает ответы от микросервисов, он агрегирует данные. Это может включать объединение данных из различных сервисов, преобразование их в согласованный формат и отфильтровывание любой ненужной информации. Если какие-либо сервисы возвращают ошибки, уровень композиции может корректно их обработать, возможно, предоставив резервные данные или вернув частичные результаты.
- ****Шаг 5. Форматирование ответа:**** После сбора всех необходимых данных уровень композиции форматирует агрегированный ответ в соответствии с требованиями клиента. Это может включать в себя структурирование данных определенным образом или добавление метаданных.
- ****Шаг 6. Отправка ответа:**** Уровень композиции отправляет клиенту единый ответ. Этот ответ содержит всю запрошенную информацию от различных микросервисов, что упрощает взаимодействие с клиентом.
- ****Шаг 7:**** [****Кэширование****](https://www.geeksforgeeks.org/system-design/caching-system-design-concept-for-beginners/)****:**** Для оптимизации производительности на уровне композиции могут быть реализованы механизмы кэширования. Часто запрашиваемые данные можно кэшировать, чтобы сократить количество повторных обращений к микросервисам и тем самым уменьшить время отклика.
- ****Шаг 8: Безопасность и мониторинг:**** На протяжении всего этого процесса уровень композиции может обеспечивать соблюдение протоколов безопасности (таких как аутентификация и авторизация) и отслеживать запросы на основе показателей производительности, что позволяет централизованно контролировать эти аспекты.

## Преимущества композиции API в микросервисах

Шаблон композиции API дает ряд существенных преимуществ при использовании в архитектуре микросервисов. Вот некоторые из них:

1. ****Упрощенное взаимодействие с клиентами:**** Клиенты взаимодействуют с одной конечной точкой API, а не с несколькими микросервисами. Это упрощает логику на стороне клиента и облегчает разработчикам создание и поддержку клиентских приложений.
2. ****Снижение****[****задержки****](https://www.geeksforgeeks.org/system-design/latency-in-system-design/)****:**** Благодаря объединению нескольких вызовов сервисов в один запрос, API-композиция минимизирует количество сетевых обращений, необходимых для сбора данных. Это позволяет сократить время отклика и повысить производительность для конечного пользователя.
3. ****Разделение сервисов:**** Композиционный уровень выступает в роли посредника, позволяя микросервисам развиваться независимо друг от друга. Клиенты не зависят от изменений в отдельных сервисах, что упрощает их модификацию или замену без ущерба для пользовательского опыта.
4. ****Гибкая агрегация данных:**** Этот шаблон обеспечивает гибкое извлечение и агрегацию данных, позволяя клиентам получать персонализированные ответы, объединяющие информацию из различных сервисов. Это особенно полезно в тех случаях, когда разным клиентам нужны разные комбинации данных.
5. ****Централизованная обработка ошибок:**** Композиционный уровень может реализовать централизованную обработку ошибок, что позволит корректно реагировать на сбои.В случае сбоя одного или нескольких микросервисов уровень может возвращать частичные результаты или использовать резервные механизмы, повышая общую отказоустойчивость системы.
6. ****Преобразование и форматирование данных:****Уровень композиции может преобразовывать данные из различных микросервисов в единый формат, чтобы клиенты получали информацию в нужной им структуре. Это избавляет клиентов от необходимости самостоятельно преобразовывать данные.

## ****Проблемы при реализации композиции API****

Несмотря на то, что паттерн композиции API дает множество преимуществ при использовании микросервисных архитектур, он также сопряжен с рядом проблем, которые организациям необходимо решить при его внедрении. Вот некоторые из них:

1. ****Повышенная сложность:**** Внедрение уровня композиции может усложнить архитектуру. Разработчикам приходится управлять логикой оркестрации, которая может быть довольно сложной, особенно при работе с несколькими сервисами и преобразовании данных.
2. ****Снижение производительности:**** Хотя композиция API может сократить количество клиентских вызовов, сам уровень композиции может приводить к задержкам, если его не оптимизировать. Процессы оркестрации и агрегации могут приводить к увеличению времени отклика, особенно если они связаны с синхронными вызовами нескольких сервисов.
3. ****Распространение и обработка ошибок:**** Управление ошибками в работе нескольких микросервисов может быть непростой задачей. Если один из сервисов выходит из строя, бывает сложно понять, как обработать эту ошибку и нужно ли возвращать частичные результаты. Для корректного снижения качества обслуживания необходимо применять соответствующие стратегии.
4. ****Проблемы с согласованностью данных:**** При агрегировании данных из нескольких источников обеспечить их согласованность может быть непросто. Разные микросервисы могут находиться в разных состояниях, и синхронизировать эти данные в режиме реального времени может быть сложно.
5. ****Проблемы с управлением версиями:**** По мере развития микросервисов обеспечение совместимости между различными версиями сервисов может усложнить логику их компоновки. Уровень компоновки должен корректно работать с несколькими версиями одного и того же сервиса.
6. ****Сложности с обеспечением безопасности:**** Централизация безопасности на уровне компоновки может упростить аутентификацию и авторизацию, но при этом требует тщательной проработки, чтобы уязвимости на уровне компоновки не ставили под угрозу безопасность базовых сервисов.

## Шаги по разработке эффективного уровня композиции API

Разработка эффективного уровня композиции API имеет решающее значение для успешной реализации шаблона композиции API в архитектуре микросервисов. Вот основные этапы процесса разработки:

- ****Шаг 1. Определите требования:****
    - ****Изучите потребности клиентов:**** Соберите требования от клиентских приложений, чтобы определить, какие данные им нужны и как они должны быть структурированы. Это поможет определить конечные точки и ответы для уровня композиции.
    - ****Определите сценарии использования:**** Опишите конкретные сценарии использования, которые должен поддерживать уровень композиции API, включая различные комбинации микросервисов, которые необходимо объединить.
- ****Шаг 2. Создайте архитектуру уровня композиции:****
    - ****Выберите архитектуру:**** Решите, будет ли композиционный уровень отдельным сервисом, шлюзом API или частью существующего сервиса. У каждого варианта есть свои плюсы и минусы с точки зрения сложности, масштабируемости и обслуживания.
    - ****Спроектируйте конечные точки API:**** Определите конечные точки, которые будут доступны на композиционном уровне. Они должны быть спроектированы с учетом конкретных потребностей клиентских приложений и при необходимости объединять данные из нескольких микросервисов.
- ****Шаг 3. Внедрение логики оркестрации:****
    - ****Взаимодействие сервисов:**** Разработайте логику оркестрации, чтобы управлять взаимодействием уровня композиции с базовыми микросервисами. Определите, будут ли вызовы выполняться параллельно или последовательно, в зависимости от зависимостей.
    - ****Агрегация и преобразование данных:**** Реализуйте логику для агрегирования и преобразования данных из разных микросервисов в единый формат ответа. Убедитесь, что уровень композиции корректно обрабатывает различные структуры данных.
- ****Шаг 4. Грамотная обработка ошибок:****
    - ****Централизованная обработка ошибок:**** Разработайте механизмы обработки ошибок для устранения сбоев в работе микросервисов. Решите, как реагировать в случае сбоя одного или нескольких сервисов — рассмотрите возможность использования резервных ответов или возврата частичных данных.
    - ****Повторные попытки и тайм-ауты:**** Внедрите стратегии повторных попыток при неудачных вызовах сервисов и установите соответствующие тайм-ауты, чтобы избежать длительных задержек.
- ****Шаг 5. Оптимизация производительности:****
    - ****Асинхронная обработка:**** По возможности используйте асинхронные вызовы, чтобы минимизировать задержки, особенно при вызове нескольких микросервисов.
    - ****Кэширование:**** Внедряйте стратегии кэширования часто запрашиваемых данных, чтобы снизить нагрузку на микросервисы и сократить время отклика.
- ****Шаг 6. Обеспечьте безопасность****
    - ****Аутентификация и авторизация:**** Централизуйте систему безопасности на уровне композиции. Внедрите механизмы аутентификации для проверки клиентских запросов и авторизации, чтобы гарантировать, что у клиентов есть разрешение на доступ к определенным данным.
    - ****Шифрование данных:**** Обеспечьте шифрование конфиденциальных данных при их передаче между уровнем композиции и микросервисами.
- ****Шаг 7. Внедрите мониторинг и ведение журналов.****
    - ****Трассировка запросов:**** Внедрите трассировку запросов, чтобы отслеживать поток запросов, проходящих через уровень композиции и попадающих в микросервисы. Это поможет выявить узкие места в производительности и устранить неполадки.
    - ****Показатели производительности:**** Настройте мониторинг для сбора показателей производительности, частоты ошибок и особенностей использования, чтобы понять, как работает уровень композиции.
- ****Шаг 8. Планируйте управление версиями****
    - ****Контроль версий:**** Внедрите стратегию управления версиями для API уровня композиции, чтобы вносить изменения в микросервисы без ущерба для существующих клиентских приложений.
    - ****Обратная совместимость:**** По возможности обеспечьте обратную совместимость новых версий уровня композиции.
- ****Шаг 9. Тщательное тестирование****
    - ****Модульное тестирование:**** Напишите модульные тесты для логики оркестрации и процессов агрегирования данных, чтобы убедиться, что они работают должным образом.
    - ****Интеграционное тестирование:****Проведите интеграционные тесты, чтобы проверить взаимодействие между уровнем композиции и базовыми микросервисами, а также убедиться, что данные агрегируются и форматируются корректно.
- ****Шаг 10. Повторяйте и совершенствуйте****
    - ****Цикл обратной связи:**** Собирайте отзывы клиентов и разработчиков, используя композиционный слой, чтобы выявлять проблемные места и области, требующие улучшения.
    - ****Постоянное совершенствование:**** Регулярно анализируйте и дорабатывайте композиционный слой на основе показателей эффективности, отзывов клиентов и меняющихся требований.

## Технологии, Поддерживающие составление API

Несколько технологий и фреймворков могут помочь в реализации шаблона композиции API в микросервисных архитектурах. Ниже мы рассмотрим ключевые технологии, которые могут способствовать эффективной композиции API.

- ****Шлюзы API:****
    - ****Kong:**** Шлюз API с открытым исходным кодом, который предоставляет такие функции, как маршрутизация запросов, аутентификация и ограничение скорости, упрощая управление несколькими микросервисами.
    - ****AWS API Gateway:**** Полностью управляемый сервис, который позволяет разработчикам создавать, публиковать и масштабировать API, а также управлять ими. Он поддерживает агрегацию запросов, а также может выполнять аутентификацию и мониторинг.
    - ****Nginx:**** Популярный веб-сервер, который также может выступать в роли шлюза API, обеспечивая балансировку нагрузки и маршрутизацию запросов.
- ****Сервисные сетки:****
    - ****Istio:**** Сервисная сетка, обеспечивающая расширенную маршрутизацию, балансировку нагрузки и взаимодействие между сервисами. Она помогает управлять трафиком и обеспечивать безопасность в средах микросервисов.
    - ****Linkerd:****Легкая сервисная сетка, которая обеспечивает наблюдаемость, управление трафиком и безопасность, а также поддерживает композицию API за счет управления взаимодействием между сервисами.
- ****GraphQL:****
    - ****Apollo Server:**** Популярная реализация сервера GraphQL, которая позволяет объединять данные из нескольких источников в единую конечную точку GraphQL.
    - ****Hasura:**** Механизм GraphQL, который предоставляет данные в режиме реального времени и позволяет создавать API на основе существующих баз данных и сервисов.
- ****Бэкенд для фронтенда (BFF):****
    - ****Express.js:**** фреймворк для веб-приложений на Node.js, который можно использовать для создания уровня BFF, упрощающего объединение данных из нескольких микросервисов.
    - ****Spring Boot:**** фреймворк для создания Java-приложений, который может выступать в роли BFF, позволяя объединять данные из различных микросервисов.
- ****Брокеры сообщений:****
    - ****RabbitMQ:**** Брокер сообщений, поддерживающий асинхронную связь между микросервисами, что обеспечивает слабую связанность архитектуры и эффективную компоновку данных.
    - ****Apache Kafka:**** Распределенная платформа потоковой передачи данных, которую можно использовать для управления потоками данных в реальном времени между микросервисами, что упрощает компоновку данных с помощью событийно-ориентированной архитектуры.
- ****Библиотеки композиций API:****
    - ****Spring Cloud Gateway:**** Библиотека, предоставляющая API-шлюз на основе Spring, который позволяет маршрутизировать и фильтровать запросы, а также поддерживает композицию API.
    - ****Microgateway:**** Легковесные API-шлюзы с возможностями композиции, такие как Tyk и 3scale, которые можно интегрировать с микросервисами.

## Реальные примеры использования шаблона проектирования «Композиция API»

Шаблон композиции API широко используется в различных отраслях и сферах для эффективного управления взаимодействием между микросервисами. Вот несколько реальных примеров использования этого шаблона в организациях:

- ****Платформы электронной коммерции:**** Платформа электронной коммерции должна собирать данные для отображения сведений о товарах, отзывов пользователей, цен и уровня запасов. Композиционный уровень API собирает данные из нескольких микросервисов, таких как сервис товаров, сервис отзывов пользователей, сервис ценообразования и сервис управления запасами. Когда клиент запрашивает информацию о товаре, композиционный уровень извлекает и объединяет данные из этих сервисов, предоставляя полную информацию о товаре в одном ответе API.
- ****Приложения для социальных сетей:**** Приложению для социальных сетей необходимо собирать профили пользователей, публикации, комментарии и списки друзей для формирования ленты пользователя. Уровень композиции API взаимодействует с несколькими микросервисами — пользовательским сервисом, сервисом публикаций и сервисом комментариев — для формирования ленты пользователя. Это позволяет эффективно извлекать данные для отображения полной ленты без необходимости делать несколько запросов.
- ****Сервисы для путешествий и бронирования:**** Приложению для бронирования путешествий требуются данные от сервисов бронирования авиабилетов, отелей и автомобилей, чтобы предлагать пользователям пакетные предложения.Композиционный уровень API одновременно запрашивает данные у каждого сервиса, объединяя результаты, чтобы отобразить доступные рейсы, отели и пункты проката автомобилей в одном ответе, тем самым улучшая пользовательский опыт и сокращая задержки.
- ****Финансовые услуги:**** Банковское приложение должно предоставлять клиентам целостное представление об их счетах, транзакциях и инвестиционных портфелях. Композиционный уровень получает данные из различных микросервисов: сервиса счетов, сервиса транзакций и сервиса инвестиций. Это позволяет клиентам видеть свои балансы, последние транзакции и доходность портфеля в одном интерфейсе.
- ****Системы управления контентом (CMS):****CMS должна объединять контент из разных источников, включая статьи, изображения и видео. Уровень API-композиции объединяет данные из сервисов контента, мультимедиа и пользовательских сервисов, позволяя CMS предоставлять полный пакет контента в рамках одного запроса. Это упрощает разработку фронтенда, поскольку ответ формируется единообразно.

## Заключение

Подводя итог, можно сказать, что паттерн композиции API — это мощный подход к управлению данными в микросервисных архитектурах. Выступая в роли связующего звена, объединяющего информацию из нескольких сервисов, он упрощает взаимодействие с клиентами и повышает производительность. Несмотря на то, что этот паттерн сопряжен с определенными трудностями, такими как повышенная сложность и необходимость обработки ошибок, его преимущества значительно перевешивают недостатки. Организации могут использовать этот паттерн для создания эффективных, гибких и масштабируемых приложений, отвечающих потребностям пользователей. По мере роста популярности микросервисов внедрение паттерна композиции API будет играть ключевую роль в создании надежных и удобных в обслуживании систем.
