**Server-Sent Events (SSE)** — технология, позволяющая серверу отправлять поток событий клиенту по одностороннему соединению через HTTP. В отличие от WebSockets, коммуникация через SSE **однонаправленная**: сообщения доставляются от сервера к клиенту




Для приложения с **ограниченными ресурсами сервера** и необходимостью поддерживать **большое количество подключённых клиентов** Server-Sent Events — оптимальный выбор по следующим причинам:

1. **Однонаправленный поток данных** (сервер → клиент) — идеально подходит, если клиенты в основном получают обновления, а не отправляют данные.
2. **Долгоживущее HTTP-соединение** — минимизирует накладные расходы на постоянные переподключения (в отличие от HTTP Polling и Long Polling).
3. **Низкая нагрузка на сервер:**
   - использует один TCP-сокет на клиента;
   - нет «пустых» запросов (как в обычном Polling);
   - поддерживает HTTP/2 multiplexing (несколько потоков данных в одном соединении).
4. **Автоматическое переподключение** — браузер сам восстанавливает соединение при разрывах.
5. **Простота реализации** — использует стандартный EventSource API в браузере.
6. **Эффективная доставка обновлений** — события доставляются почти в реальном времени, с минимальной задержкой.

**Почему другие варианты хуже подходят:**

- **Long Polling:**
  - каждый клиент держит открытое соединение, потребляя ресурсы сервера;
  - сложность управления таймаутами и переподключениями;
  - высокая нагрузка при тысячах одновременных подключений;
  - задержка выше, чем у SSE и WebSockets.
- **HTTP Polling:**
  - клиент постоянно опрашивает сервер, даже если нет обновлений;
  - огромная нагрузка на сервер при большом числе клиентов;
  - высокая задержка (зависит от интервала опроса);
  - неэффективное использование трафика.
- **Webhooks:**
  - предназначены для асинхронной передачи данных между сервисами, а не для реального времени в клиент-серверном взаимодействии;
  - требуют настройки URL-эндпоинтов на стороне клиента;
  - не подходят для постоянного потока обновлений — работают по принципу «событие → уведомление».
- **SOAP:**
  - ориентирован на синхронный обмен сообщениями (request-response);
  - избыточен для асинхронных обновлений;
  - требует больше ресурсов (XML-сериализация, сложные заголовки);
  - не предназначен для реального времени и потоковой передачи данных.

**Вывод:** SSE — баланс между низкой нагрузкой на сервер, простотой реализации и достаточной скоростью доставки обновлений для большого числа клиентов. Подходит для сценариев вроде новостных лент, дашбордов, уведомлений.
