[[API]]

Гибридный API комбинирует подходы с однонаправленным и двунаправленным режимами взаимодействия в рамках единого интерфейса, позволяя выбирать стратегию в зависимости от контекста операции. Некоторые системы предлагают «синхронную обёртку» над асинхронными процессами через long-polling или Server-Sent Events (SSE), сохраняя простоту клиентского кода при фоновой обработке. Реализация требует чёткого документирования границ применения каждого режима, единой системы мониторинга статусов задач и продуманной стратегии обработки ошибок в обоих режимах работы.

Гибридный подход комбинирует сложности обеих моделей, велик риск «протечки абстракции», monitoring должен отслеживать и HTTP- latency, и queue depth с processing time, — усложняет observability, а inconsistent error handling (HTTP-статусы vs job failure reasons) повышает когнитивную нагрузку на клиентских разработчиков.

Гибридный подход выигрывает в сложных доменах с неоднородными требованиями к разным операциям. Лучшие юзкейсы: **e-commerce платформы** — синхронное получение каталога и цен (мгновенный рендер), асинхронное оформление заказа с проверкой складских остатков и платёжной системой; **SaaS с многоуровневой обработкой** — синхронный upload файла с immediate validation, асинхронная обработка (антивирус, thumbnail generation, OCR); **финансовые сервисы** — синхронная блокировка средств для гарантии, асинхронное клиринговое settlements; **enterprise workflow** — синхронный старт процесса с возвратом instance ID, асинхронное выполнение шагов с human-in-the-loop. Hybrid особенно эффективен при **постепенной миграции legacy-систем**: старые endpoint'ы остаются sync для обратной совместимости, новые функции добавляются как async, а unified API gateway маршрутизирует запросы, абстрагируя клиентов от внутренней сложности.





# Основные подходы к реализации

- GraphQL (Query/Mutation — синхронно, Subscription — асинхронно)
- gRPC (unary calls — синхронно, streaming — асинхронно)
- HTTP/2 (стандартный req/res + server push)
- WebSocket (можно реализовать паттерн запрос-ответ поверх полнодуплексного канала)
- REST (базово синхронный, но с расширениями вроде webhooks, callbacks, async processing)

# Гибридные (поддерживают оба режима)

### GraphQL
Язык запросов и среда выполнения для API, которая позволяет клиенту точно описывать структуру и объём необходимых данных. Вместо множества эндпоинтов сервер предоставляет одну точку входа, а клиент формирует запрос под конкретную задачу, избегая как недозагрузки, так и избыточной выборки.

В стандартном режиме операции Query и Mutation работают синхронно, возвращая полный ответ после обработки резолверами. Для реального времени GraphQL поддерживает подписки (Subscriptions), работающие поверх WebSocket или SSE и позволяющие серверу пушить изменения по мере их появления.

Технология популярна в сложных мобильных и веб-интерфейсах, где данные часто обновляются и зависят друг от друга. Единая точка входа упрощает версионирование и агрегацию данных из микросервисов.

GraphQL — это гибрид по своей природе: синхронные запросы-ответы для CRUD-операций и асинхронные подписки для потоковых данных. Это требует тщательного проектирования резолверов, чтобы избежать N+1 проблем и блокировок в синхронных вызовах при высокой нагрузке.

GraphQL предоставляет единую точку входа, где клиент точно описывает структуру необходимых данных в запросе, а сервер разрешает их через резолверы, поддерживая синхронные Query/Mutation и асинхронные Subscriptions.
```graphql
query GetUser { user(id: "123") { name, email, posts(limit: 5) { title } } }
subscription OnNewPost { postCreated(authorId: "123") { id, title, createdAt } }
```

### gRPC
Высокопроизводительный фреймворк удалённого вызова процедур, использующий HTTP/2 и бинарную сериализацию через Protocol Buffers. Помимо обычных unary-запросов, gRPC поддерживает потоковую передачу: сервер может отправлять последовательность сообщений, клиент может писать в поток, или обе стороны могут обмениваться данными непрерывно.

Строгая типизация контрактов и встроенная маршрутизация обеспечивают высокую скорость и надёжность. Двунаправленные стримы позволяют обрабатывать большие объёмы данных без ожидания полного ответа, экономя память и ускоряя конвейеры.

Технология стала стандартом для внутренней коммуникации микросервисов, работы с машинным обучением и высоконагруженными распределёнными системами. Экономия на заголовках и мультиплексирование снижают сетевые издержки.

gRPC — это гибридный фреймворк, который сочетает синхронные unary-вызовы для простых операций и асинхронные bidirectional-стримы для высоконагруженных конвейеров. Оба режима работают поверх одного HTTP/2 соединения, что упрощает сетевое взаимодействие, но требует осознанного выбора паттерна под задачу.

gRPC определяет строго типизированные сервисы через Protocol Buffers, генерируя клиентские и серверные заглушки, где вызовы методов автоматически сериализуются в бинарный формат и передаются поверх HTTP/2 с поддержкой потоков.
```protobuf
service OrderService {
  rpc CreateOrder (OrderRequest) returns (OrderResponse);
  rpc StreamStatus (StatusRequest) returns (stream StatusUpdate);
}
message OrderRequest { string user_id = 1; repeated Item items = 2; }
```

### HTTP/2 (стандартный req/res + server push)
Современная версия протокола, которая сохраняет семантику HTTP/1.1, но кардинально меняет транспортный слой. Мультиплексирование позволяет передавать множество запросов и ответов параллельно в рамках одного TCP-соединения, устраняя проблему блокировки очереди и head-of-line.

Сжатие заголовков (HPACK) и приоритизация потоков ускоряют загрузку сложных страниц. Базовый режим остаётся синхронным запрос-ответ, но протокол также поддерживает асинхронные механизмы вроде Server Push и flow control для балансировки нагрузки.

Инфраструктура легко адаптируется к HTTP/2 без изменений в прикладном коде. Обратная совместимость и прозрачное улучшение производительности сделали его обязательным стандартом для современных веб-сервисов.

HTTP/2 — это гибридный транспортный стандарт. Он по умолчанию синхронен на уровне приложения, но асинхронен на уровне сети благодаря мультиплексированию. Это позволяет серверу и клиенту обмениваться данными без блокировки канала, открывая возможности для более гибких и отзывчивых архитектур.

HTTP/2 сохраняет семантику запрос-ответ, но внедряет мультиплексирование потоков в одном TCP-соединении, сжатие заголовков и приоритизацию, позволяя браузеру и серверу параллельно обмениваться данными без блокировки очереди.
```http
# Одно TCP-соединение multiplexes несколько независимых stream'ов:
Stream 1: GET /api/user -> 200 OK {json}
Stream 3: GET /api/orders -> 200 OK {json}
Stream 5: POST /api/upload -> 100 Continue -> 201 Created
# Заголовки сжаты через HPACK, порядок ответов не зависит от порядка запросов
```

### WebSocket (гибридный паттерн запрос-ответ поверх полнодуплексного канала)
Полнодуплексный транспортный протокол, который сам по себе не навязывает модель взаимодействия. Поверх открытого канала легко реализовать как асинхронный поток событий, так и классический паттерн запрос-ответ с корреляцией идентификаторов.

Клиент отправляет сообщение с уникальным ID, сервер обрабатывает его и возвращает ответ с тем же ID, эмулируя RPC поверх постоянного соединения. Это даёт гибкость: в одном канале можно смешивать фоновые уведомления, heartbeats и синхронные запросы.

Реализация требует управления очередями, таймаутами и обработкой повторных отправок. Многие фреймворки (Socket.IO, SignalR) абстрагируют эту сложность, предоставляя готовые гибридные API.

WebSocket — нейтральный транспорт, который становится гибридным только благодаря прикладной логике. Реализация запрос-ответ поверх него требует ручного управления состоянием сессии, что превращает простой канал в полноценный коммуникационный фреймворк с возможностями и синхронного, и асинхронного взаимодействия.

Поверх постоянного полнодуплексного канала легко реализовать синхронный RPC-паттерн, где клиент отправляет сообщение с уникальным `requestId`, а сервер возвращает ответ с тем же ID, эмулируя традиционный HTTP в асинхронной среде.
```json
// Клиент -> Сервер
{"id": "req_88a", "method": "getUser", "params": {"uid": 42}}
// Сервер -> Клиент
{"id": "req_88a", "result": {"name": "Alex", "status": "active"}, "error": null}
// Параллельно идут асинхронные уведомления: {"type": "event", "data": {"ping": true}}
```

### REST (гибридный асинхронный)
Архитектурный стиль, который базово построен на синхронной модели запрос-ответ через HTTP. Однако в реальных высоконагруженных системах его часто комбинируют с асинхронными паттернами для обработки длительных операций без блокировки клиента.

Клиент инициирует задачу, получает статус `202 Accepted` и URL для проверки результата, либо регистрирует webhook для обратного вызова. Это позволяет разблокировать соединение и делегировать работу фоновым обработчикам или очередям сообщений (RabbitMQ, Kafka).

Такой гибридный подход сохраняет простоту REST, добавляя отказоустойчивость и масштабируемость. Он широко применяется в обработке видео, генерации отчётов, ML-инференсе и интеграциях с внешними сервисами.

REST в чистом виде синхронен, но его гибкость позволяет легко интегрировать асинхронные сценарии через статус-коды 202/204, polling эндпоинты или сторонние брокеры. Это делает его де-факто гибридным стандартом для enterprise-интеграций, где нужно балансировать между простотой HTTP и отказоустойчивостью событийных систем.

Базово синхронный REST расширяется асинхронными паттернами через статус `202 Accepted` и callback-URL, позволяя инициировать длительную операцию, сразу освобождая соединение, а затем получать результат через polling или webhook.
```http
POST /api/v1/reports/generate HTTP/1.1
-> 202 Accepted
   Location: /api/v1/reports/jobs/rep_77b
   Callback-URL: https://myapp.com/webhooks/report-ready
# Клиент проверяет GET /api/v1/reports/jobs/rep_77b/status
# Или ждёт POST на Callback-URL с payload {"job_id": "rep_77b", "status": "done"}
```
