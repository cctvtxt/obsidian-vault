[[API]]

**Контракт API** — это понятие, обобщающее понятие протокол и API, это формальное соглашение о том, какие данные и поведения ожидаются от программы. Кроме надежности, контракты делают поведение программ предсказуемым и документируемым, что упрощает ревью, генерацию тестов и поддержку командой.
# Contract First Approach

Подход к разработке API **Contract First** подразумевает, что проектирование и утверждение контракта происходит до написания основной логики приложения. Контракт становится единым источником правды как для серверной, так и для клиентской команды, позволяя вести разработку параллельно.

В экосистеме Node.js для описания контрактов чаще всего используется спецификация **OpenAPI (ранее Swagger)**. Она позволяет формально описать API в формате YAML или JSON. На основе этого описания можно генерировать документацию, клиентские SDK и даже каркас серверного кода.

Использование контрактов обеспечивает предсказуемость, упрощает тестирование и интеграцию, а также служит надежной документацией, которая всегда остается актуальной.

Помимо формальных API контрактов, в программировании существуют и неявные контракты на уровне кода. Это соглашения о том, как функции должны себя вести.
# Design By Contract

Контрактное программирование — это подход к разработке, при котором оговаривается, что должна делать каждая часть системы. **Контракт** является способом формального описания ожиданий. 

1. **Контракты могут описываться декларативно (существовать отдельно от кода)** и проверяться до выполнения кода. Это JSDoc или TypeScript-описания.
2. Контракты могут представлять собой **негласные соглашения**, вроде callback-last-error-first для асинхронных функций, аргументом принимающих каллбек. При этом при реализации асинхронной функции, данный контракт может быть нарушен разработчиком. Преобразовать возвращающую Promise функцию в callback-last-error-first-функцию можно, использовав `node.utils.callbackify`
3. Многие контракты **формализованы официально и добавлены в спецификацию ECMA Script**: Callable, Thenable, Promise, Array-like, Iterable, Observable, Disposable. Благодаря этому возможна разработка переносимого системного кода, а также высокоуровневая коммуникация и документация 
4. **Проверка контрактов может выполняться в процессе выполнения программы**, для этого существуют такие решения, как zod или metaschema. Для проверки контракта библиотека позволяет описывать shape, после чего валидировать любой объект на соответствие описанию. Часто без проверки в рантайме невозможно гарантирование соответствие функции предъявляемым требованиям, статическая проверка до выполнения не справляется с некоторыми задачами

```javascript
// Декларативное описание контракта для метода
{
  method: compeers,
  parameters: [
    { name: "a", type: "number" },
    { name: "b", type: "number" }
  ],
  return: { type: "boolean" }
}
```

Например, контракт iterable требует для объекта реализации метода `[Symbol.iterator]()`, который возвращает итератор с методом `next()`, возвращающим `{ value, done }`. Это позволяет:

- использовать стандартные языковые конструкции: `for..of`, spread `[...xs]`, `Array.from`, деструктуризацию и другие языковые возможности, которые автоматически опираются на этот контракт;
- создавать безопасные ограниченные проходы с учётом `done`, избегая бесконечных или непреднамеренных итераций;
- декларативно комбинировать и преобразовывать данные (например, `map`, `filter`) без знания внутренней организации источника.

Практическая выгода: код, написанный на этом контракте, переносим и понятен с любой реализацией итерируемого, а тесты/валидаторы могут гарантировать, что итератор всегда возвращает ожидаемые `value` и `done`, не нарушая поведения consumer-логики.
# Domain Specific Languages

Разработка DSL — это построение мини‑языка, который отражает предметную область и позволяет командам выражать требования, конфигурации и рабочие процессы так, чтобы их было просто читать, проверять и поддерживать. Качественный DSL снижает вероятность ошибок, делает рутинную работу предсказуемой и ускоряет onboarding новых участников.

Доменно-ориентированный язык (DSL) — это инструмент, созданный специально для одной сферы или задачи. В DSL ключевые понятия, операции и правила совпадают с языком предметной области, которую он моделирует. Это позволяет описывать решения на естественном для данной задачи vocabulary, не вдаваясь в технические низкоуровневые детали общего языка программирования. Особенность DSL в чёткой адаптации под домен: выражаете бизнес-правила, констрейнты и рабочие процессы так, как принято в этой области, — это делает код читаемым для представителей предметной области и снижает риск неверной интерпретации требований. Чтобы такие языки работали, их контракты (синтаксис, валидация, правила переходов) чётко фиксируются заранее — либо через строгую gramмату, либо через набор тестов и типов. 

DSL может быть реализован как при контрактном программировании внутри конкретного ЯП, так и в виде разработки подхода к описанию конфигурационных файлов, их валидации и обработки

Основная идея: заменить общие шаблоны и boilerplate на понятные термины из предметной области, сохранив при этом возможность использовать JavaScript для вычислений.
