[[Hard Skills Framework/Protocols/Application Layer/HTTP/HTTP Headers/Routing]]

Заголовок запроса `Host` указывает доменное имя сервера, к которому обращается клиент. Он является обязательным в HTTP/1.1 и более новых версиях.

# Описание

Значение заголовка состоит из доменного имени и опционального номера порта.

`Host: www.example.com`
`Host: api.service.com:8080`

Этот заголовок позволяет одному веб-серверу с одним IP-адресом обслуживать несколько разных веб-сайтов (виртуальный хостинг). Сервер читает заголовок `Host`, чтобы понять, какой именно сайт запрашивает клиент, и отдать ему соответствующий контент.

# Юзкейсы

*   **Виртуальный хостинг**: Это основной юзкейс. Без заголовка `Host` сервер, получивший запрос на свой IP-адрес, не знал бы, какой из сотен сайтов, размещенных на нем, нужно показать.
*   **Маршрутизация запросов**: Балансировщики нагрузки и обратные прокси используют `Host` для определения, на какой внутренний сервис (бэкенд) нужно перенаправить запрос.
*   **Формирование абсолютных URL**: Серверное приложение может использовать `Host` для генерации полных URL-адресов в своем ответе.

# Проблемы и особенности

*   **Обязательность**: `Host` является единственным заголовком, который был сделан обязательным в HTTP/1.1. Запрос без `Host` будет отвергнут сервером с ошибкой `400 Bad Request`.
*   **Безопасность (Host Header Injection)**: Если серверное приложение без должной валидации использует значение из `Host` для генерации URL (например, в ссылках для сброса пароля), это может привести к атакам. Злоумышленник может подставить в `Host` свой домен, и сгенерированная ссылка будет вести на вредоносный сайт. Необходимо всегда проверять `Host` по "белому списку" доверенных доменов.
*   **Порт**: Номер порта указывается, только если он отличается от стандартного (80 для HTTP, 443 для HTTPS).
*   **В HTTP/2 и HTTP/3**: Хотя заголовок `Host` все еще используется для совместимости, его роль выполняет псевдо-заголовок `:authority`, который содержит ту же информацию.
