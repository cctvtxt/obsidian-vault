`Access-Control-Allow-Credentials` — это заголовок ответа, который сообщает браузеру, разрешено ли клиентскому коду получать доступ к ответу, если запрос был сделан с передачей учетных данных (credentials).

# Описание

"Учетные данные" в контексте CORS — это `cookie`, HTTP-аутентификация и TLS-сертификаты клиента.

По умолчанию, браузер **не** отправляет `cookie` при кросс-доменных `fetch` или `XMLHttpRequest` запросах. Чтобы их отправить, клиент должен явно указать это:
*   В `fetch`: `fetch(url, { credentials: 'include' })`
*   В `XMLHttpRequest`: `xhr.withCredentials = true`

Если запрос был сделан с таким флагом, то для успешной обработки ответа сервер **обязан** вернуть заголовок:

`Access-Control-Allow-Credentials: true`

Если этот заголовок отсутствует или имеет значение `false`, браузер заблокирует ответ.

# Юзкейсы

*   **Аутентифицированные сессии API**: Это основной юзкейс. Когда фронтенд-приложение на одном домене должно взаимодействовать с API на другом домене, используя сессионные `cookie` для аутентификации пользователя.
*   **Персонализация контента**: Любой сценарий, где кросс-доменный запрос должен быть выполнен в контексте сессии пользователя.

# Проблемы и особенности

*   **Несовместимость с `*`**: Если сервер возвращает `Access-Control-Allow-Credentials: true`, то заголовок `Access-Control-Allow-Origin` **не может** иметь значение `*`. Он должен содержать конкретный домен-источник. Это мера безопасности, чтобы избежать случайной утечки пользовательских данных на неизвестные сайты.
*   **Обязателен при `credentials: 'include'`**: Если клиент отправляет учетные данные, а сервер не возвращает `Access-Control-Allow-Credentials: true`, браузер считает это ошибкой конфигурации CORS и блокирует ответ, даже если `Access-Control-Allow-Origin` указан верно.
*   **Только `true`**: Единственное осмысленное значение — `true`. Отсутствие заголовка эквивалентно `false`.
