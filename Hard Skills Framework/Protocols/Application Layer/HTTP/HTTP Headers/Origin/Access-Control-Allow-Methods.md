`Access-Control-Allow-Methods` — это заголовок ответа, который используется в ответе на preflight-запрос. Он сообщает браузеру, какие HTTP-методы разрешены для кросс-доменных запросов к данному ресурсу.

# Описание

Если клиентский код пытается сделать кросс-доменный запрос с "непростым" методом (таким как `PUT`, `DELETE`, `PATCH`), браузер сначала отправляет preflight-запрос методом `OPTIONS`.

В этом `OPTIONS`-запросе браузер использует заголовок `Access-Control-Request-Method`, указывая метод, который он собирается использовать.

Сервер в ответ должен вернуть `Access-Control-Allow-Methods`, перечислив разрешенные методы.

`Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS`

Браузер, получив этот ответ, проверяет, есть ли в списке запрошенный им метод. Если да — он отправляет основной запрос. Если нет — блокирует его.

# Юзкейсы

*   **Разрешение "опасных" методов**: Основная задача — явно разрешить использование методов, изменяющих состояние (`POST`, `PUT`, `DELETE`, `PATCH`), в кросс-доменных запросах к API.
*   **Описание возможностей ресурса**: Этот заголовок, по сути, сообщает клиенту, какие действия можно совершать с данным URL в контексте CORS.

# Проблемы и особенности

*   **Только для preflight**: Как и `Access-Control-Allow-Headers`, этот заголовок имеет значение только в ответе на `OPTIONS` (preflight) запрос. В ответе на `GET` или `POST` он не нужен и будет проигнорирован.
*   **Список методов**: Значение — это список имен методов, разделенных запятыми.
*   **`*` (wildcard)**: Использование `*` в этом заголовке не стандартизировано и не поддерживается. Необходимо явно перечислять все разрешенные методы.
*   **`OPTIONS`**: Рекомендуется всегда включать `OPTIONS` в список разрешенных методов, так как сам preflight-запрос использует этот метод.
*   **Кэширование preflight**: Ответ на preflight-запрос может быть кэширован браузером на время, указанное в `Access-Control-Max-Age`. Это позволяет избежать отправки `OPTIONS` запроса перед каждым `PUT` или `DELETE` запросом.
