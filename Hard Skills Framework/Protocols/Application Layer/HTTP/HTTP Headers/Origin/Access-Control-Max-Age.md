`Access-Control-Max-Age` — это заголовок ответа, который используется в ответе на preflight-запрос. Он сообщает браузеру, на какой срок (в секундах) можно кэшировать результат этого preflight-запроса.

# Описание

Preflight-запросы (`OPTIONS`) могут создавать дополнительную задержку перед отправкой основного запроса. Чтобы избежать повторной отправки `OPTIONS` перед каждым `PUT`, `DELETE` или другим "непростым" запросом, браузер может закэшировать результат preflight-проверки.

`Access-Control-Max-Age: 86400`

Этот заголовок говорит браузеру: "Ты можешь считать, что эти CORS-разрешения (на методы и заголовки) будут действительны в течение 86400 секунд (24 часа). В этот период можешь не спрашивать меня повторно".

# Юзкейсы

*   **Оптимизация производительности**: Основная задача — сократить количество HTTP-запросов и уменьшить задержку для пользователя. Это особенно актуально для "разговорчивых" (chatty) API, которые требуют частого обмена данными.
*   **Снижение нагрузки на сервер**: Уменьшает количество `OPTIONS` запросов, которые должен обрабатывать сервер.

# Проблемы и особенности

*   **Только для preflight**: Заголовок имеет смысл только в ответе на `OPTIONS` (preflight) запрос.
*   **Ограничения браузеров**: Браузеры могут иметь собственное максимальное значение для `Access-Control-Max-Age` и игнорировать слишком большие значения. Например, в Firefox максимум — 24 часа, в Chromium-based браузерах — 2 часа.
*   **Изменение прав**: Если вы кэшируете preflight-ответ надолго, а затем меняете CORS-политику на сервере (например, запрещаете какой-то метод), то клиенты со старым кэшем не узнают об этом до истечения `Max-Age`. Это может привести к неожиданным ошибкам. Поэтому не стоит устанавливать слишком большие значения.
*   **Кэшируется для конкретного URL**: Результат кэшируется для конкретного URL. Запрос к другому URL на том же сервере потребует нового preflight-запроса.
