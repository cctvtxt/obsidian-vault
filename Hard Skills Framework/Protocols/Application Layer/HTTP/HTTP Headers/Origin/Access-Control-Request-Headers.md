[[Hard Skills Framework/Protocols/Application Layer/HTTP/HTTP Headers/Origin]]

`Access-Control-Request-Headers` — это заголовок запроса, который браузер использует в preflight-запросе (`OPTIONS`). Он сообщает серверу, какие HTTP-заголовки, помимо "простых", будут использованы в фактическом запросе.

# Описание

Если JavaScript-код пытается выполнить кросс-доменный запрос, добавляя в него кастомные заголовки (например, `X-API-Key`) или "непростые" стандартные (например, `Authorization`), браузер инициирует preflight-проверку.

В `OPTIONS` запросе браузер отправляет заголовок `Access-Control-Request-Headers`, перечисляя имена этих "непростых" заголовков.

`Access-Control-Request-Headers: authorization, x-api-key`

Это вопрос серверу: "Я собираюсь отправить запрос с заголовками `Authorization` и `X-API-Key`. Ты их разрешаешь?".

Сервер должен ответить заголовком `Access-Control-Allow-Headers`, перечислив те из запрошенных заголовков, которые он готов принять.

# Юзкейсы

*   **Часть механизма Preflight**: Как и `Access-Control-Request-Method`, этот заголовок является ключевой частью preflight-проверки. Он позволяет серверу заранее проверить, какие заголовки будет использовать клиент, и отклонить запрос, если они не соответствуют политике безопасности.
*   **Явное разрешение**: Требует от разработчика API явно определять, какие кастомные заголовки разрешены, что повышает безопасность.

# Проблемы и особенности

*   **Добавляется браузером**: Заголовок полностью контролируется браузером и не может быть изменен из JavaScript.
*   **Только в `OPTIONS`**: Отправляется только в preflight-запросах.
*   **Нормализация имен**: Имена заголовков в значении приводятся к нижнему регистру.
*   **"Простые" заголовки не перечисляются**: Заголовки, считающиеся "простыми" (`Accept`, `Accept-Language`, `Content-Language`, `Content-Type` с определенными значениями), не требуют preflight-проверки и не включаются в этот список.
