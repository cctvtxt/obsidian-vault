[[Auth]]

Заголовок `Authorization` является основным механизмом для передачи учетных данных клиента на сервер для аутентификации. Он позволяет защищенным ресурсам проверять, имеет ли клиент право на доступ.

# Описание

Заголовок `Authorization` содержит учетные данные для аутентификации клиента на сервере. Формат учетных данных зависит от выбранной схемы аутентификации. Общий синтаксис: `Authorization: <type> <credentials>`.

**Основные схемы аутентификации:**

1.  **Basic**: Учетные данные представляют собой строку `username:password`, закодированную в `Base64`. Это простая, но небезопасная схема, так как данные легко декодируются. Ее следует использовать только вместе с HTTPS.
2.  **Bearer**: Используется для передачи токенов доступа (например, `JWT` или `OAuth` токены). Токен непрозрачен для клиента и является самодостаточным для аутентификации. Это наиболее распространенный способ для защиты API.
3.  **Digest**: Более безопасная альтернатива `Basic`, использующая хеширование с одноразовым кодом (`nonce`), чтобы избежать передачи пароля в открытом виде.
4.  **HOBA**: Схема на основе цифровых подписей, обеспечивает более высокий уровень безопасности.
5.  **AWS4-HMAC-SHA256**: Схема, используемая Amazon Web Services для аутентификации запросов к своим API.

# Юзкейсы

*   **Доступ к API**: Клиентские приложения (веб, мобильные) используют `Authorization` с `Bearer` токеном для доступа к защищенным эндпоинтам API.
*   **Базовая аутентификация**: Быстрая защита внутренних или некритичных ресурсов, где настройка сложной системы нецелесообразна.
*   **Доступ к файлам**: Защита доступа к файлам или медиаконтенту на сервере.

# Проблемы и особенности

*   **Безопасность**: Схема `Basic` уязвима для атак "человек посередине" (`MITM`) без `HTTPS`. Учетные данные могут быть перехвачены и легко расшифрованы.
*   **Утечка токенов**: `Bearer` токены должны быть защищены. В случае утечки токен может быть использован злоумышленником для получения несанкционированного доступа. Рекомендуется использовать короткоживущие токены и механизмы их отзыва (refresh tokens).
*   **CORS и Preflight-запросы**: При кросс-доменных запросах с заголовком `Authorization` браузер отправляет `preflight` запрос (`OPTIONS`), чтобы проверить, разрешает ли сервер такой заголовок. Неправильная конфигурация `CORS` может блокировать запросы.
