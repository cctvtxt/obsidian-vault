[[Auth]]

Заголовок ответа `Proxy-Authenticate` используется прокси-сервером, чтобы сообщить клиенту, какой метод аутентификации необходим для доступа через этот прокси. Он отправляется с кодом состояния `407 Proxy Authentication Required`.

# Описание

Этот заголовок является аналогом `WWW-Authenticate`, но для прокси. Когда клиент отправляет запрос через прокси, требующий аутентификации, прокси блокирует запрос и отвечает статусом `407`, включая `Proxy-Authenticate`.

Синтаксис: `Proxy-Authenticate: <type> realm="<realm>"`

Параметры и схемы (`Basic`, `Bearer` и т.д.) аналогичны `WWW-Authenticate`. Прокси-сервер может предложить несколько схем, отправив несколько таких заголовков.

# Юзкейсы

*   **Информирование клиента**: Основная задача — сообщить клиентскому приложению (например, браузеру), что прокси требует аутентификации и как ее пройти.
*   **Динамический выбор схемы**: Клиент может получить несколько вариантов аутентификации и выбрать наиболее безопасный из поддерживаемых.
*   **Корпоративные политики**: Прокси-серверы в корпоративных сетях используют этот заголовок для принудительного применения политик безопасности, требуя от пользователей подтвердить свою личность.

# Проблемы и особенности

*   **Прозрачные прокси**: Этот механизм не работает с "прозрачными" прокси, которые перехватывают трафик без ведома клиента и не могут инициировать HTTP-аутентификацию.
*   **Поддержка клиентами**: Корректная обработка ответа `407` и заголовка `Proxy-Authenticate` полностью зависит от реализации HTTP-клиента. Некоторые утилиты командной строки или простые клиенты могут не поддерживать этот механизм "из коробки".
*   **Безопасность**: Как и в случае с `WWW-Authenticate`, использование небезопасных схем (например, `Basic`) без шифрования `HTTPS` между клиентом и прокси создает риски перехвата учетных данных.
