[[Auth]]

Заголовок ответа `WWW-Authenticate` используется сервером, чтобы указать клиенту, какая схема аутентификации требуется для доступа к запрашиваемому ресурсу. Обычно он отправляется с кодом состояния `401 Unauthorized`.

# Описание

Когда клиент запрашивает защищенный ресурс без учетных данных (или с неверными), сервер отвечает статусом `401` и заголовком `WWW-Authenticate`. Этот заголовок сообщает клиенту, как именно нужно аутентифицироваться.

Синтаксис: `WWW-Authenticate: <type> realm="<realm>", [parameters...]`

*   `<type>`: Схема аутентификации (например, `Basic`, `Bearer`, `Digest`).
*   `realm`: Описание защищенной области. Может отображаться пользователю в окне ввода логина/пароля.

Сервер может предложить несколько схем аутентификации, перечислив их в нескольких заголовках `WWW-Authenticate`.

# Юзкейсы

*   **Интерактивная аутентификация в браузере**: Когда браузер получает `401` с `WWW-Authenticate: Basic`, он автоматически отображает пользователю стандартное окно для ввода имени пользователя и пароля.
*   **API-взаимодействие**: API-клиент, получив `401`, может проанализировать `WWW-Authenticate`, чтобы понять, какую схему использовать (`Bearer`, `Digest` и т.д.) и какие параметры необходимы.
*   **Обнаружение механизмов аутентификации**: Клиент может "узнать" поддерживаемые сервером методы аутентификации, сделав пробный запрос к защищенному ресурсу.

# Проблемы и особенности

*   **Безопасность `realm`**: Не следует включать в `realm` конфиденциальную информацию, так как она видна клиенту.
*   **Неоднозначность**: Хотя заголовок стандартизирован, некоторые клиенты могут по-разному интерпретировать его параметры, особенно для кастомных схем.
*   **Блокировка всплывающих окон**: Встроенное в браузер окно аутентификации, инициируемое `WWW-Authenticate: Basic`, не может быть стилизовано и предоставляет плохой пользовательский опыт по сравнению с кастомными формами входа.
