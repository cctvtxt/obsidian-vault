[[Custom]]

`X-CSRF-Token` (или `X-XSRF-TOKEN`) — это заголовок запроса, используемый для защиты от атак типа "Межсайтовая подделка запроса" (Cross-Site Request Forgery, CSRF).

# Описание

CSRF-атака происходит, когда вредоносный сайт заставляет браузер пользователя выполнить нежелательное действие на другом сайте, на котором пользователь уже аутентифицирован (например, перевести деньги, изменить пароль).

Для защиты используется следующий механизм (Double Submit Cookie):
1.  Когда пользователь заходит на сайт, сервер генерирует случайный, уникальный токен (CSRF-токен), сохраняет его в сессии пользователя и отправляет его клиенту в виде cookie.
2.  Клиентское JavaScript-приложение читает этот токен из cookie.
3.  При каждом "опасном" запросе (POST, PUT, DELETE) клиент должен добавить этот токен в заголовок `X-CSRF-Token`.
4.  Сервер перед выполнением запроса проверяет, что токен в заголовке совпадает с токеном, сохраненным в сессии.

Поскольку вредоносный сайт не может прочитать cookie с другого домена, он не может и подставить правильный токен в заголовок.

# Юзкейсы

*   **Защита SPA (Single Page Applications)**: Этот механизм особенно популярен в SPA (React, Angular, Vue), где клиентская логика активно взаимодействует с API.
*   **Защита любых state-changing запросов**: Любая операция, которая изменяет состояние на сервере (создание, обновление, удаление данных), должна быть защищена CSRF-токеном.

# Проблемы и особенности

*   **Не для `GET`**: Запросы `GET` (и `HEAD`, `OPTIONS`) по спецификации не должны изменять состояние, поэтому их обычно не защищают CSRF-токенами.
*   **Название заголовка**: Имя заголовка не стандартизировано. Популярные фреймворки используют `X-CSRF-Token` (Ruby on Rails, Django), `X-XSRF-TOKEN` (Angular).
*   **CORS**: Если API и клиентское приложение находятся на разных доменах, необходимо правильно настроить CORS, чтобы разрешить передачу этого кастомного заголовка.
*   **Альтернативы**: Существует другой подход — `SameSite` атрибут у cookie. Установка `SameSite=Strict` или `SameSite=Lax` для сессионной cookie может во многих случаях предотвратить CSRF-атаки без использования токенов, так как браузер просто не будет отправлять cookie при запросах с другого сайта.
