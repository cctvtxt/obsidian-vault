[[Custom]]

`X-Forwarded-For` (XFF) — это де-факто стандартный заголовок, который содержит IP-адрес исходного клиента, подключившегося к веб-серверу через HTTP-прокси или балансировщик нагрузки.

# Описание

Когда трафик проходит через прокси, IP-адрес клиента в TCP-соединении с конечным сервером меняется на IP-адрес прокси. Чтобы сервер мог узнать реальный IP пользователя, прокси добавляет заголовок `X-Forwarded-For`.

`X-Forwarded-For: <client>, <proxy1>, <proxy2>`

Если запрос проходит через несколько прокси, каждый из них добавляет свой IP-адрес в конец списка. Таким образом, **самый левый IP-адрес** является наиболее достоверным кандидатом на реальный IP-адрес клиента.

# Юзкейсы

*   **Логирование и аналитика**: Запись реальных IP-адресов пользователей для анализа трафика, геолокации и статистики.
*   **Безопасность**: Используется для определения источника запросов, блокировки по IP (rate limiting) и выявления подозрительной активности.
*   **Персонализация контента**: Отображение контента, релевантного для региона пользователя (например, валюты, языка).

# Проблемы и особенности

*   **Доверие к заголовку**: `X-Forwarded-For` легко подделать. Злоумышленник может отправить запрос с этим заголовком, указав в нем любой IP-адрес. Поэтому доверять можно только тем значениям, которые были добавлены вашими собственными, доверенными прокси-серверами.
*   **Порядок IP-адресов**: Приложение должно правильно парсить заголовок, понимая, что самый левый IP — это клиент, а последующие — это прокси.
*   **Стандарт `Forwarded`**: Для решения проблем с `X-Forwarded-*` заголовками был предложен стандартный заголовок `Forwarded`, который имеет более строгую и расширяемую структуру. Однако `X-Forwarded-For` все еще используется гораздо чаще из-за своей простоты и широкой поддержки.
*   **Конфигурация сервера**: Веб-сервер или приложение должны быть настроены так, чтобы они доверяли и использовали этот заголовок. Например, в Nginx используется `proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;`.
