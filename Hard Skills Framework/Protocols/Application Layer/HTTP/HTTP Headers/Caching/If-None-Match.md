[[Caching]]

Заголовок запроса `If-None-Match` используется для условных запросов. Он содержит одно или несколько значений `ETag`, которые есть у клиента в кэше. Сервер вернет полный ответ (`200 OK`) только в том случае, если ни один из `ETag` клиента не совпадает с текущим `ETag` ресурса на сервере.

# Описание

Когда клиент кэширует ресурс с заголовком `ETag`, при последующих запросах он использует `If-None-Match` для валидации этого ресурса.

`If-None-Match: "33a64df551425fcc55e4d42a148795d9f25f89d4"`

Если `ETag` на сервере совпадает с одним из значений в `If-None-Match`, сервер понимает, что у клиента актуальная версия, и возвращает короткий ответ `304 Not Modified`. Это говорит клиенту, что можно безопасно использовать кэшированную копию.

# Юзкейсы

*   **Валидация кэша**: Это основной и самый частый способ использования `If-None-Match`. Он позволяет браузерам и CDN эффективно проверять актуальность контента, не загружая его повторно. Это экономит трафик и ускоряет загрузку.
*   **Запросы `GET` и `HEAD`**: `If-None-Match` наиболее эффективен с этими методами, так как позволяет избежать повторной передачи тела ресурса.

# Проблемы и особенности

*   **Сравнение `ETag`**: Сервер должен корректно сравнивать `ETag`. Для сильных `ETag` требуется побайтовое совпадение, для слабых (с префиксом `W/`) — семантическое.
*   **Список `ETag`**: Клиент может передать несколько `ETag`, разделенных запятыми (`If-None-Match: "etag1", "etag2"`). Сервер должен проверить совпадение с каждым из них.
*   **`If-None-Match: *`**: Специальное значение `*` означает "если ресурс вообще существует". Это может быть использовано в методах `PUT` для создания ресурса только в том случае, если он еще не существует. Если ресурс уже есть, сервер вернет `412 Precondition Failed`.
*   **Приоритет над `If-Modified-Since`**: Если в запросе присутствуют оба заголовка (`If-None-Match` и `If-Modified-Since`), `If-None-Match` имеет более высокий приоритет, и `If-Modified-Since` будет проигнорирован.
