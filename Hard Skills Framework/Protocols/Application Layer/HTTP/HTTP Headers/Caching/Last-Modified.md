[[Caching]]

Заголовок ответа `Last-Modified` содержит дату и время последнего изменения ресурса. Он используется как один из механизмов валидации кэша, обычно в паре с заголовком запроса `If-Modified-Since`.

# Описание

Значением заголовка является дата в формате HTTP-date.

`Last-Modified: Wed, 21 Oct 2025 07:28:00 GMT`

Сервер сообщает клиенту, когда ресурс был в последний раз обновлен. Клиент, получив этот заголовок, может при последующих запросах использовать его значение в заголовке `If-Modified-Since`, чтобы спросить у сервера, изменился ли ресурс с тех пор.

# Юзкейсы

*   **Валидация кэша**: Основное применение — позволить клиентам проверять актуальность кэшированных данных. Если ресурс не изменился, сервер вернет `304 Not Modified`, экономя трафик.
*   **Информационная метка**: Может использоваться для отображения пользователю даты последнего обновления страницы.
*   **Поисковые системы**: Краулеры используют `Last-Modified` для определения, нужно ли повторно индексировать страницу.

# Проблемы и особенности

*   **Менее точен, чем `ETag`**: `Last-Modified` имеет точность до одной секунды. Если ресурс может меняться несколько раз в секунду, этот заголовок не сможет отследить все изменения. `ETag` (идентификатор версии) в таких случаях предпочтительнее.
*   **Сложность для динамического контента**: Для статических файлов дата последнего изменения — это дата модификации файла в файловой системе. Для динамически генерируемых страниц (например, из базы данных) серверу необходимо специально отслеживать "дату последнего изменения", что может быть нетривиально.
*   **Проблемы синхронизации в кластере**: В распределенных системах с несколькими серверами необходимо обеспечить, чтобы все узлы возвращали одинаковое значение `Last-Modified` для одного и того же ресурса.
*   **Не используется, если есть `ETag`**: При наличии `If-None-Match` в запросе, серверы игнорируют `If-Modified-Since`.
