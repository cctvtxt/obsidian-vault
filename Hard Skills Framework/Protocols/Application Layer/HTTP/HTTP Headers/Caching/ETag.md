[[Caching]]

Заголовок ответа `ETag` (Entity Tag) — это идентификатор конкретной версии ресурса. Он позволяет кэшам быть более эффективными и экономить трафик, так как серверу не нужно повторно отправлять ресурс, если он не изменился.

# Описание

`ETag` представляет собой строку, обычно хэш содержимого ресурса или идентификатор его версии.

`ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"`

Когда браузер или CDN кэширует ответ с заголовком `ETag`, при следующем запросе этого же ресурса он отправляет значение `ETag` в заголовке `If-None-Match`.

**Валидация:**

1.  Клиент запрашивает ресурс, который у него уже есть в кэше с `ETag: "123"`.
2.  Клиент отправляет серверу запрос с заголовком `If-None-Match: "123"`.
3.  Сервер сравнивает полученный `ETag` с текущим `ETag` ресурса.
    *   Если они совпадают, сервер отвечает `304 Not Modified` с пустым телом. Клиент использует ресурс из кэша.
    *   Если они не совпадают, сервер отвечает `200 OK` с новым ресурсом и новым `ETag`.

# Юзкейсы

*   **Валидация кэша**: Основной юзкейс — проверка актуальности кэшированного ресурса без скачивания всего тела ответа.
*   **Оптимистичные блокировки**: В REST API заголовок `ETag` используется для предотвращения "потерянных обновлений". Клиент читает ресурс, получает `ETag`, а при обновлении (`PUT` или `PATCH`) отправляет его в заголовке `If-Match`. Если `ETag` на сервере изменился (кто-то другой обновил ресурс), сервер отклонит запрос со статусом `412 Precondition Failed`.

# Проблемы и особенности

*   **Сильный и слабый `ETag`**:
    *   **Сильный `ETag`** (например, `"hash"`) меняется при любом, даже минимальном, изменении тела ответа.
    *   **Слабый `ETag`** (например, `W/"hash"`) меняется только при семантических изменениях. Например, он может не меняться при сжатии или незначительных правках. Используется, когда точное побайтовое совпадение не требуется.
*   **Генерация `ETag`**: Нагрузка по генерации `ETag` ложится на сервер. Для статических файлов это просто (хэш файла), для динамического контента может быть сложнее.
*   **Кластерные среды**: В среде с несколькими серверами (балансировщик нагрузки) важно, чтобы все серверы генерировали одинаковый `ETag` для одного и того же ресурса. Иначе валидация кэша будет работать некорректно.
