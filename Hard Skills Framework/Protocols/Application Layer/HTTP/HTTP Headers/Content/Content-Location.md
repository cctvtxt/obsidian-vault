[[Content]]

Заголовок ответа `Content-Location` указывает альтернативный URL для ресурса, возвращаемого в ответе. В основном он используется для согласования контента.

# Описание

Значением является относительный или абсолютный URL.

`Content-Location: /document.html`

Представим, что у нас есть ресурс `/item`, который может быть отдан в формате `HTML` или `JSON`.

1.  Клиент запрашивает `GET /item` с заголовком `Accept: text/html`.
2.  Сервер возвращает `HTML`-представление этого ресурса и добавляет заголовок `Content-Location: /item.html`.

Теперь клиент знает, что полученное `HTML`-представление ресурса `/item` также доступно напрямую по адресу `/item.html`.

# Юзкейсы

*   **Согласование контента**: Основной юзкейс. Позволяет указать прямой URL для конкретного представления ресурса, которое было выбрано в результате content negotiation (по заголовкам `Accept`, `Accept-Language` и т.д.).
*   **Кэширование**: Кэширующие прокси могут использовать `Content-Location` для кэширования ответа под новым, более конкретным URL.
*   **Отделение ресурса от его представления**: Помогает в архитектурах, где один и тот же "абстрактный" ресурс (например, `/users/123`) имеет множество различных "физических" представлений (например, `/users/123.json`, `/users/123.xml`).

# Проблемы и особенности

*   **Путаница с `Location`**: Не следует путать с заголовком `Location`. `Location` используется в ответах с кодами `3xx` (редиректы) и `201 Created` для указания URL, на который нужно перейти, или URL созданного ресурса. `Content-Location` же просто предоставляет альтернативный URL для уже полученного контента, не вызывая никакого перехода.
*   **Относительные пути**: Путь в `Content-Location` разрешается относительно URL запроса, как и в `Location`.
*   **Редкое использование**: На практике этот заголовок встречается довольно редко, так как современные API и сайты чаще используют другие подходы к версионированию и представлению контента (например, через параметры запроса или разные эндпоинты).
