[[Stateless Auth]]

Базовая аутентификация это простейший HTTP‑метод, при котором клиент передает логин и пароль при каждом запросе. Данные кодируются в Base64 и передаются в заголовке HTTP.

Базовая аутентификация требует только имя пользователя и пароль. Клиент берет эти два удостоверения, склеивает их вместе, чтобы сформировать одно значение, и передает его в запросе в заголовке **HTTP**, называемом **Authorization**. 

Базовая аутентификация тривиальна в реализации и поддерживается везде, но крайне уязвима без шифрования HTTPS, так как учетные данные уходят в каждом запросе и фактически могут быть перехвачены, поэтому чаще используется во внутренних, защищенных средах или для простых тестовых API. 

_Процесс базовой аутентификации включает в себя объединение имени пользователя с двоеточием, за которым следует пароль, а затем прогон всей строки через алгоритм кодирования_ **_base64_**_. Таким образом, "_**_user_**_" и "_**_password_**_" становятся "_**_user:password_**_", а после кодирования получается "_`**_dXNlcjpwYXNzd29yZAo=_**`\_"
## Принцип работы 

Когда сервер получает запрос, он просматривает заголовок **Authorization** и сравнивает его с сохраненными учетными данными. Если имя пользователя и пароль совпадают с одним из пользователей в списке сервера, сервер выполняет запрос клиента от имени этого пользователя. Если совпадений нет, сервер возвращает специальный код статуса (**401**) (**special status code**), чтобы сообщить клиенту, что аутентификация не удалась и запрос отклонен
## Достоинства

Простота реализации и отладки. Метод не требует сложных библиотек на клиенте или сервере. Его легко отладить с помощью стандартных инструментов, таких как `curl` или браузерные утилиты разработчика.

Широкая поддержка. Является частью стандарта HTTP и поддерживается почти всеми клиентами и серверами без необходимости установки дополнительных модулей.

Отсутствие состояния на сервере. Сервер не хранит информацию о сессии, что упрощает его масштабирование и снижает нагрузку.
## Недостатки

Низкий уровень безопасности. Учетные данные передаются в формате Base64, который легко декодировать. Без HTTPS их можно перехватить с помощью атаки "человек посередине".

Повторная передача учетных данных. Логин и пароль отправляются с каждым запросом, что увеличивает риск их компрометации.

Отсутствие механизма выхода (Logout). Протокол не предусматривает способа завершить сессию. Браузер кеширует учетные данные и отправляет их, пока вкладка не будет закрыта.

Уязвимость к CSRF. Браузеры автоматически добавляют заголовок `Authorization` ко всем запросам к домену после успешной аутентификации, что делает возможными CSRF-атаки.
## Кейсы использования

Внутренние и тестовые API. Подходит для API внутри защищенной сети, где риск перехвата минимален.

Служебные скрипты. Используется для доступа к ресурсам, не требующим высокого уровня безопасности.

Прототипирование. Удобен для быстрой реализации аутентификации на этапе разработки.

Защита некритичных ресурсов. Может применяться для ограничения доступа к статическим страницам или панелям администрирования, где нет чувствительных данных, но только в связке с HTTPS.
