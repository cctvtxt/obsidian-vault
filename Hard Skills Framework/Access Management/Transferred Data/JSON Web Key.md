[[JWT-Based Auth]]

**JSON Web Key (JWK)** — это стандартный формат представления криптографического ключа в виде JSON-объекта. Он позволяет единообразно описывать ключи различных типов (RSA, EC) и их параметры. Спецификация определена в RFC 7517 и является основой для JWS, JWE и JWKS.

## Публичные JWK

```JSON
{
  "kty": "RSA",
  "use": "sig",
  "kid": "key-2024-07-01",
  "alg": "RS256",
  "n": "base64url-модуль",
  "e": "AQAB"
}
```

Поле `kty` (Key Type) указывает тип криптографического алгоритма и является обязательным — значение `"RSA"` определяет набор допустимых параметров ключа, тогда как значение `"EC"` потребовало бы совершенно иных полей (`crv`, `x`, `y`). Поле `use` фиксирует назначение ключа: значение `"sig"` означает использование для проверки подписи, а значение `"enc"` — для операций шифрования. Поле `kid` (Key ID) служит уникальным идентификатором, позволяющим клиентским приложениям выбрать нужный ключ из набора JWKS при валидации конкретного токена. Поле `alg` закрепляет алгоритм, предназначенный для работы с данным ключом, что даёт клиенту возможность отклонить токен, подписанный неожиданным алгоритмом.

Параметры `n` (модуль) и `e` (публичная экспонента) составляют математическую основу публичного RSA-ключа и кодируются в формате Base64url. Именно эти два значения позволяют любому участнику системы выполнить операцию проверки подписи, однако даже совместное знание `n` и `e` не даёт возможности вычислить приватную экспоненту `d` за разумное время при достаточной длине ключа.

### Назначение и распространение

Публичный JWK выполняет единственную задачу со стороны подписи — верификацию, то есть математическое подтверждение того, что токен был создан владельцем соответствующего приватного ключа и не был изменён после подписания. Публикация осуществляется через JWKS endpoint, расположенный по стандартному пути `/.well-known/jwks.json`, откуда Resource Server загружает актуальный набор ключей. Клиентские приложения кэшируют полученные ключи в соответствии с HTTP-заголовками `Cache-Control` и `max-age`, что снижает нагрузку на сервис авторизации и уменьшает латентность валидации. **Публичный ключ не является секретом** даже в минимальной степени — напротив, его широкое и беспрепятственное распространение составляет необходимое условие работы децентрализованной JWT-авторизации.

## Приватные JWK

```json
{
  "kty": "RSA",
  "kid": "key-2024-07-01",
  "alg": "RS256",
  "n": "...",
  "e": "AQAB",
  "d": "base64url-приватная-экспонента",
  "p": "...",
  "q": "...",
  "dp": "...",
  "dq": "...",
  "qi": "..."
}
```

Параметр `d` представляет собой приватную экспоненту — центральный секрет, владение которым позволяет формировать подписи, неотличимые от подлинных. Параметры `p` и `q` — это два больших простых числа, произведение которых образует модуль `n`, и знание этих множителей эквивалентно полному взлому ключа. Параметры `dp`, `dq` и `qi` являются предвычисленными значениями для оптимизации операции подписания по Китайской теореме об остатках (Chinese Remainder Theorem), сокращая время генерации подписи в 3–4 раза по сравнению с прямым возведением в степень.

### Назначение

Приватный ключ выполняет две категории операций: создание цифровой подписи JWT (JWS) и расшифровку данных, зашифрованных соответствующим публичным ключом (JWE). Сервис авторизации (Identity Provider) использует приватный ключ при каждом выпуске нового токена, вычисляя подпись над закодированными заголовком и телом JWT. Компрометация приватного ключа означает полную компрометацию системы авторизации, поскольку злоумышленник получает возможность выпускать произвольные токены с любыми claims от имени легитимного сервиса, и ни один Resource Server не сможет отличить поддельный токен от настоящего.

### Хранение

Безопасность приватного ключа определяет безопасность всей JWT-инфраструктуры, и требования к хранению носят категорический характер.

**Системы управления секретами** составляют наиболее надёжный подход: HashiCorp Vault, AWS KMS, Azure Key Vault и GCP KMS обеспечивают централизованное хранение с контролем доступа, аудитом и автоматической ротацией. В идеальной конфигурации приватный ключ не покидает периметр KMS — сервис авторизации передаёт хэш данных в KMS через защищённый API, а KMS возвращает готовую подпись, что даже при полной компрометации сервера приложения не приводит к утечке самого ключа. Hardware Security Module (HSM) обеспечивает максимальный уровень защиты, храня ключевой материал в специализированном криптографическом устройстве с физической защитой от вскрытия, что необходимо для соответствия стандартам PCI-DSS и FIPS 140-2.

**Файловое хранение** допустимо лишь при строгом ограничении прав доступа: файл с ключом в формате PEM или JWK должен иметь права `chmod 400`, а владельцем файла должен быть исключительно процесс приложения. Передача ключа через **переменные окружения** требует загрузки из защищённых источников при старте процесса, при этом значения переменных не должны попадать в логи, core-dump или трассировки ошибок.

Хранение приватных ключей в системах контроля версий, в исходном коде приложения, в незашифрованных конфигурационных файлах или передача по незащищённым каналам категорически запрещены — даже однократное попадание ключа в историю Git делает его скомпрометированным, поскольку история коммитов сохраняется бессрочно.

## Связка публичного и приватного ключей

Публичный и приватный ключи связаны математически и образуют неразрывную пару: приватный ключ создаёт подпись, а публичный — проверяет её, и подмена одного из ключей делает пару неработоспособной. **Типичный поток JWT-авторизации** разворачивается следующим образом: сервис авторизации подписывает JWT приватным ключом и помещает `kid` в JOSE-заголовок токена; клиент или Resource Server извлекает `kid`, загружает соответствующий публичный ключ из JWKS и выполняет проверку подписи. При наличии нескольких активных ключей в JWKS именно `kid` служит селектором, однозначно указывающим, какой публичный ключ следует использовать для валидации данного конкретного токена.

## Жизненный цикл ключей в JWT-авторизации

Корректное управление жизненным циклом криптографических ключей определяет как отказоустойчивость, так и безопасность JWT-системы. Ошибки на этапе ротации приводят к массовому отклонению действительных токенов и каскадным сбоям авторизации, тогда как ошибки при утилизации оставляют скомпрометированный криптографический материал в эксплуатации.

### 1. Генерация (Generation)

Сервис авторизации генерирует приватный ключ с использованием криптографически стойкого генератора псевдослучайных чисел (CSPRNG), гарантирующего непредсказуемость ключевого материала. При использовании KMS или HSM генерация происходит непосредственно внутри защищённого периметра устройства, и приватный ключ никогда не существует в оперативной памяти сервера приложения. Публичный ключ математически выводится из приватного — для RSA он представляет собой пару значений `(n, e)`, извлечённых из полной структуры ключа. Каждой сгенерированной паре присваивается уникальный `kid`, в качестве которого рекомендуется использовать усечённый хэш SHA-256 от публичного ключа, что обеспечивает детерминированность идентификатора и позволяет клиентам при необходимости самостоятельно верифицировать соответствие `kid` содержимому ключа.

### 2. Активация и использование (Activation)

После генерации публичный ключ добавляется в массив `keys` JWKS-эндпоинта, становясь доступным для всех Resource Server. Сервис авторизации переключает процесс подписания на новый приватный ключ и начинает указывать соответствующий `kid` в заголовке каждого выпускаемого JWT. Resource Server при получении токена извлекает `kid` из JOSE-заголовка, обращается к кэшированному JWKS или запрашивает актуальную версию с эндпоинта и выполняет проверку подписи найденным публичным ключом. Отсутствие ключа с нужным `kid` в JWKS приводит к принудительному обновлению кэша — если же ключ не обнаружен и после обновления, токен отклоняется с ошибкой валидации.

### 3. Ротация (Key Rotation)

Плановая ротация ключей должна происходить регулярно — типичный интервал составляет от 30 до 90 дней — для ограничения масштаба ущерба в случае незамеченной компрометации. Процедура ротации требует неукоснительного соблюдения порядка действий, нарушение которого приводит к сбоям авторизации.

Корректный алгоритм ротации состоит из четырёх шагов. Первый — генерация новой пары ключей с новым уникальным `kid`. Второй — добавление нового публичного ключа в JWKS при сохранении старого, в результате чего эндпоинт одновременно содержит оба публичных ключа. Третий — переключение процесса подписания на новый приватный ключ, после чего все вновь выпускаемые токены содержат новый `kid` в заголовке. Четвёртый — вывод старого приватного ключа из эксплуатации (подробнее в разделе "Утилизация").

Типичная и крайне опасная ошибка заключается в одновременном удалении старого публичного ключа из JWKS при добавлении нового: все ранее выпущенные и ещё действительные токены мгновенно перестают проходить валидацию, что порождает лавинообразные ошибки `401 Unauthorized` на стороне клиентских приложений.

### 4. Вывод публичного ключа (Retirement from JWKS)

Публичный ключ не прекращает существование в момент ротации — напротив, он продолжает оставаться в JWKS ровно до тех пор, пока не истечёт срок действия последнего токена, подписанного соответствующим приватным ключом. Если максимальное значение `exp` для access token составляет 3600 секунд, старый публичный ключ должен находиться в JWKS не менее 3600 секунд после момента ротации плюс grace period в 5–10 минут, компенсирующий рассинхронизацию часов между серверами (clock skew). По истечении этого срока публичный ключ удаляется из массива `keys`, и любой токен, подписанный соответствующим приватным ключом, становится неверифицируемым — однако к этому моменту все такие токены уже гарантированно истекли по полю `exp`.

### 5. Утилизация приватного ключа (Destruction)

Старый приватный ключ должен быть выведен из эксплуатации немедленно после переключения подписания на новый ключ — с этого момента он не используется ни для каких операций и представляет собой исключительно объект риска.

**Плановая утилизация** зависит от способа хранения. В KMS статус ключа изменяется на `Disabled`, что блокирует любые криптографические операции с ним, а спустя определённый период (согласно политике облачного провайдера) ключ переводится в состояние `Pending Deletion` и окончательно уничтожается — этот механизм реализует концепцию криптографического уничтожения (cryptographic erasure). При файловом хранении удаление через стандартную команду `rm` недостаточно, поскольку данные остаются на диске до перезаписи секторов файловой системой — вместо этого необходимо использовать утилиты безопасного затирания (shred в Linux), многократно перезаписывающие содержимое файла случайными данными перед удалением.

**Аварийная утилизация при компрометации** требует принципиально иного подхода, поскольку злоумышленник, владеющий приватным ключом, способен генерировать валидные токены неограниченно. Первый шаг — немедленный отзыв ключа в KMS или безопасное удаление файла. Второй — формирование и публикация обновлённого JWKS, из которого удалён публичный ключ, соответствующий скомпрометированному приватному. Третий — принудительная инвалидация всех активных сессий, поскольку естественное истечение срока действия токенов более не является надёжным механизмом защиты: злоумышленник мог уже выпустить токены с произвольным значением `exp`. Инвалидация реализуется через blacklist идентификаторов токенов (`jti`) в Redis или через принудительный re-login всех пользователей с обнулением refresh token.
