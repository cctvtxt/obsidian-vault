[[JWT-Based Auth]]

Внедрение **JWKS (JSON Web Key Set)** — это стандартный способ решения проблемы ротации ключей в микросервисной архитектуре. Он позволяет сервисам-потребителям проверять подписи токенов, не имея прямого доступа к секретам сервера авторизации. Этот подход описан в стандарте **RFC 7517** и превращает статичный процесс проверки подписи в динамическую и масштабируемую систему управления ключами.

JWKS **предназначен исключительно для публичного использования** и должен содержать только **открытые ключи**. Приватные ключи никогда не должны быть в нем опубликованы. Для проверки подписи JWT независимым сервисам нужен именно публичный ключ — приватный ключ применяется только тем, кто подписывает токены. Приватный же ключ ротируют и хранят безопасно на стороне сервиса, выдающего JWT-токены авторизации. 

**Проблема ротации публичных ключей** заключается в том, что в распределённых системах JWT-токены используются для аутентификации и авторизации между сервисами. Подписывающий сервис (Identity Provider, Authorization Server) хранит приватный ключ и публичные ключи и периодически меняет их (ротация) по соображениям безопасности. Потребители токенов (микросервисы, API‑шлюзы) должны проверять подписи с помощью публичных ключей. **Если потребитель хранит статическую копию публичного ключа или конфиг с ключами, ротация приведёт к несовместимости**: новые токены подписаны новым ключом и валидатор их отвергает; или, наоборот, старые токены перестают проверяться до их истечения — это ломает сессии и межсервисные вызовы. Передача приватного ключа потребителям недопустима в соображениях безопасности. Централизованная синхронизация ключей вручную — ненадёжна и тяжела в эксплуатации при большом количестве сервисов и окружений (dev/stage/prod). Временные окна (grace periods) и кеширование ключей добавляют сложности: нужно поддерживать проверку как старых, так и новых ключей одновременно, корректно обрабатывать отозванные ключи и предотвращать replay‑атаки.

JWKS решает эту проблему, предоставляет публичный набор ключей в стандартизированном JSON‑формате, доступный по HTTPS URL; потребители, независимо производящие авторизацию сервисы с JWKS могут периодически или при ошибке запрашивать этот URL и выбирать ключ по `kid`. Это позволяет разделить приватную сторону (только у IdP) и публичную (доступна всем валидаторам) без передачи секретов и с минимальной координацией между сервисами.

# Механизм ротации ключей

Ротация ключей представляет собой двухфазный процесс, минимизирующий риск компрометации системы при регулярной смене секретов доступа. Первоначально в JWKS добавляется новый публичный ключ, при этом подписание токенов по-прежнему осуществляется старым, что гарантирует непрерывность обслуживания клиентов. После завершения обновления кэша на всех сервисах, система переходит на использование нового ключа для генерации JWT, оставляя старый ключ доступным в течение всего периода жизни уже выпущенных токенов. Таким образом, старый ключ удаляется из JWKS только по истечении срока действия последнего токена, подписанного им, с учетом времени на рассинхронизацию часов и задержки в обновлении кэшей.

## Взаимодействие с внешним IdP

Интеграция с внешним Identity Provider, например Google или иными OAuth 2.0 сервисами, требует автоматизированного получения публичных ключей через стандартный протокол OIDC и его эндпоинт `.well-known/jwks.json`. Система обязана периодически опрашивать данный ресурс, чтобы динамически обновлять свой локальный набор доверенных ключей без вмешательства администратора. При этом нет нужды запрашивать ключи вручную, так как большинство современных библиотек для работы с JWT поддерживают автоматический JWKS-клиент, эффективно управляющий жизненным циклом ключей внешнего поставщика. Подобная реализация избавляет от необходимости жесткого хранения секретов, перекладывая управление безопасностью на надежного внешнего партнера.

# Структура

JWKS — это JSON-объект, содержащий массив `keys`, где каждый элемент является объектом **JSON Web Key (JWK)**.

```json
{
  "keys": [
    {
      "kty": "RSA",
      "use": "sig",
      "kid": "1",
      "alg": "RS256",
      "n": "...",
      "e": "AQAB"
    },
    {
      "kty": "RSA",
      "use": "sig",
      "kid": "2",
      "alg": "RS256",
      "n": "...",
      "e": "AQAB"
    }
  ]
}
```
Ключевое поле — `kid` (Key ID), которое позволяет получателю токена идентифицировать правильный ключ для проверки подписи.

# Архитектура с JWKS

JWKS встраивается как **публичный эндпоинт** (обычно по пути `/.well-known/jwks.json`), который предоставляет сервис авторизации.

**Auth-сервер (Издатель)**  
Генерирует пары ключей (например, RSA/ECDSA), подписывает JWT приватным ключом, а публичные ключи (в формате JWK) выставляют в виде JSON-массива по указанному URL.

**Микросервисы (Потребители)**  
Вместо хранения статического публичного ключа, сервис при старте или по расписанию скачивает JWKS с Auth-сервера, кэширует его в памяти и использует для проверки подписи входящих токенов.

# Процесс валидации токена

Процесс проверки подписи на стороне API-сервиса выглядит следующим образом:

1. **Получение JWT.** API-сервис получает токен, содержащий в заголовке поле `kid` (Key ID), которое указывает, какой именно ключ из набора был использован для подписи.
2. **Поиск ключа.** Сервис обращается к своему локальному кэшу JWKS, чтобы найти ключ с нужным `kid`. Если ключ не найден, сервис принудительно обновляет кэш, сделав запрос к JWKS-эндпоинту.
3. **Верификация подписи.** Используя найденный публичный ключ, сервис проверяет целостность подписи токена.
4. **Валидация claims.** После успешной проверки подписи выполняются стандартные проверки утверждений: `iss` (издатель), `aud` (аудитория), `exp` (срок действия), `nbf` (не раньше чем) и `iat` (время создания).
5. **Проверка прав.** Только после всех проверок можно доверять содержимому токена и проверять права доступа (`scope`, `roles`).

# Сценарии применения

##### Распределенная валидация токенов
Распределенная модель предполагает независимую проверку подлинности JSON Web Token (JWT) каждым микросервисом посредством обращения к единому источнику публичных ключей. Внедрение JWKS в такой системе обязательно, ибо оно исключает жесткое кодирование ключей, обеспечивая гибкое управление ими без необходимости пересборки приложений. Каждый микросервис реализует локальный кэш JWKS, который периодически обновляется для получения актуальных данных о ключах, что позволяет системе эффективно обрабатывать изменения в инфраструктуре безопасности. Напротив, отсутствие централизации требует строгой дисциплины в синхронизации времени и политик валидации `iss` (issuer) на каждом узле. Подобный подход минимизирует зависимость от шлюза, однако значительно усложняет конфигурацию отдельных компонентов системы.

##### Централизованная проверка шлюзом
Централизованная архитектура переносит бремя аутентификации на API Gateway, который валидирует JWT при входе в систему и транслирует очищенные данные в заголовках запросов. В такой схеме остальные микросервисы выполняют исключительно авторизацию на основе метаданных токена, что снижает нагрузку на сеть и упрощает управление политиками доступа. Реализация JWKS остается необходимой на уровне шлюза, так как именно он несет ответственность за криптографическую проверку подписи токенов, выдаваемых Identity Provider (IdP). Данный метод обеспечивает высокую степень безопасности, поскольку логика обработки токенов сосредоточена в одной точке, доступ к которой жестко контролируется.

При этом даже при наличии базы данных для хранения метаданных пользователей или рефреш-токенов, Auth Service должен выступать в роли доверенного поставщика ключей через стандартный JWKS-интерфейс. Это гарантирует, что микросервисы, отвечающие за валидацию токенов, всегда имеют актуальный набор публичных ключей для обеспечения безопасности запросов, не вступая в конфликт с требованиями централизованного управления доступом

**Эндпоинт JWKS должен реализовывать каждый инстанс сервиса, ответственного за подписание токенов**, обеспечивая доступ к актуальным публичным ключам, хранящимся в централизованном защищенном хранилище. При наличии нескольких инстансов балансировщик нагрузки распределяет запросы к этому эндпоинту между ними, обеспечивая высокую доступность и отказоустойчивость сервиса ключей.

# Преимущества использования JWKS

**Бесшовная ротация ключей**  
Вы можете менять приватные ключи на Auth-сервере (например, раз в месяц), добавляя новые публичные ключи в JWKS. В процессе ротации в наборе временно существуют и старый, и новый ключи, что позволяет без остановки сервиса проверять как старые токены, так и новые.

**Масштабируемость и отказоустойчивость**  
Микросервисы работают автономно, периодически обновляя набор ключей в фоне. Это исключает зависимость от Auth-сервера в момент каждого запроса. Если Auth-сервер временно недоступен, проверка токенов продолжается с использованием кэшированных ключей.

**Разделение ответственности (Decoupling)**  
Командам не нужно координировать друг с другом безопасную передачу публичных ключей. Микросервис просто знает URL эндпоинта JWKS и самостоятельно забирает актуальные данные для верификации.

## Хранение и публикация на сервере

Сервер авторизации публикует JWKS по стандартному, хорошо известному URL-адресу. Обычно это `/.well-known/jwks.json`.

1. **Генерация**: Сервер генерирует одну или несколько пар криптографических ключей (приватный и публичный). Приватные ключи сохраняются в безопасном хранилище.
2. **Формирование JWKS**: Из публичных ключей формируется JSON-структура. Каждый ключ снабжается уникальным `kid`.
3. **Публикация**: JWKS размещается на эндпоинте, доступном по HTTPS. Использование TLS обязательно для защиты от атак типа "человек-по-середине" (MITM).
4. **Кэширование**: Сервер должен использовать HTTP-заголовки кэширования (`Cache-Control`, `ETag`), чтобы клиенты (Resource Servers) не запрашивали ключи при каждой валидации токена.
5. **Хранение:** При реализации масштабируемой системы с несколькими экземплярами сервиса, ответственного за подписание токенов, JWK должны храниться не локально, а в специализированных защищенных хранилищах секретов (например, HashiCorp Vault, AWS Secrets Manager, Google Secret Manager). Использование подобных инструментов обеспечивает централизованное управление жизненным циклом криптографических пар, исключая риск несанкционированного доступа к закрытым ключам и упрощая их безопасную ротацию.

Такая архитектура обеспечивает централизованное управление ключами и позволяет осуществлять их ротацию без необходимости переконфигурации всех клиентов.

При развертывании нескольких экземпляров сервиса, ответственного за подписание токенов, целесообразно использовать подход, при котором каждый инстанс при старте и периодически в процессе работы **опрашивает централизованное хранилище секретов** для обновления своего локального кэша активных ключей. Это гарантирует, что любой экземпляр сервиса, получивший запрос на выдачу токена, будет использовать одну и ту же пару ключей, что необходимо для согласованной работы всей микросервисной системы.

## Обработка на клиенте (Resource Server)

Клиент, которому необходимо проверить JWT, выполняет следующие шаги:
1. Извлекает `kid` из заголовка JWT.
2. Находит в своем кэше или запрашивает JWKS с сервера авторизации.
3. Ищет в массиве `keys` объект JWK с совпадающим `kid`.
4. Использует найденный публичный ключ для проверки подписи JWT.

Если ключ с нужным `kid` не найден в кэше, клиент должен обновить свой кэш, повторно запросив JWKS. Это позволяет бесшовно обрабатывать ротацию ключей на сервере.
