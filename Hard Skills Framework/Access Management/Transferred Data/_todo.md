---

### 🔐 1. Современные защиты OAuth 2.1 и OIDC

Эти элементы сейчас становятся стандартом для защиты от перехвата токенов и манипуляций:

- **DPoP Token (Demonstrating Proof-of-Possession)** — _Критически важно!_ В отличие от Bearer-токенов, DPoP привязывает токен доступа к конкретному клиенту через криптографический ключ. Это полностью нейтрализует угрозу утечки и повторного использования Access Token злоумышленником.
- **PAR Request URI (Pushed Authorization Request)** — Референс на запрос авторизации, отправленный заранее через защищенный back-channel. Защищает от подделки параметров authorization request (например, подмены redirect_uri).
- **Device Code / User Code** — Если у вас есть сценарии авторизации устройств без браузера (IoT, Smart TV, CLI-утилиты) по flow Device Authorization Grant.
- **CIBA Auth Request / Token** — Для сценариев Client-Initiated Backchannel Authentication (когда авторизация происходит вне браузера, например, push-уведомление в банковском приложении).

### 🛡 2. Passwordless и FIDO2 / WebAuthn

Если вы планируете или уже используете аутентификацию без паролей, вам нужно документировать структуры данных WebAuthn:

- **Authenticator Data** — Данные, возвращаемые аппаратным ключом/биометрией, содержащие флаг присутствия пользователя и флаг верификации пользователя.
- **Client Data JSON (CollectedClientData)** — Данные о контексте операции (тип операции, origin, challenge), которые клиент отправляет для защиты от phishing.
- **Attestation Object / Statement** — Данные для верификации подлинности самого аутентификатора (важно для корпоративных сценариев, где разрешены только ключи определенного производителя).

### 🔗 3. Целостность API-запросов и Webhook

Access Token защищает ресурс, но не защищает от подделки тела запроса или replay-атак на уровне API:

- **HTTP Message Signature** — Заголовки для криптографической подписи HTTP-запросов (RFC 9421). Позволяет подписывать не только URL, но и тело запроса, защищая от MITM и модификации данных в пути.
- **HMAC Secret / Shared Secret** — Секретный ключ для подписи webhook-уведомлений или взаимодействия между микросервисами, где использование полноценных JWT избыточно.
- **Request Timestamp / JTI (JWT ID)** — Обязательно для защиты от replay-атак. JTI должен документироваться как уникальный идентификатор токена, который проверяется на уникальность в окне времени.

### 🏢 4. Контекст, Risk-Based Access и Multi-tenancy

Современный контроль доступа редко бывает бинарным ("доступен/недоступен"). Он учитывает контекст:

- **Tenant ID / Organization ID** — Критически важный элемент для B2B/SaaS приложений. Должен передаваться и валидироваться на каждом шаге, чтобы избежать уязвимостей горизонтального доступа между тенантами.
- **AMR (Authentication Method Reference) и ACR (Authentication Context Class Reference)** — Массивы, указывающие, _как именно_ был аутентифицирован пользователь (например, `["pwd", "mfa"]` или `["fido2"]`). Нужны для step-up аутентификации (когда для доступа к чувствительным данным требуется повторная аутентификация более строгим методом).
- **Device Fingerprint / Risk Signals** — Хэш характеристик устройства или оценка риска от системы Fraud/Risk management, передаваемые при аутентификации для принятия решения о выдаче токена.

### 🧩 5. SAML и Enterprise SSO (если применимо)

У вас есть файл `Assertions.md`, но для полноты SAML-интеграций стоит выделить:

- **NameID / Subject Identifier** — Формат идентификатора пользователя в SAML-ассерциях (например, transient, persistent, email).
- **Entity ID / Issuer Identifier** — Уникальный идентификатор IdP и SP, критичный для валидации доверия.
- **SAML Response / Request** — Как отдельные сущности, так как их структура и требования к подписи отличаются от OIDC.

### 🍪 6. Безопасность сессий на транспорте

Вы описали `Session ID.md`, но **безопасность сессии определяется не только ID, но и атрибутами его доставки**:

- **Cookie Security Attributes** — Документируйте требования к флагам: `Secure`, `HttpOnly`, `SameSite=Strict/Lax`, а также новейший `Partitioned` (CHIPS) для кросс-сайтового контекста. Без этих атрибутов Session ID уязвим к XSS и CSRF.

Однако в современной «безопасной настройке контроля доступа» приложение часто обменивается и другими, не менее критичными, данными. Вот что стоит рассмотреть, чтобы сделать документацию полнее.

### 1. Идентификаторы и ключи клиента

- **Client ID** (идентификатор клиента) – публичный строковый идентификатор, передаётся в URL редиректов, в теле запросов к /token.
    
- **Client Secret** (секрет клиента) – конфиденциальная строка для аутентификации конфиденциальных клиентов. Хотя у вас есть общий `Credentials.md`, эти два объекта настолько фундаментальны, что заслуживают отдельных описаний (особенно требования к хранению, передаче через `client_secret_post` / `client_secret_basic`).
    

### 2. Параметры запроса авторизации

- **Redirect URI (`redirect_uri`)** – один из важнейших параметров безопасности; передаётся при инициации потока и должен строго валидироваться.
    
- **Scope** (`scope`) – список разрешений, который передаётся в запросе авторизации и включается в токены.
    
- **Audience** (`aud`) – целевой ресурс токена, критичен при многокомпонентных API, передаётся в запросе (часто через параметр `audience`) и в JWT-токенах.
    

### 3. Современные расширения защиты токенов

- **DPoP Proof JWT** (Demonstration of Proof-of-Possession) – JWT, передаваемый в HTTP-заголовке `DPoP`, доказывающий владение приватным ключом, связанным с access/refresh токенами. Заменяет устаревший Token Binding, активно внедряется.
    
- **Token Binding ID** (может быть историческим, но ещё встречается в корпоративных системах) – идентификатор привязки токена к TLS-соединению.
    

### 4. Дополнительные артефакты OAuth/OIDC

- **Request Object (JWT)** – подписанный объект запроса, передаваемый в параметре `request` или `request_uri`; используется для безопасной передачи параметров авторизации.
    
- **Device Code** и **User Code** – для Device Authorization Grant (RFC 8628): тикеты, которыми обмениваются устройство и пользователь.
    
- **Logout Token** – JWT, отправляемый провайдером на back-channel logout URI при RP-initiated logout или других событиях выхода.
    
- **ID Token Hint** – параметр, передаваемый при выходе или аутентификации для указания текущей сессии.
    
- **Login Hint** – подсказка провайдеру о том, какого пользователя аутентифицировать.
    

### 5. Специфика SAML (если у вас детально)

У вас есть `Assertions.md` и `Metadata.md`. Типично вместе с ними передаются:

- **SAML Response** (целое сообщение, содержащее assertion).
    
- **SAML Request** (AuthnRequest).
    
- **RelayState** – механизм сохранения состояния (аналог OAuth `state`), передаваемый туда-обратно для маршрутизации после аутентификации.
    

### 6. Транспортные механизмы с атрибутами безопасности

- **Cookie** (Session Cookie / Auth Cookie) – хотя `Session ID.md` есть, сама кука как контейнер имеет критичные атрибуты безопасности: `Secure`, `HttpOnly`, `SameSite` (Lax/Strict/None). Передача этих флагов — часть настройки контроля доступа.
    
- **Bearer Token** – обобщённая схема передачи токена в заголовке Authorization. Может быть выделена как отдельный элемент с правилами формирования и защиты.
    

### 7. Альтернативные доказательства владения

- **Macaroons** / **Macaroon Caveats** – децентрализованные токены с контекстными ограничениями (используются в системах с распределённой авторизацией).
    
- **Signed HTTP Messages (HTTP Message Signatures)** – подпись запроса, передающая подтверждения владения ключом без токенов (например, для server-to-server).
    

### 8. WebAuthn / FIDO2 артефакты (если контроль доступа включает passwordless)

- **PublicKeyCredential** (содержит id, rawId, response с authenticatorData и signature).
    
- **Attestation Object** и **Assertion Object** – если двухфакторная аутентификация реализована на стороне приложения.
