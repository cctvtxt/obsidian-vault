[[Token-Based Auth]]

Классический **CSRF** обычно использует сессионные cookies, так как браузер всегда их прикрепляет к запросам на домен, однако **CSRF**-атака возможна при любой **автоматической** передаче аутентификационных данных браузером, которые сложно подделать извне. При CSRF вредоносный сайт заставляет браузер жертвы отправить нежелательный запрос на уязвимый сайт, под доменом которого пользователь в настоящий момент авторизован 

Anti-CSRF-токен защищает от CSRF, так как добавляет в процесс аутентификации информацию, которую вредоносный сайт самостоятельно не может генерировать.
__________
Если API соответствует RFC7231, то методы `GET`, `HEAD`, `OPTIONS` и `TRACE` являются безопасными: они предназначены только для получения информации и не должны изменять состояние сервера. Защищать необходимо небезопасные методы, к которым относятся: `POST`, `PUT`, `DELETE`, `PATCH`.

Во-вторых, Anti-CSRF токен (synchronizer token) необходимо **хранить и передавать отдельно от аутентификационных данных**, чтобы злоумышленник не мог получить его так же, как и данные аутентификации
# Требования к токену

- Уникальный токен для каждой операции
- В идеале действует единожды
- Имеет размер, устойчивый к подбору
- Сгенерирован криптографически стойким генератором псевдослучайных чисел
- Имеет ограниченное время жизни
# Места внедрения на клиенте

Anti-CSRF токен может быть отправлен на клиент с ответом внутри заголовка Set-Cookie и добавлен в **non-HttpOnly cookie** (HttpOnly-куки недоступны коду приложения, которое должно обрабатывать Anti-CSRF токен)
```http 
HTTP/1.1 200 OK 
Set-Cookie: jwt_access=eyJ...; HttpOnly; Secure; SameSite=Strict; Path=/api 
Set-Cookie: csrf_token=abc123; HttpOnly=false; Secure; SameSite=Strict; Path=/ 
Set-Cookie: jwt_refresh=eyJ...; HttpOnly; Secure; SameSite=Strict; Path=/auth/refresh
```

Anti-CSRF токен может быть отправлен на клиент внутри body ответа и быть размещенным в **localStorage** или **sessionStorage**, откуда по умолчанию данные не передаются вместе с запросом, поэтому токен не сможет быть отправленным с вредоносного сайта 

Anti-CSRF токен может быть отправлен на клиент при генерации HTML-документа и размещен в **meta-теге** как `<meta name="csrf-token" content="abc123">` или в **скрытом поле формы** отправки данных пользователем
```html
<input type="hidden" name="_csrf" value="abc123def456789">
```
# Передача на сервер с запросом 

Anti-CSRF токен целесообразно отправлять в специальном заголовке для того, чтобы не загрязнять body
```json
"headers": { "X-CSRF-Token": csrfToken }
```
# Обработка и хранение на сервере

Anti-CSRF токен генерируется сервером с использованием криптографически стойкого генератора случайных чисел (CSPRNG), чтобы обеспечить непредсказуемость, его размер минимум 32 байта для устойчивости к brute-force

При **steteful**-аутентификации пользователей токен целесообразно помещать **рядом с session-id в БД** и проверять доставая из базы данные при входящих запросах с возможной заменой его на новый после использования

При **stateless**-аутентификации целесообразно генерировать токены, основанные аутентификационных данных

Проверка валидности токена может быть встроена как middleware при обработке входящих запросов
