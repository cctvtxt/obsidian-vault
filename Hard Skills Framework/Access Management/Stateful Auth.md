[[Authentification]]

**Stateful**-аутентификация необходима для сценариев, где сервер должен точно отслеживать состояние сессии каждого пользователя, храня данные о логине, что обеспечивает **_повышенную безопасность через серверный контроль, защиту от подделки сессий и удобство для приложений с долгоживущими сессиями_**.

Состояние — это набор данных, описывающих текущий статус системы, компонента или приложения в определённый момент времени. Например: содержимое корзины покупок в интернет‑магазине, данные авторизации пользователя, прогресс выполнения многошаговой операции.

При этом **TCP** является **stateful**-протоколом, поскольку поддерживает состояние соединения (handshake, последовательность пакетов, подтверждения). **HTTP** же **stateless** — каждый запрос независим и не сохраняет контекст предыдущих. При независимости отдельных запросов, HTTP-транспорт предоставляет возможность передачи данных для **stateful**-аутентификации, все данные о сессиях сохраняются в базе данных сервера. 

При этом **аутентификация по сеансу не является стандартным методом для REST API**. По умолчанию REST API не поддерживает информацию о сессиях: сервер не хранит данные входа клиента, каждый запрос действует независимо, проверяя себя без зависимости от прошлых взаимодействий. Для сессионной аутентификации предпочтительны другие подходы и инструменты, не привязанные строго к чистому REST, чтобы избежать конфликта со stateless-принципом. Например, это GraphQL с контекстом, gRPC или WebSocket.
______________
Примером реализации данного подхода аутентификации является Session-Based-аутентификация, кроме этого, существуют различные session-flow для OAuth 2.0 
___
**Аутентификация на основе сессий подходит для небольших и средних проектов**, где важна простота и скорость разработки, а также для приложений, где требуется централизованное управление сессиями. **Для больших приложений** аутентификация на основе сессий может быть не очень подходящей — она хранит данные сессии на сервере, что может увеличить нагрузку и сложность при большом количестве пользователей
## Область применения

**Stateful**-аутентификация обычно используется в приложениях, где пользователь долго работает в интерфейсе, а сессию после этого необходимо надёжно разлогинить по таймауту или принудительно (через админ‑панель). В таких случаях сервер хранит сессию и решает, активен ли конкретный пользователь.

**Централизованное управление сессиями** — сервер полностью контролирует сессии, что упрощает их инвалидацию и управление данными пользователя.  
**Общая простота реализации** — простота выполнения всех этапов аутентификации, менеджмента сессий.
**Отмена сессий** — сервер может мгновенно аннулировать скомпрометированную сессию, удалив её из хранилища.
## Производительность

- **Высокое потребление ресурсов**. Каждая сессия занимает ресурсы сервера, что снижает производительность при увеличении числа пользователей.
- **Задержка в распределённой системе**. Поддержание данных сессии на нескольких серверах требует синхронизации хранилища сессий, что вызывает дополнительную задержку, потому что сервер должен проверять хранилище сессий каждый раз, когда выполняется запрос.
- **Сложность для мобильных приложений**. Аутентификация на основе сессий хранит данные сессии на сервере, что может увеличить нагрузку, и использует cookie, которые сложнее обрабатывать на мобильных устройствах.
## Масштабируемость

- **Stateful-природа аутентификации на основе сессий может быть проблемой для горизонтального масштабирования** — требуется общее хранилище сессий (например, Redis, база данных) для всех экземпляров сервера. Это вызывает дополнительную задержку, потому что сервер должен проверять хранилище сессий каждый раз, когда выполняется запрос.
- **Сложность для распределённых систем**. При использовании микросервисов проблема усугубляется — каждый сервис должен как-то проверять валидность сессии.
- **Сложность для сайтов, использующих несколько доменных имён**. Cookies привязывается к определённому домену, и сложно поддерживать такую авторизацию для сайтов с несколькими доменами.
- Кроме того, **пользователь оказывается жёстко привязанным к устройству**, в котором в cookies хранится идентификатор сессии. Попытка зайти на сайт с другого устройства (если там нет актуальной записи в cookies) будет безуспешной — нужно будет заново вводить логин и пароль

## Безопасность

- **Зависимость от функционирования сервера**. Аварии на сервере при хранении сессий в оперативной памяти приводят к потере данных сессий пользователей
- **Риск похищения сессий**. Похищение сессий (например, через украденные cookie) может позволить получить несанкционированный доступ к учётным записям 
- **Уязвимости для атак**. Токены аутентификации хранятся как на фронтенде, так и в бэкенде, часто отправляются по сети, поэтому уязвимы для нескольких типов атак
  - **Атаки XSS** — злоумышленник может внедрить JavaScript-код в приложение, запущенное в браузере жертвы, и прочитать и передать токены аутентификации злоумышленнику.
  - **CSRF** — атака позволяет злоумышленнику использовать текущую активную сессию. Для предотвращения таких атак обычно требуется использование анти-CSRF-токена или SameSite cookie.
- **Жёсткая привязка к устройству**. Пользователь оказывается привязанным к устройству (браузеру), в котором в cookies хранится идентификатор сессии. Попытка зайти на сайт с другого устройства (если там нет актуальной записи в cookies) будет безуспешной — нужно будет заново вводить логин и пароль
