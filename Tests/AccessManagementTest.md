# Тест по Управлению Доступом (Access Management)

#flashcards-test

Что такое сессионная аутентификация?::Классический подход к управлению состоянием пользователя на сервере с использованием cookie для хранения идентификатора сессии.
<!--SR:!2026-03-28,2,241-->
Где сервер хранит сессию при сессионной аутентификации?::В памяти или базе данных.
<!--SR:!2026-03-30,3,258-->
Как Session ID передается клиенту?::В cookie.
<!--SR:!2026-03-28,2,230-->
Какой главный недостаток сессионной аутентификации при горизонтальном масштабировании?::Необходимость в общем хранилище сессий (shared storage), например Redis.
<!--SR:!2026-03-30,3,258-->
К какой уязвимости потенциально подвержена сессионная аутентификация?::CSRF (Cross-Site Request Forgery).
<!--SR:!2026-03-28,1,218-->
Как можно решить проблему уязвимости к CSRF в сессионной аутентификации?::С помощью CSRF-токенов и атрибутов cookie SameSite.
<!--SR:!2026-03-28,2,244-->
В каком случае аутентификация на основе сессий является предпочтительной?::Когда требуется контроль сеансов в реальном времени, централизованное управление или масштабируемость не является основной проблемой.
<!--SR:!2026-03-30,3,258-->
Почему аутентификация на основе сессий не очень подходит для мобильных приложений?::Она хранит данные сессии на сервере, что может увеличить нагрузку и сложность, а также cookie сложнее обрабатывать на мобильных устройствах.
<!--SR:!2026-03-28,1,204-->
Что происходит при выходе пользователя из системы при сессионной аутентификации?::Сервер удаляет сессию.
<!--SR:!2026-03-30,3,258-->
Где целесообразнее всего хранить состояние активных сессий?::В in-memory БД.
<!--SR:!2026-03-29,3,261-->
Что такое базовая аутентификация (Basic Auth)?::Простейший HTTP-метод аутентификации, при котором клиент передает логин и пароль при каждом запросе.
<!--SR:!2026-03-29,2,238-->
В каком виде передаются данные при Basic Auth?::В кодировке Base64 в заголовке Authorization.
<!--SR:!2026-03-30,3,258-->
Какой главный недостаток Basic Auth?::Низкий уровень безопасности, так как учетные данные легко перехватить без шифрования HTTPS/TLS.
<!--SR:!2026-03-29,3,261-->
В каких случаях обычно используется Basic Auth?::Во внутренних, защищенных средах или для простых тестовых API.
<!--SR:!2026-03-30,3,244-->
Какой HTTP статус возвращает сервер при неудачной базовой аутентификации?::401 Unauthorized.
<!--SR:!2026-03-28,1,201-->
Что добавляет Digest Authentication по сравнению с Basic Auth для повышения безопасности?::Хеширование и одноразовый `nonce`, чтобы пароль не передавался в открытом виде.
<!--SR:!2026-03-28,1,218-->
Почему Digest Auth редко используется в современных REST API?::Протокол сложнее, опирается на устаревшие алгоритмы.
<!--SR:!2026-03-29,2,238-->
Что такое JWT?::JSON Web Token - открытый стандарт для создания токенов доступа, основанный на формате JSON.
<!--SR:!2026-03-30,3,258-->
Из каких трех частей состоит JWT?::Header (заголовок), Payload (полезная нагрузка) и Signature (подпись).
<!--SR:!2026-03-29,2,224-->
Какая часть JWT гарантирует целостность токена?::Signature (подпись).
<!--SR:!2026-03-30,3,258-->
Нужно ли серверу хранить сессию при использовании JWT?::Нет, JWT-аутентификация является stateless.
<!--SR:!2026-03-30,3,258-->
Как решается проблема отзыва JWT до истечения его срока действия?::Использованием короткого времени жизни (TTL) и ведением черного списка (blacklist) отозванных токенов.
<!--SR:!2026-03-28,2,230-->
Где на клиенте можно хранить JWT?::В localStorage, sessionStorage или в cookie.
<!--SR:!2026-03-30,3,258-->
Какая уязвимость возникает при хранении JWT в localStorage?::XSS (Cross-Site Scripting).
<!--SR:!2026-03-29,2,238-->
Какая уязвимость возникает при хранении JWT в cookie без атрибута HttpOnly?::XSS (Cross-Site Scripting).
<!--SR:!2026-03-28,2,241-->
Какая уязвимость возникает при хранении JWT в cookie?::CSRF (Cross-Site Request Forgery), если не использовать атрибут SameSite.
<!--SR:!2026-03-28,2,241-->
Какой атрибут cookie делает его недоступным для клиентского JavaScript?::HttpOnly.
<!--SR:!2026-03-29,2,238-->
Какой атрибут cookie гарантирует его передачу только по HTTPS?::Secure.
<!--SR:!2026-03-28,1,201-->
Что такое "access token" в контексте JWT?::Короткоживущий токен, который проверяется при каждом обращении к защищенному API.
<!--SR:!2026-03-28,2,244-->
Что такое "refresh token" в контексте JWT?::Долгоживущий одноразовый токен для получения новой пары access и refresh токенов.
<!--SR:!2026-03-30,3,258-->
Какое основное преимущество JWT в микросервисной архитектуре?::Каждый сервис может самостоятельно проверить токен без обращения к центральному хранилищу сессий.
<!--SR:!2026-03-30,3,258-->
Что такое "claims" в JWT?::Утверждения (данные) о пользователе, хранящиеся в payload, например, `userId`, `role`, `exp`.
<!--SR:!2026-03-28,1,218-->
Какой стандарт определяет JWT?::RFC 7519.
<!--SR:!2026-03-29,2,238-->
Как передается JWT в HTTP-запросе?::Обычно в заголовке `Authorization` со схемой `Bearer`.
<!--SR:!2026-03-30,3,258-->
Что такое аутентификация по ключу API (Key-Based Auth)?::Метод доступа к API с помощью уникального ключа (строки), сгенерированного для клиента.
<!--SR:!2026-03-30,3,258-->
Где может передаваться API ключ?::В заголовках, параметрах запроса или теле запроса.
<!--SR:!2026-03-30,3,258-->
В чем заключается основной недостаток API ключей?::Риск перехвата ключей, если они передаются по незащищенному каналу.
<!--SR:!2026-03-28,1,218-->
Что такое OAuth 2.0?::Протокол делегированной авторизации, позволяющий приложениям получать ограниченный доступ к ресурсам пользователя без получения его пароля.
<!--SR:!2026-03-30,3,258-->
Какие четыре роли определены в OAuth 2.0?::Resource Owner (пользователь), Client (приложение), Authorization Server (сервер авторизации), Resource Server (сервер ресурсов).
<!--SR:!2026-03-29,2,238-->
Что такое "scope" в OAuth 2.0?::Механизм для ограничения доступа приложения к определенным ресурсам пользователя.
<!--SR:!2026-03-28,2,244-->
Какой grant type в OAuth 2.0 является наиболее распространенным для веб-приложений?::Authorization Code Flow.
<!--SR:!2026-03-28,1,218-->
Что такое `redirect_uri` в OAuth 2.0?::URL, на который сервер авторизации перенаправляет пользователя после предоставления (или отказа в предоставлении) разрешения.
<!--SR:!2026-03-30,3,258-->
Что возвращает сервер авторизации в Authorization Code Flow после успешной авторизации пользователя?::Authorization Code (код авторизации).
<!--SR:!2026-03-28,1,218-->
На что клиент обменивает Authorization Code?::На Access Token и, опционально, Refresh Token.
<!--SR:!2026-03-29,2,238-->
Что такое `state` параметр в OAuth 2.0?::Случайная строка для защиты от CSRF-атак.
<!--SR:!2026-03-28,1,218-->
Что такое OpenID Connect (OIDC)?::Слой идентификации поверх OAuth 2.0, который позволяет клиентам проверять личность пользователя.
<!--SR:!2026-03-29,3,264-->
Какую основную проблему OAuth 2.0 решает OIDC?::OAuth 2.0 - это протокол авторизации, а не аутентификации. OIDC добавляет стандартный способ аутентификации.
<!--SR:!2026-03-30,3,258-->
Какой новый токен вводит OIDC?::ID Token.
<!--SR:!2026-03-29,2,238-->
В каком формате обычно представлен ID Token?::В формате JWT.
<!--SR:!2026-03-28,1,218-->
Какой обязательный `scope` должен запросить клиент для использования OIDC?::`openid`.
<!--SR:!2026-03-28,1,190-->
