# Access Data JWK

#flashcards/access-data/jwk

Что такое JSON Web Key (JWK) и каков его основной стандарт?
?
**JSON Web Key (JWK)** — это стандартный формат (RFC 7517) для представления криптографических ключей в виде JSON-объекта, позволяющий единообразно описывать ключи разных типов (RSA, EC) и их параметры.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key.md`.

Каково назначение полей `kty`, `use`, `kid` и `alg` в публичном JWK?
?
`kty` (Key Type) указывает тип ключа (например, "RSA"), `use` определяет назначение ("sig" для подписи, "enc" для шифрования), `kid` (Key ID) является уникальным идентификатором ключа, а `alg` закрепляет конкретный алгоритм (например, "RS256").
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key.md`.

Какова основная задача публичного JWK и как он распространяется?
?
Публичный JWK предназначен для верификации подписи JWT, подтверждая, что токен не был изменен. Он распространяется через JWKS endpoint (обычно `/.well-known/jwks.json`), откуда Resource Server'ы загружают и кэшируют ключи. Публичный ключ не является секретом.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key.md`.

Какие параметры отличают приватный JWK от публичного и каково их назначение?
?
Приватный JWK содержит секретные параметры: `d` (приватная экспонента) для создания подписи, `p` и `q` (простые числа, составляющие модуль `n`), а также `dp`, `dq`, `qi` для оптимизации вычислений по Китайской теореме об остатках.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key.md`.

Для каких двух основных операций используется приватный ключ в контексте JWT?
?
Приватный ключ используется для создания цифровой подписи JWT (JWS) и для расшифровки данных, зашифрованных соответствующим публичным ключом (JWE). Его компрометация означает полную компрометацию системы авторизации.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key.md`.

Какие существуют безопасные методы хранения приватных ключей JWK?
?
Наиболее надежные методы — системы управления секретами (HashiCorp Vault, AWS KMS), Hardware Security Modules (HSM). Допустимо файловое хранение с правами `chmod 400` или передача через переменные окружения из защищенных источников. Категорически запрещено хранить ключи в Git или незашифрованных конфигах.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key.md`.

Как связаны публичный и приватный ключи в потоке JWT-авторизации?
?
Они образуют неразрывную математическую пару: приватный ключ создает подпись, а публичный — её проверяет. В потоке JWT сервис авторизации подписывает токен приватным ключом и помещает `kid` в заголовок, по которому клиент находит соответствующий публичный ключ в JWKS для валидации.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key.md`.

Какова основная цель корректного управления жизненным циклом криптографических ключей в JWT-системе?
?
Корректное управление жизненным циклом ключей определяет как отказоустойчивость, так и безопасность системы. Ошибки при ротации могут привести к массовым сбоям авторизации, а ошибки при утилизации — оставить скомпрометированный материал в эксплуатации.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key.md`.

Как происходит безопасная генерация ключевой пары для JWT-авторизации?
?
Ключи генерируются с помощью криптографически стойкого генератора (CSPRNG), часто внутри защищенного периметра KMS или HSM. Каждой паре присваивается уникальный `kid`, обычно на основе хэша публичного ключа.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key.md`.

Что происходит на этапе активации нового ключа в системе JWT?
?
Новый публичный ключ добавляется в JWKS endpoint. Сервис авторизации переключается на подписание токенов новым приватным ключом и указывает соответствующий `kid` в заголовке JWT, по которому Resource Server находит ключ для валидации.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key.md`.

Опишите корректный и некорректный алгоритмы ротации ключей в JWT.
?
Корректно: сгенерировать новую пару, добавить новый публичный ключ в JWKS (сохранив старый), переключить подписание на новый приватный ключ. Некорректно: одновременно удалять старый публичный ключ при добавлении нового, так как это мгновенно инвалидирует все ранее выпущенные токены.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key.md`.

Как долго старый публичный ключ должен оставаться в JWKS после ротации?
?
Старый публичный ключ должен оставаться в JWKS до тех пор, пока не истечёт срок действия (`exp`) последнего токена, подписанного соответствующим старым приватным ключом, плюс дополнительный "grace period" для компенсации рассинхронизации часов.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key.md`.

В чем разница между плановой и аварийной утилизацией приватного ключа?
?
Плановая утилизация включает безопасное затирание (shred) или отключение ключа в KMS после истечения срока жизни токенов. Аварийная утилизация при компрометации требует немедленного отзыва ключа, удаления его публичной части из JWKS и принудительной инвалидации всех активных сессий.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key.md`.
