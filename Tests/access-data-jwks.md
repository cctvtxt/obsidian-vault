# Access Data JWKS

#flashcards/access-data/jwks

Что такое JSON Web Key Set (JWKS) и какую проблему он решает?
?
**JWKS (JSON Web Key Set)** — это стандартный способ (RFC 7517) публикации набора публичных криптографических ключей в формате JSON. Он решает проблему ротации ключей в микросервисной архитектуре, позволяя сервисам-потребителям динамически получать актуальные ключи для проверки подписей JWT без прямого доступа к секретам сервера авторизации.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key Set.md`.

Что должно и не должно содержаться в JWKS?
?
JWKS предназначен исключительно для публичного использования и должен содержать только открытые (публичные) ключи. Приватные ключи никогда не должны быть в нем опубликованы, так как они используются только стороной, подписывающей токены.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key Set.md`.

В чем заключается проблема ротации публичных ключей в распределенных системах и как JWKS ее решает?
?
Проблема в том, что при смене ключа на сервере авторизации, клиенты со статичными ключами перестают валидировать новые токены. JWKS решает это, предоставляя стандартизированный URL (`/.well-known/jwks.json`), откуда клиенты могут динамически загружать актуальный набор публичных ключей, выбирая нужный по `kid` из токена.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key Set.md`.

Опишите двухфазный процесс ротации ключей с использованием JWKS.
?
Сначала в JWKS добавляется новый публичный ключ, но подписание продолжается старым. После обновления кэшей на клиентах, система переключается на подписание новым ключом. Старый ключ остается в JWKS до истечения срока действия всех подписанных им токенов.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key Set.md`.

Как происходит интеграция с внешним IdP (например, Google) для получения ключей?
?
Интеграция требует автоматизированного получения публичных ключей через стандартный OIDC эндпоинт `.well-known/jwks.json`. Современные JWT-библиотеки поддерживают JWKS-клиент, который периодически опрашивает этот ресурс и автоматически управляет жизненным циклом ключей.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key Set.md`.

Какова структура JSON-объекта JWKS и каково назначение поля `kid`?
?
JWKS — это JSON-объект с массивом `keys`, где каждый элемент является JWK (JSON Web Key). Ключевое поле `kid` (Key ID) позволяет получателю токена идентифицировать, какой именно ключ из набора нужно использовать для проверки подписи.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key Set.md`.

Как JWKS встраивается в архитектуру микросервисов?
?
Сервер авторизации (издатель) предоставляет публичный эндпоинт (обычно `/.well-known/jwks.json`) с набором публичных ключей. Микросервисы-потребители периодически скачивают и кэшируют этот набор для валидации входящих токенов.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key Set.md`.

Опишите по шагам процесс валидации токена с использованием JWKS.
?
1. Получить JWT и извлечь `kid` из заголовка. 2. Найти ключ с этим `kid` в локальном кэше JWKS (или обновить кэш с эндпоинта). 3. Верифицировать подпись токена найденным публичным ключом. 4. Валидировать claims (`iss`, `aud`, `exp`). 5. Проверить права (`scope`, `roles`).
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key Set.md`.

В чем заключается сценарий распределенной валидации токенов с помощью JWKS?
?
В этой модели каждый микросервис независимо проверяет JWT, обращаясь к единому JWKS-эндпоинту и используя локальный кэш ключей. Это исключает жесткое кодирование ключей и минимизирует зависимость от API-шлюза.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key Set.md`.

В чем заключается сценарий централизованной проверки токенов через API Gateway?
?
API Gateway валидирует входящий JWT, используя JWKS, и передает информацию о пользователе и его правах дальше во внутренние сервисы в заголовках. Микросервисы доверяют шлюзу и выполняют только авторизацию.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key Set.md`.

Назовите три ключевых преимущества использования JWKS.
?
1. **Бесшовная ротация ключей:** возможность менять ключи без остановки сервиса. 2. **Масштабируемость и отказоустойчивость:** сервисы работают автономно с кэшированными ключами. 3. **Разделение ответственности:** командам не нужно координировать передачу ключей.
Источник: `Information Tecnologies/Access Management/Transferred Data/JSON Web Key Set.md`.

Как правильно организовать хранение и публикацию JWKS на сервере авторизации?
?
Приватные ключи хранятся в защищенном хранилище (Vault, KMS). Публичные ключи публикуются через HTTPS эндпоинт (`/.well-known/jwks.json`) с HTTP-заголовками кэширования. Каждый инстанс сервиса должен иметь доступ к централизованному хранилищу ключей.
