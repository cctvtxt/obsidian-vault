# Руководство по формированию Obsidian Vault 

Проект представляет собой базу знаний Obsidian, документ GEMINI.md содержит правила написания и структурирования заметок. 

1. Ключевой принцип — точность, формальность изложения, всестороннее раскрытие темы, всех сопутствующих понятий и сущностей. 
2. Отсутствие воды, домыслов, неопределенности, "возможно", "скорее всего" и подобного. 
3. Только обезличенные утверждения, описания фактов (запрещены "я", "мы", "вы", "представьте", "например", "допустим"). 

Создавай точные, осмысленные предложения с литературной красотой русского языка: плавный ритм, разнообразие синтаксиса (придаточные, однородные члены, инфинитивы), образные сравнения только для технических понятий (кластер как оркестр узлов). 

Не допускай генерацию артефактов в словах, не создавай иероглифы, символы из неподходящих кодировок. Не генерируй эмоджи. Пиши текст на русском языке с вставкой ключевых понятий на английском при необходимости. 

Для предложений. Длина: от 25 до 40 слов. Форма: составные или сложносочиненные, активный залог где возможно; добавление наречий, дополнительная связь предложений между собой через "напротив", "наоборот" и так далее, а также отрицательные и усилительные частицы "же" и "даже" . Пример: "OAuth 2.0 использует access token для авторизации API-запросов, где token с TTL в 3600 секунд обеспечивает временную авторизацию и минимизирует риски компрометации". 

Для абзацев. Длина: 4–6 предложений, 120–200 слов. Один абзац — одна подтема с плавным переходом к следующей через связующие фразы (таким образом, в связи с этим, напротив). Между абзацами — логический переход; при заданном шаблоне оглавления генерируемой заметки и резкой смене темы используй заголовок для перехода. Обеспечь элегантный поток текста, как в научной статье высокого уровня.
# Общие правила формирования заметок

1. `[[ссылка]]` ссылка: добавляется в начале файла, определяет ссылку на существующий файл внутри Obsidian Valut, который раскрывает родительское понятие по отношению к текущей заметке. Ссылка на файл может отсутствовать, если связь с родительской заметкой не задана в запросе на генерацию или ссылки отсутствуют при редактировании. Другие ссылки, кроме ссылки в начале файла, в тексте не допускаются. 
2. `сплошной текст` основное текстовое содержимое: всегда перед любым абзацем вставляй необходимо добавление пустой строки.

При формировании первого абзаца в файле запрещается начинать текст с наименования файла или понятия и описания его после вставки знака тире.

4. ` ` или `   ` или `\t` отступы перед фрагментами текста, табуляция: полностью запрещены, если данные знаки присутствуют в исходном тексте, выполняй редактирование
5. `#` заголовок первого уровня: максимум 3-5 на заметку, только для ключевых разделов. Если заметка типовая (находится в одном разделе-папке и раскрывает аналогичную тему), то заголовки в генерируемых заметках должны быть типовыми - необходимо проверять завершенные файлы и использовать аналогичную структуру. Перед заголовком никогда не добавляй пустую строку - знак заголовка ставится на следующей строке после текста, после заголовка добавляй пустую строку перед новым абзацем, перед таблицей не добавляй
6. `##` заголовок второго уровня: аналогичные правила как для `#`, но по смыслу - это подтема, используй реже
7. `###` заголовок третьего уровня: автоматическое добавление запрещено;
8. `*курсив*` выделение курсивом: автоматическая генерация запрещена; 
9. `**жирный**` выделение жирным: выделяй ключевые понятия, иногда словосочетания передающие ключевые идеи, которые являются самыми главными. Абзацы без заголовком допускаются; 
10. `~~зачеркнутый~~` зачёркивание: автоматическая генерация запрещена; 
11. `<u>подчёркнутый</u>` подчёркивание: автоматическая генерация запрещена; 
12. `***жирный курсив***` или `___жирный курсив___` жирный + курсив: автоматическая генерация запрещена; 
13. `` `код` `` моноширинный текст (inline code): используется для понятий, словосочетаний, которые встречаются в коде;
14. ```` ```блок кода ``` ```` блок кода: всегда используется в явным указанием языка программирования, пример кода должен быть 
15. `-` или `*` или `+` ненумерованный список: добавление запрещено;
16. `1.` нумерованный список: 
17. `> ` цитата: добавление запрещено;
18. `[текст](ссылка)` ссылка: добавление запрещено;
19. `![альт](изображение.jpg)` изображение: добавление запрещено;
20. `[текст][якорь]` ссылка на якорь: 
21. `[якорь]: https://example.com` определение якоря:
22. ``| Заголовок1 | Заголовок2 |`` таблица: формируется только при четком запросе о необходимости создания таблицы, для описания сгруппированных данных генерируй нумерованные списки или 
23. `---` или `***` горизонтальная линия: добавление запрещено;
24. `___` горизонтальная линия: для разделения разделов при резком изменении темы, когда добавление заголовка является плохим решением. Добавляется не более 3-х разделительных горизонтальных линий в файл
25. HTML-форматирование: разрешено только при ручном вводе.

# Уникальные правила для `.\NPM Registry`

Каталог предназначен для создания заметок, относящихся к 

**Полное Отсутствие Заголовков**: В файлах, описывающих NPM-библиотеки (например, стратегии Passport), заголовки (`#`) не используются вовсе. Весь текст должен представлять собой единое повествование, разделенное на абзацы и блоки кода.

**Структура Повествования**: Для NPM-библиотек текст должен последовательно раскрывать следующие аспекты, изложенные в свободной форме: основное назначение и принцип работы библиотеки, пример интеграции в виде блока кода, и развернутый анализ применимости, целесообразности, сравнение с альтернативами и важные аспекты использования.
# Уникальные правила для `.\Application Programming Interface\Regulatory Frameworks`

Раздел посвящен стандартам API

Первый абзац: краткое описание формата, в котором существует стандарт, описание специфики применения, того, в каких областях и в каких случаях необходимо соответствие стандарту (об этот не нужно упоминать, но речь должна идти именно о применении стандартов в проектах в РФ)

Далее: короткий абзац с определением задачи которую решает стандарт 

Далее: описание содержания стандарта - описание в виде одного абзаца связного повествовательного текста ключевых тем стандарта и-или областей, на которые накладывает ограничения стандарт, речь должна идти именно о главах-разделах документа, но поверхностно
