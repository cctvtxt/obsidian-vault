[[HTTP]]

Стандарт HTTP определяет систему аутентификации, которую сервер может использовать для проверки запроса клиента, а клиент — для предоставления информации для аутентификации.

Процесс вызова и ответа выглядит следующим образом:

1. Сервер отвечает клиенту статусом 401 (неавторизованно) и предоставляет информацию о том, как пройти авторизацию, с помощью заголовка **WWW-Authenticate** ответа, содержащего по крайней мере один запрос на авторизацию.
2. Клиент, который хочет пройти аутентификацию на сервере, может сделать это, добавив в запрос заголовок **Authorization** с учетными данными.
3. Обычно клиент запрашивает у пользователя пароль, а затем отправляет запрос с правильным заголовком **Authorization** .

Тот же механизм запроса и ответа можно использовать для аутентификации прокси. Поскольку аутентификация ресурсов и аутентификация прокси могут сосуществовать, требуется другой набор заголовков и кодов состояния. В случае с прокси-серверами используется код состояния **407** (требуется аутентификация прокси), заголовок **Proxy-Authenticate** ответа содержит по крайней мере один запрос, применимый к прокси-серверу, а заголовок **Proxy-Authorization** запроса используется для предоставления учетных данных прокси-серверу.

Если (прокси)сервер получает неверные учетные данные, он должен ответить **401** **Unauthorized** или **407 Proxy Authentication Required**, после чего пользователь может отправить новый запрос или заменить поле **Authorization** в заголовке.

Если (прокси-)сервер получает действительные учетные данные, которые не подходят для доступа к данному ресурсу, сервер должен ответить кодом состояния **403 Forbidden**. В отличие от **401 Unauthorized** или **407 Proxy Authentication Required**, аутентификация для этого пользователя невозможна, и браузер не предложит повторить попытку.

Во всех случаях сервер может предпочесть вернуть код состояния **404 Not Found**, чтобы скрыть существование страницы от пользователя без соответствующих привилегий или с неправильно выполненной аутентификацией.

Заголовки **WWW-Authenticate** и **Proxy-Authenticate** ответа определяют метод аутентификации, который следует использовать для получения доступа к ресурсу. В них необходимо указать, какая схема аутентификации используется, чтобы клиент, желающий пройти авторизацию, знал, как предоставить свои учетные данные. Заголовки **Authorization** и **Proxy-Authorization** запроса содержат учетные данные для аутентификации пользовательского агента на (прокси-)сервере. Здесь снова требуется \***\*type\*\*** и **_учетные данные_**, которые могут быть закодированы или зашифрованы в зависимости от используемой схемы аутентификации.

---

**Основные схемы** аутентификации-авторизации: Basic, Bearer, Digest, HOBA, Mutual, Negotiate / NTLM, VAPID, SCRAM, AWS4-HMAC-SHA256

**Запрос клиента:**

```http
GET /api/v1/users/123 HTTP/1.1
Host: api.example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: application/json
```

**Ответ сервера (401 Unauthorized):**

```http
HTTP/1.1 401 Unauthorized
Date: Mon, 16 Feb 2026 03:54:50 GMT
Server: nginx/1.22.1

WWW-Authenticate: Basic realm="API Access"
Content-Type: application/json
Content-Length: 85

{"error": "Unauthorized", "message": "Authentication required"}
```

Клиент кодирует логин и пароль в Base64 (например, `alice:secret123` → `YWxpY2U6c2VjcmV0MTIz`) и добавляет заголовок `Authorization`.

**Запрос клиента (с аутентификацией):**

```http
GET /api/v1/users/123 HTTP/1.1
Host: api.example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: application/json

Authorization: Basic YWxpY2U6c2VjcmV0MTIz
```

**Ответ сервера (200 OK):**

```http
HTTP/1.1 200 OK
Date: Mon, 16 Feb 2026 03:54:51 GMT
Server: nginx/1.22.1
Content-Type: application/json
Content-Length: 123
Cache-Control: no-cache

{"id": 123, "name": "Alice", "email": "alice@example.com"}
```
