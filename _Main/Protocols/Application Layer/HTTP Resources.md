[[HTTP]]


HTTP‑ресурсы — это данные или объекты, доступные через веб по уникальному URL. По сути, любой элемент, к которому можно обратиться через интернет — страница, изображение, запись в базе данных через API — может считаться HTTP‑ресурсом. Ключевая особенность в том, что каждый ресурс имеет свой собственный адрес (URL), который однозначно его идентифицирует. Например, `https://example.com/api/users/123` указывает на конкретного пользователя с ID 123. При этом один и тот же ресурс может предоставляться в разных форматах: чаще всего в JSON для API, но также в HTML, PNG, MP3 и других форматах. Выбор формата происходит через согласование между клиентом и сервером с помощью HTTP‑заголовков Accept (что хочет получить клиент) и Content‑Type (в каком формате сервер отдаёт данные).

Взаимодействие с ресурсами строится на стандартных методах HTTP. Чтобы получить данные, используют GET; для создания нового объекта — POST; полное обновление существующего ресурса выполняется через PUT, а частичное — через PATCH; удаление осуществляется методом DELETE. Каждый такой запрос является самодостаточным: HTTP по своей природе безсостоятельный (stateless), то есть сервер не хранит информацию о предыдущих запросах клиента. В ответ на запрос сервер возвращает не только данные, но и метаинформацию через заголовки — например, тип содержимого (Content‑Type), размер ответа (Content‑Length), дату последнего изменения (Last‑Modified) или идентификатор версии ресурса (ETag для кэширования). Статус выполнения операции сообщается через код ответа: 200 OK означает успех, 201 Created — успешное создание, 404 Not Found — ресурс не найден, а 500 Internal Server Error указывает на проблему на стороне сервера.

Теперь о связи с CORS (Cross‑Origin Resource Sharing). Да, тема HTTP‑ресурсов напрямую связана с CORS, поскольку этот механизм регулирует, какие внешние ресурсы может запрашивать браузер из текущего домена. По умолчанию браузер блокирует запросы к ресурсам на других доменах (например, когда JavaScript на сайте site‑a.com пытается получить данные с api.site‑b.com) — это защита от межсайтовых атак. CORS позволяет серверу явно разрешить такие запросы, отправляя специальные заголовки ответа:

- Access‑Control‑Allow‑Origin — указывает, каким доменам разрешён доступ;
- Access‑Control‑Allow‑Methods — перечисляет допустимые HTTP‑методы;
- Access‑Control‑Allow‑Headers — определяет разрешённые заголовки запроса.

Таким образом, даже если HTTP‑ресурс существует и доступен, браузер может отказать в его получении из‑за ограничений CORS, если сервер не отправил соответствующие разрешения. Это важный аспект при разработке веб‑приложений, использующих API на других доменах: без правильной настройки CORS клиентский код не сможет взаимодействовать с нужными ресурсами.
