Cross-Site Request Forgery

Подделка запроса

Эффективным и общепринятым на сегодня способом защиты от CSRF-Атаки является **токен**. Под токеном имеется в виду случайный набор байт, который сервер передает клиенту, а клиент возвращает серверу.

  

Защита сводится к проверке токена, который сгенерировал сервер, и токена, который прислал пользователь.



Если вы пишете свой Web-Сервис в соотвествии со стандартом [RFC7231](https://tools.ietf.org/html/rfc7231#section-4.2), то методы `GET`, `HEAD`, `OPTIONS` и `TRACE` являются безопасными: они предназначены только для получения информации и не должны изменять состояние сервера.

  

Таким образом, защищать необходимо небезопасные методы, к которым относятся: `POST`, `PUT`, `DELETE`, `PATCH`.




**Synchronizer Token Pattern** (STP) — это классический механизм защиты от CSRF-атак (Cross-Site Request Forgery), где сервер генерирует уникальный непредсказуемый токен для каждого state-changing запроса (например, POST, PUT, DELETE), связывает его с сессией пользователя и проверяет при получении.


Если на нашем сайте существует эндпоинт, на который может обратиться пользователь при автоматизированной отправке аутентификационных данных, сохраненных в браузере, то добавление выгодного злоумышленнику эндпоинта на его сайт, может незаметно для клиента выполнить обращение на внешний сайт


CSRF-атака не ограничивается только куками — она возможна при любой **автоматической** передаче аутентификационных данных браузером, которые сложно подделать извне.



CSRF (Cross-Site Request Forgery) — это атака, при которой вредоносный сайт заставляет браузер жертвы отправить нежелательный запрос на уязвимый сайт, где пользователь уже авторизован через cookies. Запрос выполняется автоматически, так как браузер прикрепляет cookies к любому запросу на домен жертвы, независимо от источника инициации.

## Механизм атаки

Жертва авторизуется на уязвимом сайте (например, банке) и получает сессионную cookie. Затем она переходит на вредоносный сайт, который содержит скрытую форму или тег (img, script), отправляющий POST/GET-запрос на уязвимый endpoint, например, для перевода денег. Браузер выполняет запрос с валидными cookies жертвы, и сервер его обрабатывает как легитимный.

## Условия уязвимости

Для успеха атаки на уязвимом сайте должно быть действие (изменение email, перевод средств), зависящее только от cookies для аутентификации, без дополнительных проверок. Злоумышленник заранее анализирует параметры запроса (URL, поля формы) через инструменты разработчика.
