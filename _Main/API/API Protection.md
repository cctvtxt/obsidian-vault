Открытый API приложений является самым популярным направлением для эксплуатации уязвимостей. Основные виды открытых API:

1. REST: Распространённый подход к построению API. Это набор архитектурных рекомендаций вида «всё делаем через ресурсы в URL», «используем стандартные HTTP-методы», «один и тот же ресурс может поддерживать разные операции в зависимости от метода» и т. п. REST не привязан к конкретному формату сообщений, но часто используется с JSON.
2. SOAP: Протокол обмена сообщениями на основе XML. Исторически применялся для RPC-взаимодействий, а сегодня чаще встречается в enterprise- и legacy-системах (банки, государственные сервисы, интеграции с SAP и Oracle).
3. RPC-подобные HTTP API: Многие современные API используют HTTP, но по логике ближе к RPC — когда каждая операция представлена отдельным методом или endpoint-ом.

# Типы атак на API

1. Атаки на параметры (манипуляция входными данными с целью эксплуатации ошибок сериализации, десериализации и бизнес-логики);
2. Раскрытие данных (Data Exposure) (получение доступа к чувствительной информации из-за ошибок авторизации, фильтрации или логики обработки запросов);
3. MITM (Man-in-the-Middle) (перехват и модификация трафика между клиентом и API-сервером);
4. Злоупотребление функциональностью (abuse) (использование легитимных API-возможностей в недокументированных сценариях).

# Методы защиты API

**Использование актуальных библиотек:** обновлённые библиотеки закрывают уязвимости и улучшают производительность — их регулярный апгрейд обязателен для кибербезопасности.
**Единые парсеры на всех этапах обработки:** обеспечивают консистентность данных, снижают риски ошибок и атак (например, инъекций) за счёт единого подхода к интерпретации данных.
**Строгая валидация входных данных:** защищает от SQL-инъекций, XSS, XXE — проверяет формат, длину, тип данных на соответствие ожидаемым значениям.
**Явная проверка бизнес-ограничений:** контролирует соответствие данных бизнес-правилам (например, права доступа), предотвращает несанкционированные операции.
**Внедрение WAF и API-Security:** создаёт дополнительный барьер для атак — WAF фильтрует HTTP-трафик, API-Security защищает интерфейсы взаимодействия сервисов.
**Виртуальный патчинг уязвимостей:** блокирует известные векторы атак на уровне инфраструктуры без срочного обновления ПО — критично для систем с затруднённым апгрейдом.
**Контроль структуры и содержания API-запросов:** проверяет формат (JSON/XML), типы данных и допустимые значения параметров — предотвращает ошибки обработки и инъекции кода.

# 1. Уязвимости REST и JSON

1. Дублирующиеся ключи JSON:

Стандарт JSON (RFC 8259) не запрещает повторяющиеся ключи и не определяет приоритет их обработки. Разные парсеры могут вести себя по-разному. Большинство современных парсеров (Python, Go encoding/json) используют последнее значение, но при использовании нестандартных или потоковых парсеров возможны расхождения.

Проблема возникает, когда: запрос валидируется одним парсером → бизнес-логика обрабатывается другим → итоговая логика расчётов расходится.

2. Unicode-коллизии и нормализация строк:

Некоторые атаки основаны на различиях в обработке Unicode-символов, суррогатных пар и нормализации строк между различными компонентами системы.

Это не так характерно для современных систем, но в legacy может встречаться такая ситуация:

- одна подсистема принимает строку →
- другая — интерпретирует её иначе →
- можно обойти различные ограничения.

3. Комментарии и нестандартный JSON:

По стандарту JSON не поддерживает комментарии. Но некоторые парсеры и расширенные форматы (JSON5, HJSON) допускают их.

Если разные компоненты системы используют разные реализации парсинга, возможны ситуации, когда один парсер принимает данные как валидные, а другой — интерпретирует их иначе, и это создаёт условия для атак.

# 2. Уязвимости SOAP и XML

1. Логические ошибки авторизации:

Многие уязвимости в SOAP-сервисах связаны не с самим протоколом, а с ошибками бизнес-логики: некорректной валидацией обязательных полей, неправильной обработкой альтернативных параметров и утечкой внутренних сообщений об ошибках.

Это может приводить к обходу авторизации и получению повышенных прав. Это достигается с помощью SOAP-инъекций.

2. SOAPAction Spoofing:

SOAP допускает использование HTTP-заголовка SOAPAction.  
В устаревших или некорректно сконфигурированных системах возможны сценарии, при которых приложение доверяет значению SOAPAction, не сверяя его с телом запроса.

3. XXE (XML External Entity):

Уязвимость заключается в уязвимости XML-парсеров, когда приложение разрешает обработку внешних сущностей. XXE позволяет:

- Читать локальные файлы;
- Выполнять SSRF;
- Извлекать конфигурации и секреты.

XXE не выполняет код, но может приводить к серьёзным утечкам данных и компрометации инфраструктуры.
