[[API]]

Проектирование **API** относится к созданию правил и протоколов, регулирующих взаимодействие систем. Для этого им необходимо определить различные методы, форматы данных, механизмы аутентификации и конечные точки, которые позволяют отдельным программным сущностям функционировать.

Хорошо спроектированный **API** имеет несколько ключевых преимуществ:

- Предоставляет разработчикам понятный, интуитивно понятный интерфейс для доступа к ресурсам
- Обеспечивает эффективную, надежную и предсказуемую интеграцию с низким риском ошибок кодирования
- Способствует взаимодействию между разнообразными системами, программным обеспечением и приложениями
______
Первый этап представляет собой выбор стиля проектирования, где **выбирают из архитектурных стилей и протокольных подходов**. На практике чаще всего сравнивают REST, GraphQL, RPC/gRPC, SOAP, иногда WebSockets, webhooks, MQTT и event-driven подходы.

Обычно выбор делают по нескольким осям:

- **Тип взаимодействия**: синхронное или асинхронное.
- **Модель данных**: ресурсы и HTTP-методы, либо вызов процедур, либо гибкий запрос данных.
- **Требования к производительности**: обычный HTTP/JSON, бинарный протокол, streaming.
- **Контракт и стандартизация**: строгая схема и генерация кода, или более свободный стиль.
- **Безопасность и совместимость**: встроенные enterprise-механизмы, поддержка старых систем, удобство для браузера и мобильных клиентов.


Проектирование API — это не только про эндпоинты. Это про создание надежной, безопасной и удобной платформы.

- **API как продукт.** Ваш API — это продукт для разработчиков. У него есть свой жизненный цикл, своя документация (маркетинг), свои пользователи и своя поддержка. Относитесь к нему соответственно. Плохой API отпугнет интеграторов и партнеров так же, как плохой UI отпугивает конечных пользователей.
    
- **Безопасность по умолчанию.** Безопасность не "прикручивается" в конце. Она должна быть встроена в дизайн.
    
    - **Аутентификация и авторизация:** Используйте стандартные протоколы (OAuth 2.0, OpenID Connect). Не изобретайте свои. Авторизация должна проверяться на каждом запросе, на уровне доступа к конкретному ресурсу.
        
    - **Валидация на входе:** Никогда не доверяйте данным от клиента. Внедрите строгую валидацию на границе API (например, через JSON Schema). Любой невалидный запрос должен отбрасываться с ошибкой 400.
        
- **Производительность и кэширование.** Хороший API должен быть быстрым.
    
    - **HTTP-кэширование:** Используйте заголовки Cache-Control, ETag и Last-Modified. Для GET запросов, которые возвращают редко меняющиеся данные, кэширование может снизить нагрузку на порядки. ETag особенно полезен для условных запросов.
        
    - **Rate Limiting:** Защитите свой API от злоупотреблений и DoS-атак. Внедрите ограничения на количество запросов. Важно сообщать клиенту о лимитах через заголовки (X-RateLimit-Limit, X-RateLimit-Remaining).
        

### Опыт разработчика

Это то, что отличает просто работающий API от API, с которым приятно работать.

- **Документация — это не опция.** Отсутствие документации или ее плохое качество — это неуважение к пользователям вашего API.
    
    - **OpenAPI (Swagger):** Это стандарт де-факто. Он позволяет не только описать ваш API, но и сгенерировать интерактивную документацию, клиентские SDK и наборы тестов. Документация должна быть частью CI/CD и обновляться вместе с кодом.
        
- **Песочница (Sandbox).** Предоставьте разработчикам безопасную среду, где они могут экспериментировать с вашим API, не боясь сломать реальные данные. Песочница должна быть максимально приближена к продакшен-среде.
    
- **Клиентские SDK.** Предоставление готовых библиотек для популярных языков может значительно снизить порог вхождения. Однако это создает дополнительную нагрузку по их поддержке.
    

### Выбор правильного инструмента для задачи

1. **Простой внутренний CRUD-сервис.**
    
    - Ресурс-ориентированный подход, пагинация offset/limit. Минимум сложностей. Главное — скорость разработки.
        
2. **Публичный API для партнеров.**
    
    - Строгий контракт. Обязательное версионирование в URL. Стандартизированные и подробные ошибки. Документация OpenAPI. Идемпотентность для всех POST.
        
3. **API для высоконагруженного мобильного приложения.**
    
    - Максимальная производительность. Пагинация на основе курсора. Поддержка fields и embed. Отдельные batch-эндпоинты.
        
4. **Сложная микросервисная система.**
    
    - GraphQL-фасад для внешних клиентов. Паттерн "Сага" для распределенных транзакций. Асинхронные операции с вебхуками для межсервисного взаимодействия.

Для REST API мы производим

- выбор ресурсов. Один из способов решения проблемы — пошагово рассмотреть то, что включает в себя типичное взаимодействие. 
- назначение URL-адресов. Существует множество возможностей, но, к счастью, соглашения **REST** дают некоторые указания. В типичном **REST API** ресурсу будет назначено два шаблона URL. **Первый** — это множественное число имени ресурса, `/orders`, а **второй** — это множественное число имени ресурса плюс уникальный идентификатор для указания одного ресурса, например `/orders/<order_id>`
- определение действий клиента. Следуя соглашениям **REST**, мы говорим, что конечная точка множественного числа (/orders) предназначена для перечисления существующих заказов и создания новых. Конечная точка множественного числа с уникальным идентификатором (/orders/<order_id>) предназначена для извлечения, обновления или отмены определенного заказа. Клиент сообщает, какое действие выполнить, передавая правильный **HTTP-глагол** (**GET**, **POST**, **PUT** или **DELETE**)
- определение данных для обмена. После того, как действия для наших конечных точек заказа были конкретизированы, необходимо решить, какими данными необходимо обмениваться между клиентом и сервером. Нам также нужно выбрать формат данных, который клиент и сервер могут использовать для передачи этой информации туда и обратно
- определение необходимости и способа объединения ресурсов. Практикующие **REST** разделились во мнении о том, как решить проблему связывания ресурсов. Некоторые говорят, что иерархия должна продолжать расти, предоставляя конечные точки вроде /customers/5/orders для всех заказов клиента. Другие утверждают, что нужно сохранять все плоским, включая связанные детали в данные для ресурса. В рамках этой парадигмы создание заказа требует отправки поля customer_id с деталями заказа
- определение способа поиска данных. Отправка запроса **GET** на /orders и получение всех трех миллионов заказов не принесет особой пользы. К счастью, **REST** имеет отличный способ поиска по данным, связанный с query-параметрами (query parameters) в URI.
_________
При работе с REST API и его масштабировании возникает ряд нетривиальных проблем, требующих глубокой проработки. **Вопросы производительности** решаются внедрением многоуровневого **кэширования**, использованием заголовков `ETag` и `Cache-Control` для управления кэшем на стороне клиента и промежуточных прокси. Для снижения нагрузки на сервер и улучшения времени отклика критически важна эффективная **пагинация**, а также возможность частичного ответа, когда клиент запрашивает только необходимые ему поля ресурса. В сложных системах синхронная обработка запросов может стать узким местом, что приводит к необходимости внедрения асинхронных операций с использованием **очередей сообщений**, где клиент инициирует задачу и позже запрашивает ее результат по другому URL.

**Надежность и безопасность** системы обеспечиваются не только стандартными механизмами **аутентификации и авторизации**, но и более тонкими инструментами. **Ограничение частоты запросов** (rate limiting) защищает API от злоупотреблений и DDoS-атак. Важно спроектировать **стандартизированный и информативный формат ошибок**, который не только сообщает о проблеме, но и предоставляет контекст, достаточный для ее решения на стороне клиента, не раскрывая при этом чувствительной информации о внутренней архитектуре сервера.

Жизненный цикл API неизбежно связан с изменениями. Чтобы избежать поломок у клиентов, применяется **версионирование**, которое может быть реализовано через URL, заголовки запроса или query-параметры. Каждый подход имеет свои компромиссы с точки зрения чистоты REST и удобства использования. Параллельно с этим, поддержание актуальной и подробной **документации**, например, с помощью спецификации OpenAPI, становится ключевым фактором для успешной интеграции и поддержки API в долгосрочной перспективе.

Сложные бизнес-операции, требующие атомарности, представляют собой еще один вызов для REST-архитектуры. Поскольку REST не имеет встроенной поддержки транзакций, охватывающих несколько запросов, для таких сценариев приходится реализовывать паттерны Saga, где **сложная операция разбивается на серию локальных транзакций, а для отката используются компенсирующие операции**. Для длительных задач, выполнение которых превышает тайм-аут HTTP-запроса, применяется подход, при котором сервер немедленно отвечает статусом `202 Accepted` и предоставляет URL для отслеживания статуса задачи, что позволяет разгрузить клиентские соединения и обеспечить надежную обработку.
