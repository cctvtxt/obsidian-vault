Правильный ответ — **применить Apollo Federation с поддержкой схемы, в которой разные типы данных обрабатываются разными микросервисами.**

**Обоснование:**

Apollo Federation — оптимальное решение для микросервисной архитектуры, позволяющее:

- объединить GraphQL-схемы нескольких микросервисов в единый суперграф (единую точку доступа для клиента);
- сохранить автономность микросервисов (каждый сервис управляет своей схемой и данными);
- обеспечить единый GraphQL-шлюз (gateway), через который клиент делает запросы;
- автоматически резолвить (разрешать) запросы, распределяя их между микросервисами .

**Как работает Apollo Federation в данной задаче:**

1. **Сервис с данными о пользователях** экспортирует схему с типами и полями, описывающими профиль пользователя.
2. **Сервис с данными о заказах** экспортирует схему с типами и полями, описывающими заказы.
3. **Федеративный шлюз (gateway)** объединяет эти схемы в одну, создавая иллюзию единого API.
4. Когда клиент отправляет запрос на получение профиля пользователя и его заказов, шлюз:
   - разбирает запрос и определяет, к каким микросервисам нужно обратиться;
   - отправляет подзапросы к соответствующим сервисам;
   - объединяет ответы в единый JSON;
   - возвращает результат клиенту .

**Почему другие варианты хуже:**

1. **Subscription на каждый сервис + объединение на клиенте:**
   - не решает задачу агрегации данных в одном запросе;
   - требует реализации логики объединения на стороне клиента;
   - подходит только для real-time обновлений, а не для обычных запросов .

2. **Mutation для объединения информации:**
   - Mutation предназначен для изменения данных, а не для их получения;
   - не подходит для агрегации данных из разных сервисов в одном запросе .

3. **Directive @external для каждого микросервиса:**
   - @external — это лишь часть механизма Federation, сама по себе не решает задачу;
   - директива указывает, что поле не может быть разрешено локально, но требует инфраструктуры Federation для работы .

4. **Один общий Resolver + REST-подзапросы:**
   - создаёт узкое место — единый Resolver становится точкой отказа и бутылочного горлышка;
   - увеличивает латентность за счёт последовательных REST-запросов;
   - нарушает принципы GraphQL (множество запросов вместо одного);
   - усложняет поддержку (нужно вручную синхронизировать схемы).

**Ключевые преимущества Apollo Federation:**

- **Единая схема API** для клиента (не нужно знать, где лежат данные).
- **Автономность сервисов** (можно развивать микросервисы независимо).
- **Эффективность** (шлюз может оптимизировать запросы, использовать batching).
- **Поддержка сложных связей** между сущностями (например, связь «пользователь — заказы»).
- **Масштабируемость** (легко добавлять новые микросервисы и типы данных).
- **Стандартное решение** с развитой экосистемой и документацией
