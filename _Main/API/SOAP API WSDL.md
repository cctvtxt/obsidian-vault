**WSDL расшифровывается как Web Services Description Language — язык описания веб-сервисов.** Это XML-документ, который полностью описывает интерфейс SOAP-сервиса: что он умеет, какие данные принимает/возвращает, по какому протоколу. Он формируется на сервере при публикации сервиса и доступен по URL с суффиксом ?wsdl.

## Этап передачи WSDL

WSDL передаётся на **предварительном этапе**, до основного обмена SOAP-сообщениями. Клиент запрашивает его по HTTP GET (например, /service?wsdl), сервер отвечает файлом WSDL, после чего клиент генерирует прокси (stubs) для вызовов.

## Формирование WSDL

WSDL формируется **автоматически на сервере** при публикации веб-сервиса (например, в 1C, .NET или Java). Он генерируется на основе кода сервиса: описывает операции, типы данных, протоколы (HTTP/SOAP) и endpoint.

## Влияние на систему

WSDL **улучшает интеграцию**, но добавляет overhead: начальный запрос увеличивает latency при первом подключении, усложняет изменения (нужно обновлять клиентов). В микросервисах редко используется в пользу REST/OpenAPI из-за XML-тяжести и статичности.

## Последовательность общения

1. **Клиент запрашивает WSDL**: GET /service?wsdl → Сервер отдаёт XML-описание.
2. **Генерация прокси**: Клиент парсит WSDL, создаёт stubs для методов.​
3. **Обмен SOAP**: Клиент формирует XML-запрос (в HTTP POST), сервер выполняет операцию и отвечает SOAP-ответом.
4. **Повторные вызовы**: Используют пул соединений без повторного WSDL.​

## Ключевые разделы WSDL-документа

- **types**: Определяет XSD схемы для запроса (AddRequest: два double) и ответа (AddResponse: result).​
- **message**: Связывает типы с SOAP-сообщениями.
- **portType**: Интерфейс с операцией "Add".
- **binding**: Привязка к SOAP/HTTP (document/literal стиль).
- **service**: URL endpoint для доступа.

---

Наиболее функциональные библиотеки SOAP используют WSDL-описания для формирования запросов к сервисам и проверки передаваемых данных. Библиотеки скачивают WSDL-описание, анализируют его и формируют структуры данных, необходимые для вызова методов. Приложению остается заполнить эти структуры данными и выполнить запрос. Информация из ответного сообщения также выдается в виде структуры, свойственной языку программирования. Все это избавляет приложение от необходимости обрабатывать XML.

Приложение может самостоятельно формировать SOAP-запросы в формате XML, однако это трудоемкая задача и при наличии полнофункциональной SOAP-библиотеки выполнять ее вручную нецелесообразно.
