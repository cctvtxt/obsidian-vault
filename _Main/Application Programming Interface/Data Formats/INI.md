Пример Создан в 1980-х (MS-DOS/Windows) как простой формат секций для конфигов. **Характеристика: Секционный (section-based), key=value, плоский и минималистичный — "INI" (Initialization file).** Цель — легкие настройки приложений без компиляции. Стандартизирован частично (Windows API), расширен в современных парсерах. INI использует [sections], строки key=value, поддерживает строки, числа, булевы (true/false). Комментарии с ; или #.

```ini
[dependencies]
serde = 1.0
serde_features = ["derive"]
toml = 0.8

[build]
version = 0.1.0
date = 2026-03-04T20:39:00Z
authors = dev@example.com
```

Формат эволюционировал от Windows .ini к кросс-платформе. Альтернатива JSON/YAML (проще для плоских настроек), CSV (секции вместо строк). Часто сравнивают с properties-файлами Java.

## Преимущества

Максимальная простота — редактируется в Notepad.  
Быстрый парсинг, малый размер, нулевой overhead.  
Читаемость как словарь, легко diff в Git.  
Широкая встроенная поддержка (Python configparser, Rust ini crate).  
Идеален для legacy/embedded без зависимостей.

## Недостатки

Нет вложенности (только плоские секции).  
Отсутствие типизации, массивов, дат.  
Проблемы с escape (пробелы, спецсимволы).  
Неоднозначный стандарт (разделители, кодировки).  
Не для сложных структур или обмена данными.

## Область применения

INI популярен в Windows (system.ini), играх (config.ini), Python (setup.cfg), Rust Cargo (частично). Идеален для простых локальных настроек без иерархии.
