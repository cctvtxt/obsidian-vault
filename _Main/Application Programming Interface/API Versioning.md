[[API]]

Версионирование веб API — это механизм, который позволяет развивать интерфейс сервиса без поломки уже работающих клиентов. Оно необходимо, когда меняются форматы данных, правила валидации, структура ответов или поведение эндпоинтов, при одновременном требовании сохранить обратную совместимость.
# Версионирование позволяет

- добавлять новые возможности без остановки старых клиентов
- постепенно выводить устаревшие контракты из эксплуатации
- разделять эволюцию внутренней реализации и публичного интерфейса
- управлять жизненным циклом API предсказуемо

Обычно отдельно различают версию самого API и версию его спецификации или документации. Это помогает не смешивать описание контракта с технической версией реализации.
## Версия в URL

Пример: `/v1/users`, `/v2/users`.

Это самый простой и понятный вариант. Его легко маршрутизировать, логировать и документировать. Минус в том, что версия становится частью адреса, а значит, при каждом major-изменении появляется новый набор маршрутов и соответствующая необходимость перевода клиента на использование их. 
## Версия в заголовке

Пример: `X-API-Version: 2` или `Accept: application/vnd.company.v2+json`.

Такой подход оставляет URL чистым и позволяет выбирать версию на уровне контента или заголовка запроса. Он удобен для более зрелых систем, но сложнее для ручного тестирования и отладки.
## Версия в query-параметре

Пример: `/users?version=2`.

Это простой способ внедрения, но он обычно считается менее строгим с точки зрения контракта. Его используют реже, особенно в публичных API.
## Версия через media type

Пример: `Accept: application/json;v=2`.

Это вариант content negotiation, когда клиент и сервер договариваются о формате ответа через заголовки. Подход гибкий, но требует дисциплины в клиентских приложениях и инфраструктуре.
# Как это реализуется в архитектуре

На уровне приложения версии обычно сводятся к разным обработчикам, контроллерам или слоям преобразования данных. Для каждой версии можно использовать отдельные схемы валидации, отдельные DTO и отдельные правила сериализации.

При небольшом количестве изменений часто делают общую бизнес-логику и разные адаптеры для входа и выхода. Это снижает дублирование и упрощает поддержку.

**API gateway** при этом часто используется как центральная точка маршрутизации версий. Он может:

- направлять `/v1/*` и `/v2/*` в разные backend-сервисы;
- выбирать upstream по заголовку версии;
- применять разные политики авторизации и лимитов для разных версий;
- добавлять трансформации запросов и ответов;
- обеспечивать единое логирование и метрики по всем версиям.
# OpenAPI и документация

В OpenAPI обычно отдельно указывают версию спецификации и версию самого API. Это важно, потому что изменение файла описания не всегда означает изменение контракта, и наоборот.

Для крупных изменений часто ведут отдельные спецификации для разных major-версий. Это упрощает генерацию SDK, публикацию документации и контроль совместимости.
# Практика применения

- несовместимые изменения часто публикуют как новую major-версию;
- совместимые изменения в пределах одной версии стараются делать без изменения контракта;
- старую версию поддерживают ограниченное время;
- дату прекращения поддержки объявляют заранее;
- новые версии вводят только при наличии реальной необходимости.

**Для внешних API** часто **выбирают версию в URL** из-за простоты. **Для внутренних платформ** и более сложной инфраструктуры чаще **подходят gateway-based схемы и версионирование через заголовки**
