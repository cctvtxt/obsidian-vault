
[[API]]

Синхронный API обеспечивает последовательное получение ответов на каждый запрос в рамках одного HTTP-соединения: сервер обрабатывает запрос и возвращает результат до закрытия соединения. При этом само клиентское приложение не блокируется — современные среды используют асинхронные механизмы для сетевых операций. Термин «синхронный» здесь относится к протокольному уровню: клиент ожидает завершения операции на сервере перед получением ответа, в отличие от API, где сервер сразу возвращает подтверждение приёма, а результат приходит отдельно. Синхронный подход доминирует в RESTful-архитектурах и оптимален для операций с предсказуемой задержкой: получение справочных данных, валидация, аутентификация. Преимущества — простота ментальной модели (запрос → ответ), лёгкость отладки и предсказуемость потока выполнения.
# Недостатки синхронных API

При синхронном взаимодействии повторные попытки запросов усложняются тем, что <mark style="background:#ff4d4f">клиент зависает в ожидании ответа</mark>. Если сервер перегружен или временно недоступен, клиент вынужден либо ждать таймаута (заставляя пользователя смотреть на loader), либо прерывать соединение и терять контекст операции. В отличие от асинхронных систем, где retry-логика работает прозрачно в фоне через очереди с exponential backoff, **синхронные ретраи требуют явной реализации на клиенте и увеличивают общее время ожидания линейно**. Это создаёт UX-проблемы: пользователь не понимает, почему интерфейс «завис», и может инициировать дублирующие запросы, усугубляя нагрузку на систему.

Синхронный API <mark style="background:#ff4d4f">принудительно удерживает HTTP-соединение открытым до полного выполнения операции</mark>, что приводит к неэффективному использованию ресурсов. Для длительных процессов (генерация отчётов, сложные вычисления, интеграция с медленными внешними сервисами) это означает удержание потока сервера, соединения TCP/IP и сокета клиента. В условиях высокой конкурентности такое ожидание быстро исчерпывает пул соединений (connection pool) и приводит к отказу в обслуживании новых запросов. Клиентское приложение вынуждено показывать индикаторы загрузки или блокировать UI-элементы, ухудшая восприятие отзывчивости системы даже при технически корректной работе.

Жёсткие временные ограничения в синхронной модели создают дилемму: либо установить короткий таймаут и рисковать прервать успешно выполняющуюся операцию на сервере (orphaned operation), либо удлинить таймаут и терпеть ухудшение responsiveness. В распределённых системах deadline propagation — передача ограничения времени через цепочку вызовов — требует сложной координации между сервисами. Часто возникает ситуация, когда клиент уже отказался от ожидания (по таймауту), <mark style="background:#ff4d4f">но сервер продолжает обработку, тратя ресурсы на бесполезную работу</mark> и потенциально создавая побочные эффекты (например, двойное списание средств при повторном ретрае).

Наиболее критичный недостаток в микросервисной архитектуре представляет собой <mark style="background:#ff4d4f">подверженность каскадным сбоям</mark>. Если сервис A синхронно вызывает сервис B, а тот — сервис C, то медленный ответ от C парализует всю цепочку, система коллапсирует даже при локализованной проблеме. Это классический сценарий распределённого deadlock'а, против которого применяются **circuit breaker**, **bulkhead** и **timeout patterns** — но эти защитные механизмы добавляют сложности и частично нивелируют предсказуемость, ради которой выбирался синхронный подход.
__________
Синхронный API доминирует в сценариях, требующих немедленного результата для продолжения пользовательского потока. Классические юзкейсы: **CRUD-операции над справочными данными** — получение профиля пользователя, списка товаров, настроек приложения; **валидация и авторизация** — проверка токена, верификация платёжных данных перед подтверждением заказа; **операции реального времени** — поиск с автодополнением, калькуляторы цен, чат-боты с low-latency ответами; **координация распределённых транзакций** — бронирование с immediate confirmation, когда пользователь должен видеть результат до перехода к следующему шагу. Sync оптимален, когда latency предсказуема (< 500 мс), операция атомарна, и задержка критичнее для UX, чем риск каскадного сбоя — типично для фронтенд-ориентированных endpoint'ов и мобильных приложений.

# Синхронные (преимущественно запрос-ответ)

### REST
Это архитектурный стиль, а не протокол. Он задаёт правила построения веб-сервисов: ресурсы идентифицируются через URL, операции выполняются стандартными HTTP-методами, а сервер не хранит состояние между запросами. Каждый запрос должен содержать всю информацию, необходимую для его обработки, что упрощает горизонтальное масштабирование.

Для оптимизации в REST активно используется кэширование на стороне клиента, CDN и промежуточных прокси. Это снижает нагрузку на бэкенд и ускоряет отклики за счёт повторного использования ранее полученных данных. Унифицированный интерфейс и предсказуемые статус-коды делают REST предсказуемым и простым в документации.

API на основе REST легко интегрируются в любые платформы благодаря стандартной экосистеме HTTP. Инструменты мониторинга, балансировщики и брандмауэры понимают его «из коробки», что снижает порог входа для новых разработчиков.

REST — не конкретный протокол, а набор архитектурных ограничений. По умолчанию он синхронен (клиент ждёт ответ), но легко комбинируется с асинхронными паттернами вроде Webhooks или очередей сообщений для длительных операций.

REST строится вокруг ресурсов, идентифицируемых URL-адресами, где стандартные HTTP-методы определяют действия, а каждый запрос полностью самодостаточен и не зависит от состояния сервера.
```http
GET    /api/v1/users/123          # Получить пользователя
POST   /api/v1/users              # Создать пользователя
PUT    /api/v1/users/123          # Полное обновление
PATCH  /api/v1/users/123          # Частичное обновление
DELETE /api/v1/users/123          # Удалить пользователя
```

### RPC (традиционный)
Это парадигма взаимодействия, при которой вызов удалённой функции выглядит для разработчика как обычный локальный вызов. Клиент передаёт параметры, сервер выполняет бизнес-логику и возвращает результат, полностью абстрагируя сетевую коммуникацию от прикладного кода.

Исторически RPC строился на бинарных протоколах поверх TCP или HTTP, а современные реализации используют строго типизированные контракты и эффективную сериализацию. Акцент сделан на действиях и методах, а не на манипуляции ресурсами, что упрощает проектирование сложных процедур.

Подход снижает когнитивную нагрузку на разработчиков, скрывая детали маршрутизации и обработки пакетов. Чёткие сигнатуры функций позволяют легко генерировать клиентские обёртки под разные языки программирования.

RPC — это не конкретный протокол, а архитектурный паттерн вызова удалённых функций. Поэтому он не привязан жёстко ни к синхронности, ни к асинхронности. Исторически первые реализации RPC проектировались как синхронные, ожидая немедленного ответа от сервера.

RPC абстрагирует сеть за счёт вызова удалённых процедур так, будто они локальные, где клиент передаёт имя функции и параметры, а сервер возвращает результат, фокусируясь на действиях, а не на ресурсах.
```javascript
// Клиент вызывает как обычную функцию
const result = await UserService.createUser({ name: "Alex", role: "admin" });
// Сервер экспортирует процедуры
rpc createUser(params) -> { id, status };
```

### SOAP
Это строгий протокол обмена структурированными сообщениями, который стал отраслевым стандартом для корпоративных интеграций. Все данные передаются в виде XML-конвертов, а контракт взаимодействия жёстко описывается в файле WSDL, генерируемом автоматически.

Протокол включает встроенные механизмы безопасности (WS-Security), транзакций (WS-AtomicTransaction) и надёжной доставки, что гарантирует предсказуемость в критически важных системах. Верификация схемы происходит до выполнения логики, отсекая некорректные запросы на раннем этапе.

Несмотря на высокую нагрузку из-за многословности XML и сложности парсинга, SOAP остаётся востребованным там, где требуется строгая валидация, аудит и соответствие регуляторным стандартам. Инструментарий и спецификации поддерживаются десятилетиями.

SOAP — это полноценный протокол с жёсткой спецификацией, а не просто стиль общения. Его синхронная природа и тяжёлые XML-конверты оправданы только в средах с высокими требованиями к безопасности, транзакционной целостности и обратной совместимости.

SOAP использует строгие XML-конверты для обмена сообщениями, где каждый запрос и ответ валидируется по схеме WSDL, а операции описываются в виде строго типизированных методов с встроенной поддержкой транзакций и безопасности.
```xml
<soap:Envelope>
  <soap:Body>
    <m:ProcessPayment xmlns:m="urn:banking">
      <m:AccountID>8472</m:AccountID>
      <m:Amount currency="USD">50.00</m:Amount>
    </m:ProcessPayment>
  </soap:Body>
</soap:Envelope>
```

### SERVERLESS (как модель доставки)
Это модель развёртывания и выполнения кода, при которой разработчик пишет отдельные функции, а облачный провайдер берёт на себя управление инфраструктурой, масштабированием и доступностью. Код запускается только в момент поступления триггера и автоматически останавливается после завершения работы.

Оплата взимается только за фактическое время выполнения и потреблённые ресурсы, что исключает затраты на простой серверов. Платформа автоматически распределяет нагрузку, создавая новые инстансы при росте трафика и утилизируя их в периоды спада.

Модель идеально подходит для event-driven архитектур, фоновых задач, обработки файлов и прототипирования. Она позволяет маленьким командам выводить продукты на рынок без найма DevOps-инженеров.

Serverless — это не протокол связи, а способ доставки и исполнения бизнес-логики. В контексте синхронности он чаще всего работает по модели запрос-ответ, но «холодные старты» и лимиты времени выполнения требуют асинхронного проектирования для тяжёлых операций.

Serverless моделирует доставку логики через события, где функция активируется триггером (HTTP-запрос, очередь, таймер), получает контекст и полезную нагрузку, выполняет код и возвращает ответ, полностью скрывая инфраструктуру.
```python
# AWS Lambda / аналог
def handler(event, context):
    user_id = event["pathParameters"]["id"]
    return { "statusCode": 200, "body": json.dumps(get_user(user_id)) }
# Триггер: HTTP API Gateway -> POST /api/order
```

### tRPC
Это современная библиотека для создания строго типизированных API в экосистеме TypeScript, которая устраняет необходимость в кодогенерации или отдельных схемах. Разработчик описывает процедуры на сервере, а клиент автоматически получает автодополнение, проверку типов и интроспекцию на уровне компилятора.

Библиотека работает поверх стандартных HTTP-запросов (обычно JSON-RPC), сохраняя простоту интеграции и отладки через браузерные инструменты. Это значительно ускоряет разработку fullstack-приложений и минимизирует расхождения между клиентом и сервером.

Подход снижает количество runtime-ошибок и упрощает рефакторинг: изменение сигнатуры на бэкенде немедленно подсвечивается в клиентском коде. Экосистема плагинов позволяет легко добавлять middleware, логирование и аутентификацию.

tRPC — это надстройка над синхронным HTTP-транспортом, которая фокусируется на безопасности типов, а не на сетевых особенностях. Она идеально синхронна по умолчанию, но не поддерживает нативные потоки или push-уведомления, требуя внешних решений для асинхронных сценариев.

tRPC устраняет разрыв между клиентом и сервером через общие TypeScript-типы, позволяя описывать процедуры на бэкенде и вызывать их на фронтенде с автодополнением и проверкой типов на этапе компиляции без кодогенерации.
```typescript
// server.ts
export const appRouter = router({ getUser: procedure.input(z.string()).query(({ input }) => db.users.find(input) });
// client.ts
const user = await trpc.getUser.query("123"); // Автодополнение и типизация из коробки
```

### HTTP POLLING
Простейший метод получения обновлений, при котором клиент периодически отправляет запросы к серверу через фиксированные интервалы. Сервер отвечает немедленно, возвращая либо новые данные, либо пустой результат, если изменений нет.

Механизм работает поверх стандартного HTTP и не требует специальной поддержки на стороне инфраструктуры или сложных библиотек. Логика реализуется за несколько строк кода на любом языке программирования.

Простота делает его универсальным, но неэффективным при высокой частоте опросов или редких обновлениях. Трафик расходуется на холостые запросы, а задержка доставки привязана к длине интервала.

Polling — это полностью синхронный и клиенто-ориентированный паттерн. Он генерирует избыточный трафик и задержки, поэтому сегодня используется только в легаси-системах, для проверки здоровья сервисов или как временное решение до внедрения push-технологий.

Polling реализует получение данных через периодические синхронные запросы, где клиент самостоятельно инициирует проверку обновлений через фиксированные интервалы, получая либо актуальное состояние, либо подтверждение отсутствия изменений.
```javascript
setInterval(async () => {
  const res = await fetch('/api/notifications?last_seen=1690000000');
  const data = await res.json(); // [] или [{id: 1, text: "New alert"}]
}, 5000); // Запрос каждые 5 секунд независимо от наличия данных
```

### LONG POLLING
Улучшенная эволюция обычного polling, при которой клиент отправляет запрос, а сервер удерживает соединение открытым до появления новых данных или истечения таймаута. Как только данные готовы, сервер возвращает ответ, и клиент мгновенно отправляет следующий запрос.

Это эмулирует push-поведение в рамках стандартного HTTP/1.1 без необходимости в дополнительных расширениях протокола. Подход снижает количество холостых обращений и уменьшает задержку доставки обновлений до долей секунды.

Сервер вынужден держать множество висящих соединений, что требует грамотного управления пулами потоков и асинхронной обработки. Современные фреймворки решают эту проблему через event-loop или worker-пулы.

Long Polling — это компромисс между синхронным HTTP и асинхронным push. Технология остаётся жизнеспособной, когда WebSocket или SSE недоступны из-за политик корпоративных прокси или брандмауэров, но усложняет масштабирование сервера из-за долгоживущих соединений.

Long Polling эмулирует push-уведомления, заставляя сервер удерживать HTTP-соединение открытым до появления данных или таймаута, после чего клиент мгновенно открывает новое ожидание, минимизируя холостые запросы.
```http
GET /api/events?timeout=30 HTTP/1.1
Host: api.example.com
# Ответ приходит через 15 сек при появлении события или через 30 сек с 204 No Content
{"event": "order_completed", "order_id": "8842"}
```
