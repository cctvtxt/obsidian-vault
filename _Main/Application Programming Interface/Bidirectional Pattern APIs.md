[[API]]

В современной архитектуре под **bidirectional** API понимается API, где **клиент и сервер являются равноправными участниками**: оба могут независимо и параллельно отправлять данные, не блокируя друг друга. В классической синхронной модели клиент единственный инициатор запросов. Сервер не может самостоятельно уведомить клиента об изменениях, что создаёт задержки в доставке данных. Bidirectional API — решение данной проблемы. Обработка запроса происходит в фоновом режиме, а результат доставляется клиенту позже через механизмы обратного вызова (webhooks), очереди сообщений или polling-запросы либо при стриминговой передаче данных между клиентом и сервером. 
_________
**Bidirectional API** незаменим для сценариев реального времени и постоянного взаимодействия, где мгновенная двусторонняя связь критична. Ключевые юзкейсы: чаты и коллаборативные инструменты — обмен сообщениями в мессенджерах, совместное редактирование документов (Google Docs), live-комментарии; финансовые трейдинг-платформы — streaming котировок, ордера buy/sell в реальном времени, уведомления о транзакциях; IoT и телеметрия — сбор данных с датчиков, удалённое управление устройствами, live-мониторинг (умный дом, промышленные сенсоры); интерактивные игры и стриминг — multiplayer сессии, live-статус зрителей, синхронизация состояний; колл-центры и customer support — live-чаты с агентами. Также bidirectional идеален для систем мониторинга — когда сервер должен "толкать" алерты (server push), статусы задач, обновления дашбордов без polling'а. Выбор bidirectional оправдан, когда важнее отзывчивость и отсутствие задержек, чем простота и надежность request-response-модели взаимодейсвтия.

Преимущества такого подхода: **эффективность ресурсов** — одно соединение заменяет тысячи polling-запросов; **мгновенная доставка** без задержек polling-интервала; **истинный full-duplex** — оба участника соединения являются одновременно инициаторами и получателями данных, что критично для интерактивных сценариев реального времени, где синхронный request-response цикл создаёт неприемлемую latency.

1. Когда запросы может отправлять только клиент, простейшим решением для поддержания его в актуальном состоянии на сервере является простой запрос обновлений. Это может быть достигнуто путем многократного запроса одного и того же ресурса, метод, известный как опрос (polling). **Решение неэффективно**. Большинство запросов, которые делает клиент, пропадают впустую, потому что ничего не меняется.
2. Одна из оптимизаций, основывающаяся на опросе, называется **длительным опросом (long polling).** При **длительном опросе** используется та же идея, что и при повторном запросе обновлений клиентом к серверу, но с изюминкой: сервер не отвечает немедленно. Вместо этого сервер ждет, пока что-то изменится, а затем отвечает обновлением.
3. Другой доработкой стали webhooks - технология, при которой клиент одновременно и отправляет запросы, и прослушивает их, позволяя серверу легко отправлять на него обновления. Webhooks работает благодаря тому, что клиент тоже становится сервером. С технической точки зрения, иногда очень просто расширить функциональность клиента, чтобы он также мог прослушивать запросы, обеспечивая двустороннюю связь.

В своей простейшей форме **webhooks** требует, чтобы клиент предоставлял **URL**-адрес обратного вызова, по которому он может получать события, а на сервере было место, где пользователь мог бы ввести этот **URL**-адрес обратного вызова. Затем, всякий раз, когда что-то меняется на сервере, сервер может отправить запрос на **URL** **обратного вызова** (**callback**) клиента, чтобы сообщить клиенту об обновлении.

**Современные подходы к двусторонней коммуникации клиент — сервер**

**WebSocket** обеспечивает постоянное двунаправленное соединение поверх TCP, позволяя клиенту и серверу обмениваться сообщениями в реальном времени без дополнительных HTTP‑запросов. Этот протокол оптимален для чатов, онлайн‑игр, торговых платформ и дашбордов с живыми данными, где критична минимальная задержка. Среди преимуществ — низкая задержка, экономия ресурсов за счёт одного соединения для множества сообщений и поддержка бинарных данных. Однако требуется специальная серверная поддержка, а масштабирование на тысячи соединений может быть сложным; кроме того, соединение не кэшируется прокси‑серверами и CDN.

**Server‑Sent Events (SSE)** реализует однонаправленный поток событий от сервера к клиенту по HTTP через тип содержимого `text/event-stream`. Клиент открывает соединение и слушает события, а сервер отправляет обновления. SSE подходит для новостных лент, уведомлений и логов в реальном времени, где достаточно передачи данных только от сервера к клиенту. Плюсы: простота реализации (используется стандартный HTTP), автоматическое переподключение при обрыве, встроенная поддержка в браузерах через API `EventSource` и возможность кэширования прокси. Ограничения: однонаправленность, ограниченная поддержка в старых браузерах и максимум одно соединение на домен в браузере.

**GraphQL Subscriptions** расширяет GraphQL возможностью реал‑тайм обновлений: клиент подписывается на события, а сервер уведомляет его при изменении данных. Этот подход уместен в приложениях на GraphQL, требующих реактивности (соцсети, таск‑трекеры), особенно если уже используется GraphQL и нужно сохранить единообразие API. Преимущества: интеграция с GraphQL‑экосистемой, точная подписка на нужные поля (без избыточных данных) и гибкость запросов. Недостатки: необходимость отдельной серверной реализации (обычно поверх WebSocket или SSE) и более сложная настройка по сравнению с комбинацией REST и WebSocket.

**MQTT** — лёгкий протокол публикации/подписки (pub/sub) поверх TCP/IP, оптимизированный для IoT и низкоскоростных сетей. Он эффективен для устройств с ограниченной мощностью (датчики, микроконтроллеры), распределённых систем с множеством узлов и сетей с нестабильным соединением. Ключевые плюсы: минимальный overhead (заголовки около 2 байт), поддержка уровней качества обслуживания (QoS) для гарантированной доставки, а также встроенные механизмы keep‑alive и ретрансмиссии. Минусы: нестандартность для веб‑браузеров (требуется брокер и шлюзы) и меньшее количество инструментов для веб‑разработки.

**WebTransport** — новый стандарт W3C для двунаправленной передачи данных поверх HTTP/3 (QUIC), сочетающий низкую задержку WebSocket с безопасностью и мультиплексированием HTTP/3. Подходит для будущих веб‑приложений, где критична высокая производительность и устойчивость к потере пакетов. Преимущества: низкая задержка благодаря QUIC, встроенное шифрование (TLS 1.3) и мультиплексирование потоков. Недостатки: экспериментальный статус (ограниченная поддержка в браузерах) и мало готовых решений.

**Firebase Realtime Database / Firestore** представляет собой BaaS (Backend as a Service) от Google с готовой реал‑тайм синхронизацией данных: клиент подписывается на изменения коллекции или документа. Решение идеально для быстрых прототипов, стартапов и мобильных приложений, где важно избежать написания серверной логики. Плюсы: готовая реал‑тайм функциональность, простая интеграция через SDK для Web, iOS и Android, а также масштабируемость без дополнительного администрирования. Минусы: привязка к экосистеме Google, возрастающая стоимость при высоком трафике и ограниченная кастомизация.

**Kafka + HTTP/WebSocket шлюзы** используют Apache Kafka как брокер событий, а поверх него разворачивают HTTP/WebSocket API для клиентов. Подход актуален для крупных систем с высокой нагрузкой (тысячи событий в секунду) и микросервисных архитектур с event‑driven взаимодействием. Преимущества: гарантированная доставка, репликация, отказоустойчивость, возможность буферизации и обработки потоков, а также интеграция с инструментами Big Data. Недостатки: сложность развёртывания и мониторинга, а также высокий порог входа для команды.

Вебхуки (webhooks) — оптимальный выбор для снижения нагрузки на сервер при сохранении возможности оперативно информировать клиентов об обновлениях. Они работают по принципу «проталкивания» (push): сервер самостоятельно отправляет HTTP POST-запрос клиенту **только при возникновении события** (изменении данных). Это кардинально отличается от постоянного поддержания соединений (как в WebSocket), что решает проблему перегрузки сервера

# Асинхронные (преимущественно событийные/потоковые)






### WEBSOCKET
Протокол для постоянного двустороннего соединения поверх TCP. В отличие от классического HTTP, где каждый запрос открывает и закрывает канал, WebSocket устанавливает соединение один раз через стандартное рукопожатие, после чего переводит канал в режим полного дуплекса.

Данные передаются в обе стороны мгновенно в виде бинарных или текстовых фреймов без избыточных заголовков. Протокол требует специальной поддержки на стороне инфраструктуры (обратная совместимость, балансировка по upgrade-заголовкам), но обеспечивает предсказуемое поведение в реальном времени.

Используется там, где критична минимальная задержка и непрерывный обмен: онлайн-игры, мессенджеры, торговые терминалы и совместные редакторы. Клиент и сервер могут инициировать передачу в любой момент без ожидания запроса.

WebSocket — это транспортный уровень, а не готовый протокол прикладного уровня. Он полностью асинхронен и stateful, поэтому разработчик сам должен реализовать логику повторного подключения, контроля состояния и маршрутизации сообщений поверх открытого канала.

WebSocket создаёт постоянное полнодуплексное соединение, позволяя клиенту и серверу обмениваться бинарными или текстовыми фреймами в любой момент без overhead HTTP-заголовков, что идеально для интерактивных сценариев.
```javascript
// Клиент
const ws = new WebSocket("wss://chat.example.com/stream");
ws.onmessage = (msg) => console.log(JSON.parse(msg.data));
ws.send(JSON.stringify({ type: "chat", text: "Hello" }));
// Сервер принимает фреймы и рассылает их подключённым пирам
```

### MQTT
Легковесный протокол обмена сообщениями, построенный по архитектуре «издатель-подписчик». Устройства не общаются напрямую, а отправляют данные центральному брокеру, который маршрутизирует сообщения нужным подписчикам по иерархическим темам.

Протокол использует минималистичные заголовки, поддерживает три уровня качества доставки (QoS 0/1/2) и эффективно работает в условиях нестабильных сетей с ограниченной пропускной способностью. Компактность и низкое потребление ресурсов делают его идеальным для ограниченных устройств.

MQTT стал отраслевым стандартом для интернета вещей, телеметрии, умных домов и мобильных push-уведомлений. Брокер берёт на себя гарантию доставки, хранение ретранслируемых сообщений и управление сессиями.

MQTT — это чисто асинхронный протокол, оптимизированный для событийной модели IoT. Его сила в декоуплинге издателей и подписчиков, но это же требует централизованного брокера и явного управления топиками, что добавляет архитектурную сложность по сравнению с прямыми каналами.

MQTT использует архитектуру «издатель-подписчик», где клиенты публикуют сообщения в топики на брокере, а подписчики получают их асинхронно, гарантируя доставку через уровни QoS даже в нестабильных сетях.
```text
# Публикация
PUBLISH topic="home/livingroom/temp" qos=1 payload="{\"celsius\": 22.5}"
# Подписка
SUBSCRIBE topic="home/+/temp" qos=1
# Брокер маршрутизирует сообщение всем подписчикам с совпадающим паттерном
```

### SSE (Server-Sent Events)
Односторонний канал, позволяющий серверу непрерывно отправлять текстовые обновления клиенту по обычному HTTP. В отличие от WebSocket, соединение остаётся в рамках стандарта HTTP/1.1, не требует специального рукопожатия и автоматически восстанавливается браузером при обрывах связи.

Клиент только получает поток событий с возможностью именования, меток и повторной передачи, но не может отправлять данные по тому же каналу. Подход нативно поддерживается современными браузерами и прост в реализации на любом бэкенде.

Используется для новостных лент, мониторинга статусов, трансляций и AI-стриминга, где достаточно однонаправленной передачи данных. Ответы на действия пользователя отправляются отдельными POST-запросами, что сохраняет простоту архитектуры.

SSE — это асинхронный push-механизм, который жертвует двунаправленностью ради простоты и совместимости с существующей HTTP-инфраструктурой. Он идеален, когда клиенту достаточно только слушать, а обратная связь не требует постоянного канала.

SSE предоставляет однонаправленный HTTP-канал, по которому сервер непрерывно отправляет текстовые события с метками и ID, а браузер автоматически восстанавливает соединение при обрывах и доставляет данные через нативный `EventSource`.
```text
HTTP/1.1 200 OK
Content-Type: text/event-stream

event: price_update
id: 42
data: {"asset": "BTC", "price": 68500}

event: heartbeat
 {"ts": 1712900000}
```

### WEBHOOKS
Механизм обратного вызова, при котором сервер инициирует отправку HTTP-запроса клиенту в момент наступления определённого события. Вместо постоянных опросов приложение регистрирует публичный URL, куда будет приходить полезная нагрузка в формате JSON или XML.

Подход переводит взаимодействие в событийную модель, экономя ресурсы обеих сторон и обеспечивая почти мгновенную доставку уведомлений. Интеграция требует минимального кода, но подразумевает ответственность получателя за обработку ошибок и идемпотентность.

Webhooks повсеместно используются в платежных шлюзах, CI/CD пайплайнах, Git-репозиториях и интеграциях между SaaS-сервисами. Это стандартный способ связать разрозненные системы без постоянного опроса.

Webhooks — это асинхронный паттерн доставки событий, а не постоянный канал связи. Их надёжность зависит от реализации retry-логики, идемпотентности обработчиков и верификации подписей, так как сеть может терять или дублировать вызовы.

Webhooks реализуют обратные вызовы, при которых сервер-источник самостоятельно инициирует HTTP POST-запрос на заранее зарегистрированный URL получателя в момент наступления события, перенося логику ожидания на инициатора.
```http
POST /webhooks/payment-received HTTP/1.1
Host: myapp.com
X-Signature: sha256=abc123...
Content-Type: application/json

{ "event": "payment.succeeded", "data": { "invoice_id": "INV-991", "amount": 150 } }
```

### HTTP/2 PUSH
Функция протокола, позволяющая серверу проактивно отправлять ресурсы клиенту до того, как тот явно запросит их в разметке. Идея заключалась в ускорении загрузки страниц за счёт устранения лишних round-trip задержек и параллельной передачи данных в одном мультиплексированном соединении.

Сервер анализировал начальный запрос и пушил связанные CSS, JS или изображения. Однако на практике технология оказалась сложной в управлении и часто конфликтовала с механизмами кэширования браузеров, отправляя уже сохранённые файлы.

На современном этапе браузеры постепенно отключили её поддержку в пользу более предсказуемых альтернатив вроде `<link rel="preload">` и статуса `103 Early Hints`. Архитектура сместилась в сторону декларативного управления ресурсами.

HTTP/2 Push — это асинхронная оптимизация на транспортном уровне, которая оказалась избыточной. Современные браузеры отказались от неё, так как проактивная отправка без знания реального кэша клиента часто приводила к бесполезному расходу трафика и ухудшению метрик производительности.

HTTP/2 Push позволяет серверу проактивно отправлять статические ресурсы в том же мультиплексированном соединении до того, как браузер обнаружит их в HTML, экономя round-trip задержки.
```http
HTTP/2 200 OK
Content-Type: text/html
Link: </styles.css>; rel=preload; as=style, </app.js>; rel=preload; as=script
# Сервер автоматически отправляет байты стилей и скриптов в параллельных stream'ах
```

### WEBTRANSPORT
WebTransport — это современный протокол обмена данными, построенный поверх HTTP/3 и QUIC. Он предлагает более гибкую и производительную альтернативу WebSocket, поддерживая как надёжные потоки, так и ненадёжные датаграммы в рамках одного соединения.

Протокол использует UDP в качестве транспортной основы, что полностью устраняет проблему head-of-line blocking, характерную для TCP. Клиент и сервер могут динамически открывать несколько независимых каналов с разными гарантиями доставки, передавать неупорядоченные датаграммы с минимальными задержками и управлять приоритетами трафика на лету.

Технология идеальна для сценариев, где критична гибкость и низкая латентность: облачный гейминг, стриминг в реальном времени, телеметрия с IoT-датчиков и распределённые приложения, сочетающие критичные команды и фоновую аналитику. Нативная поддержка в современных браузерах и серверных рантаймах упрощает внедрение без тяжёлых зависимостей.

WebTransport — это эволюционный гибридный транспорт, который заменяет бинарный выбор между WebSocket и HTTP/2. Он позволяет разработчику явно указывать тип доставки для каждого сообщения, но требует инфраструктуры с поддержкой UDP/QUIC и современных сертификатов (HTTP/3), что пока ограничивает его использование в корпоративных сетях со строгими правилами фильтрации трафика.

WebTransport объединяет надёжные стримы и ненадёжные датаграммы в одном QUIC-соединении, позволяя выбирать модель доставки под задачу прямо в коде API без переключения между разными протоколами.
```javascript
// Клиент
const transport = new WebTransport('https://api.example.com/live');
await transport.ready;

// Надёжный двунаправленный стрим (как TCP, но без HOL-блокировки)
const stream = await transport.createBidirectionalStream();
stream.writable.getWriter().write(new TextEncoder().encode('{"cmd":"join_room","id":"42"}'));

// Ненадёжные датаграммы с минимальной задержкой (как UDP)
transport.datagrams.writable.getWriter().write(new Uint8Array([0x01, 0x02, 0xFF]));

// Сервер обрабатывает оба типа каналов через один endpoint /live
```
