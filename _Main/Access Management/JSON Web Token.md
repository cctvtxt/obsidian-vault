[[JWT-Based Auth]]

Формат токена прописан в открытом стандарте RFC 7519, представляет собой формат данных и механизм подписи для безопасной передачи claims (утверждения, или key-value пары в полезной нагрузке токена) между сторонами

**JWT** — это самодостаточный формат для передачи утверждений (**claims**) в JSON между сторонами. JWT передаёт claims stateless-способом: получатель извлекает данные локально после проверки, без БД

JWT используется как access token, refresh токеном в паре с ним является opaque-токен без содержимого. Имеет несколько форм, зависящих от выбранного подхода к защите данных: **JWS**, **JWE**, **JWS-inside-JWE**. По умолчанию (**JSON Web Signature**) используется подпись для проверки подлинности токена и данных в нем: все содержимое токена доступно для чтения, даже если оно подписано, таким образом, при помощи JWT принято передавать идентификаторы, флаги, роли, служебные данные: iat, exp, iss, aud, не персональные данные

## Категории claims 

Основное, что вводит стандарт RFC 7519, это **зарегистрированные** утверждения в payload

- iss (issuer) — издатель токена;
- sub (subject) — назначение токена;
- aud (audience) — аудитория, для которой предназначен токен;
- exp (expire time) — срок действия токена;
- nbf (not before) — время, до которого токен недействителен;
- iat (issued at) — время создания токена;
- jti (JWT id) — уникальный идентификатор токена.

Также claims могут быть **публичными**, то есть пользовательскими с уникальными именами, и **приватными** `name`, `email`, `role`, `scope` и так далее
## Безопасность 

Защита JWT реализуется криптографически и применяется к полной структуре (header + payload)

**Подпись**: создаётся по секретному/приватному ключу (HMAC, RSA, ECDSA); получатель проверяет по публичному/секретному, **гарантируя целостность и аутентичность**: данные не подделаны и от доверенного издателя

<font color="#de7802">HEADER</font>.<font color="#245bdb">PAYLOAD</font>.<font color="#c00000">SUGNATURE</font>

**Шифрование**: payload (и иногда header) шифруется симметричным/асимметричным алгоритмом (AES-GCM, RSA-OAEP); **обеспечивает конфиденциальность**: содержимое скрыто от посторонних

<font color="#ffc000">PROTECTED</font>.<font color="#6425d0">ENCRYPTED_KEY</font>.<font color="#ff0000">IV</font>.<font color="#245bdb">CHIPHERTEXT</font>.<font color="#00b050">AUTH_TAG</font>

**Шифрование + Подпись**: комбинированный подход, когда JWS полностью шифруется внутри ciphertext

<font color="#ff0000">CHIPHERTEXT</font> = <font color="#de7802">HEADER</font>.<font color="#245bdb">PAYLOAD</font>.<font color="#c00000">SUGNATURE</font>
___
В контексте TLS JWS остаётся достаточным и **по умолчанию рассматривается именно такая форма токена**

1. HTTPS (TLS 1.3) обеспечивает конфиденциальность, целостность и аутентификацию трафика end-to-end между клиентом и сервером — токен не читаем в сети
2. **JWS** lобавляет app-level проверку claims (не подделан, от issuer), независимо от TLS — сервер верифицирует stateless без сессий
3. **JWS**-защита требует меньше вычислений, дате меньший размер токена, чем **JWE**-защита

Данный подход недостаточен в случае отсутствия TLS, необходимости передавать HIPAA/PCI DSS, finance чувствительные данные, а также при большом TTL
## Основная форма токена (JWS-подход)

<font color="#de7802">HEADER</font>.<font color="#245bdb">PAYLOAD</font>.<font color="#c00000">SUGNATURE</font>

_**Заголовок**_ содержит алгоритм подписи (например, HS256) и тип токена - JWT или JOSE (редко)

`{ "alg": "HS256", "typ": "JWT" }`

Base64 encoded: `eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9`

_**Полезная нагрузка**_ содержит основные утверждения (claim), такие как идентичность пользователя, роль пользователя и время истечения срока действия

`{ "sub": 123123, "name": "John Doe", "role": "guest" }`

Base64 encoded: `eyJzdWIiOiIxMjM0NTY3ODkwIiwibmtZI6IkpvaG9lIQ`

_**Подпись**_ использует ключ для подписания заголовка и полезной нагрузки, что позволяет проверить, была ли подпись подделана.

`HMAC-SHA256(base64Url(header) + '.' + base64Url(payload), secret)`

Generate via algorithm: `XM-XSs2Lmp76IcTQ7tVdFcZ zN4W_WcoKMNANp925Q9g`
## Генерация на сервере

**Криптографическая подпись** создается с использованием секретного ключа, 

соединяют их через точку, а затем к этой строке применяют алгоритм подписи с ключом






Порядок создания подписи JWT:

1. **Кодирование заголовка и полезной нагрузки** при помощи алгоритма base64url.
2. **Объединение закодированных частей.** Для этого используется точка («.») в качестве разделителя.
3. **Генерация подписи.** Для этого может применяться, например, алгоритм HMAC-SHA256. Подпись добавляется к исходной строке также через точку.

## Отправка на клиент

## Хранение на клиенте 

**использовать куки с атрибутами `HttpOnly` и `Secure` для хранения токенов.**

**Обоснование:**

Этот метод является наиболее безопасным для хранения и передачи JWT-токенов, так как сочетает несколько уровней защиты

1. **Защита от XSS-атак (Cross-Site Scripting):**

- атрибут `HttpOnly` делает куки недоступными для клиентского JavaScript-кода;
- даже если злоумышленник внедрит вредоносный скрипт на страницу, он не сможет получить доступ к токену, так как тот хранится в куки и не читается через `document.cookie` или `localStorage`;
- это критически важно для браузерных приложений, где XSS — одна из самых распространённых угроз  +1.

2. **Передача только по защищённому соединению:**

- атрибут `Secure` гарантирует, что куки передаются только через HTTPS-соединение;
- это предотвращает перехват токена при передаче по незащищённому каналу (MITM-атаки)  +1.

3. **Автоматическая отправка с запросами:**

- браузер автоматически включает куки в каждый HTTP-запрос к домену, для которого они установлены — не требуется дополнительных действий со стороны клиентского кода;
- это упрощает реализацию на стороне клиента по сравнению с ручным добавлением токена в заголовок `Authorization`.

4. **Совместимость с мобильными приложениями:**

- куки поддерживаются как в веб-вью (WebView) мобильных приложений, так и в нативных HTTP-клиентах;
- большинство HTTP-библиотек для мобильных платформ (OkHttp, Alamofire и др.) автоматически отправляют сохранённые куки с запросами.

5. **Контроль над сроком действия и областью действия:**

- можно настроить параметры `Expires` / `Max-Age` для управления сроком жизни токена;
- атрибут `SameSite` (например, `SameSite=Strict` или `SameSite=Lax`) защищает от CSRF-атак, ограничивая отправку куки только для запросов с того же сайта .

## Отправка на сервер

## Валидация на сервере





Хранение секретного ключа:

**Клиент** обладает только JWT-токеном, ему не передается секретный ключ.

**Сервер** хранит ключ в переменных окружения / хранилище секретов.

Хранение JWT-токена:

**Сервер** не хранит JWT, в этом основное преимущество данного метода.

**Клиент** сохраняет токен в веб-хранилище (**_localStorage_** или **_sessionStorage_**), в таком случае браузер не будет автоматически включать что-либо из веб-хранилища в HTTP-запросы, что делает его неуязвимым для **CSRF**, доступ к данным может быть получен только с помощью js, работающего в том же домене. Однако токен недоступен для js, работающего в поддомене, который (возможно) создал эти данные (значение, записанное example.com, не может быть прочитано как sub.example.com). В данном случае токен уязвим для **XSS**

Либо **клиент** сохраняет токен в **cookie**, в таком случае токен неуязвим для XSS, файл cookie может быть создан в домене верхнего уровня и использоваться в запросах, выполняемых поддоменами, при этом необходимо знать и учитывать возможное использование файлов cookie в поддоменах. При таком хранении токен уязвим для **CSRF**, и если не быть осторожным, то можно реализовать стратегию защиты от **CSRF**, которая уязвима для **XSS**.

Все части закодированы в **_Base64URL_**.

```javascript
const jwt = require("jsonwebtoken")
const token = "eyJhbGciOiJIUzI1NiIs.InR5cCI6IkpXVCJ9.eyJzdWIiOiJ1c2V"
const secret = "your-secret-key" // или публичный ключ для RS256
const verified = jwt.verify(token, secret, { algorithms: ["HS256"] })
```

При неправильной реализации со стороны разработчиков возрастает риск компрометации конфиденциальных данных пользователей. Поэтому при использовании JWT следует строго соблюдать рекомендации по безопасности: ограничивать срок действия токенов и применять различные алгоритмы шифрования, такие как RS256, A128CTR и другие. Если все настроено правильно, JWT является отличным инструментом, который работает надежно и безопасно.
