[[JWT-Based Auth]]

Формат токена прописан в открытом стандарте RFC 7519, представляет собой формат данных и механизм подписи для безопасной передачи claims (утверждения, или key-value пары в полезной нагрузке токена) между сторонами

**JWT** — это самодостаточный формат для передачи утверждений (**claims**) в JSON между сторонами. JWT передаёт claims stateless-способом: получатель извлекает данные локально после проверки, без БД

В JWT claims делятся на три категории: зарегистрированные (стандартные) `iss` (издатель), `sub` (субъект/ID пользователя), `aud` (аудитория), `exp` (срок действия), `nbf` (не ранее), `iat` (выпущен), `jti` (уникальный ID), публичные и приватные. Зарегистрированные claims — это рекомендованные поля из RFC 7519 для совместимости, публичные — пользовательские с уникальными именами для избежания коллизий, а приватные — `name`, `email`, `role`, `scope`
______
Имеет несколько форм, зависящих от выбранного подхода к защите данных: **JWS**, **JWE**, **JWS-inside-JWE**. По умолчанию (**JSON Web Signature**) используется подпись для проверки подлинности токена и данных в нем: все содержимое токена доступно для чтения, даже если оно подписано, таким образом, при помощи JWT принято передавать идентификаторы, флаги, роли, служебные данные: iat, exp, iss, aud, не персональные данные
## Безопасность 

Защита JWT реализуется криптографически и применяется к полной структуре (header + payload):

**Подпись**: создаётся по секретному/приватному ключу (HMAC, RSA, ECDSA); получатель проверяет по публичному/секретному, **гарантируя целостность и аутентичность**: данные не подделаны и от доверенного издателя

<font color="#de7802">HEADER</font>.<font color="#245bdb">PAYLOAD</font>.<font color="#c00000">SUGNATURE</font>

**Шифрование**: payload (и иногда header) шифруется симметричным/асимметричным алгоритмом (AES-GCM, RSA-OAEP); **обеспечивает конфиденциальность**: содержимое скрыто от посторонних

<font color="#ffc000">PROTECTED</font>.<font color="#6425d0">ENCRYPTED_KEY</font>.<font color="#ff0000">IV</font>.<font color="#245bdb">CHIPHERTEXT</font>.<font color="#00b050">AUTH_TAG</font>

**Шифрование + Подпись**: 

внутренний JWS "спрятан" внутри зашифрованного ciphertext

В контексте TLS JWS остаётся достаточным для большинства случаев, но не всегда, если требуется дополнительная защита за пределами транспортного уровня.

## Почему JWS + TLS достаточно

- **TLS шифрует канал**: HTTPS (TLS 1.3) обеспечивает конфиденциальность, целостность и аутентификацию трафика end-to-end между клиентом и сервером — токен не читаем в сети.
- **JWS дополняет подписью**: Добавляет **app-level** проверку claims (не подделан, от issuer), независимо от TLS — сервер верифицирует stateless без сессий.
- **Стандарт практика**: OAuth2/OIDC API (Google, Auth0) используют JWS over HTTPS — payload с ID/ролями не sensitive, т.к. не содержит секретов.

## Когда JWS + TLS недостаточно

- **Сохранение/логирование токена**: Payload читаем в логах прокси (NGINX, API Gateway), browser storage (LocalStorage), утечках — TLS не защищает после получения.
- **Многоуровневые системы**: Прокси, load balancers, микросервисы видят/логируют JWS claims; нужен JWE для "zero-knowledge".
- **Высокие compliance**: HIPAA/PCI DSS, finance — требуют шифрования claims даже over TLS (e.g., медицинские данные).
- **Долгоживущие токены**: Если JWT хранится клиентски или TTL большой, JWE предотвращает misuse при компрометации.

## Основная форма токена (JWS-подход)

<font color="#de7802">HEADER</font>.<font color="#245bdb">PAYLOAD</font>.<font color="#c00000">SUGNATURE</font>

_**Заголовок**_ содержит алгоритм подписи (например, HS256) и тип токена - JWT или JOSE (редко)

`{ "alg": "HS256", "typ": "JWT" }`

Base64 encoded: `eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9`

_**Полезная нагрузка**_ содержит основные утверждения (claim), такие как идентичность пользователя, роль пользователя и время истечения срока действия

`{ "sub": 123123, "name": "John Doe", "role": "guest" }`

Base64 encoded: `eyJzdWIiOiIxMjM0NTY3ODkwIiwibmtZI6IkpvaG9lIQ`

_**Подпись**_ использует ключ для подписания заголовка и полезной нагрузки, что позволяет проверить, была ли подпись подделана.

`HMAC-SHA256(base64Url(header)+'.'+base64Url(payload), secret`

Generate via algorithm: `XM-XSs2Lmp76IcTQ7tVdFcZ zN4W_WcoKMNANp925Q9g`
