[[TLS-Based Auth]]

**Приватный (закрытый) ключ (Private Key)** — это секретный компонент криптографической пары ключей, используемой в асимметричном шифровании. Он должен храниться в строжайшем секрете и никогда не покидать пределы своей защищенной среды. Его основное назначение — расшифровка данных, зашифрованных соответствующим публичным ключом, и создание цифровых подписей.

---
## Генерация

Приватный ключ всегда генерируется в паре с `Public Key`.

*   **Клиент и Сервер**: Генерация выполняется с помощью стандартных криптографических утилит или библиотек.
    ```bash
    # Генерация пары RSA ключей (приватный и публичный)
    openssl genpkey -algorithm RSA -out private_key.pem -pkeyopt rsa_keygen_bits:2048

    # Извлечение публичного ключа из приватного
    openssl rsa -pubout -in private_key.pem -out public_key.pem
    ```
    Алгоритмы: `RSA`, `ECDSA`.

## Хранение

**Правило №1: Приватный ключ — это ваш главный секрет. Его компрометация ведет к компрометации всей системы.**

### Сервер
*   **Файловая система**: Хранится в файле с максимально ограниченными правами доступа (например, `chmod 400 private_key.pem`), владельцем которого является только пользователь, запускающий приложение.
*   **Переменные окружения**: Содержимое ключа передается в переменной окружения (например, из `.env` файла, который **никогда** не коммитится в Git).
*   **Секрет-менеджеры**: Наиболее безопасный способ. Ключ хранится в специализированных системах, таких как `HashiCorp Vault`, `AWS Secrets Manager` или `Kubernetes Secrets`. Приложение получает к нему доступ в момент запуска.

### Клиент (Пользовательское устройство)
*   Хранится в защищенной системной директории, например, `~/.ssh/id_rsa` для SSH-ключей. Доступ к таким файлам должен быть только у владельца.

## Передача

**Приватный ключ никогда и ни при каких обстоятельствах не должен передаваться по сети.** Он должен генерироваться и храниться в том месте, где будет использоваться.

## Использование
*   **Аутентификация**: Клиент доказывает владение приватным ключом, подписывая случайное сообщение (`challenge`), отправленное сервером. Сервер проверяет подпись с помощью заранее известного публичного ключа клиента.
*   **Расшифровка**: Сервер расшифровывает своим приватным ключом данные (например, симметричный ключ сессии), которые были зашифрованы клиентом с помощью публичного ключа сервера.
