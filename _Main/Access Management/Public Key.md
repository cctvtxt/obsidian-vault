[[TLS-Based Auth]]

**Публичный (открытый) ключ (Public Key)** — это несекретный компонент криптографической пары ключей, который предназначен для свободного распространения. Он математически связан с `Private Key`, но получить приватный ключ из публичного невозможно. 

Публичные и приватные ключи используются в **асимметричной криптографии**: один ключ шифрует или проверяет, а второй — расшифровывает или подписывает. Основные процессы — это шифрование и цифровая подпись данных.

Подпись решает две задачи:

- **Целостность**: после подписи любая правка данных делает подпись недействительной.
- **Подлинность**: проверяющая сторона может убедиться, что подпись создал именно владелец приватного ключа.

Шифрование решает одну основную задачу:

- **Конфиденциальность:** данные становятся недоступны для посторонних и могут быть прочитаны только тем, у кого есть нужный ключ.

---
## Генерация

Публичный ключ генерируется (извлекается) из `Private Key` в момент создания криптографической пары.

```bash
# Извлечение публичного ключа из уже существующего приватного
openssl rsa -pubout -in private_key.pem -out public_key.pem
```

## Хранение

Публичный ключ не является секретом и может храниться открыто.

### Сервер
*   **База данных**: Хранение публичных ключей клиентов для проверки их подписей.
*   **Файлы конфигурации**: Например, файл `~/.ssh/authorized_keys` на сервере хранит публичные ключи пользователей, которым разрешен SSH-доступ.

### Клиент
*   Клиент хранит публичные ключи серверов, которым он доверяет, чтобы шифровать для них данные или проверять их подлинность (например, файл `~/.ssh/known_hosts`).

## Передача

Публичный ключ можно безопасно передавать по любым, даже незащищенным каналам связи, так как его компрометация не несет прямого риска безопасности. Обычно он предоставляется всем сторонам, которые должны взаимодействовать с владельцем соответствующего приватного ключа.

## Использование
*   **Аутентификация**: Сервер использует публичный ключ клиента для проверки подписи, созданной приватным ключом клиента.
*   **Шифрование**: Клиент использует публичный ключ сервера для шифрования данных, которые затем могут быть расшифрованы только приватным ключом сервера. Это часто используется для безопасной передачи симметричного ключа при установке `TLS`-соединения.








Симметричное шифрование — ключ нужен, и это главная проблема. Один ключ, он же шифрует, он же расшифровывает. AES работает именно так. Проблема фундаментальная: как передать этот ключ собеседнику, если канал открытый? Если ты отправишь ключ по интернету — его перехватят и расшифруют всё остальное. Поэтому симметричное шифрование само по себе в вебе не применяется для первого соединения — только после того, как ключ был согласован через асимметричный обмен (это и есть суть TLS handshake). Дальнейший HTTP-трафик уже шифруется AES — он быстрый.

Асимметричное — 2 ключа, проблема передачи решена
Публичный ключ можно слать кому угодно — он буквально публичный. Зашифровать им может любой, расшифровать — только владелец приватного. Так решается проблема первой передачи: сервер шлёт публичный ключ открыто, клиент шифрует им pre-master secret, сервер расшифровывает приватным. Никто в сети это не прочтёт. Минус — медленно, поэтому так не шифруют весь трафик.

Хэширование — никаких ключей, это вообще не шифрование
Хэш — это односторонняя функция. SHA-256(password) даёт фиксированную строку, из которой пароль не восстановить. Ключей нет вообще. Применяется не для "скрыть данные", а для "проверить что данные те же". Пароль в БД хранится как bcrypt(password + salt), при логине просто сравниваются два хэша.

HMAC — это хэш с секретом, симметричный
HMAC = hash(data + secret). Тут появляется один симметричный ключ — secret. Но он не передаётся — он известен обеим сторонам заранее (выдан при регистрации, в env-переменных). В сети летит только результат — подпись. Используется в JWT HS256: сервер подписывает токен своим секретом, он же его и проверяет. Если у тебя один сервер — ок. Если микросервисы — проблема: все сервисы должны знать секрет, значит любой из них может подделать токен.

Цифровая подпись — асимметрия, но наоборот
В шифровании: публичным шифруют → приватным расшифровывают. В подписи всё перевёрнуто: приватным подписывают → публичным проверяют. Приватный ключ знает только автор (auth-сервер), проверить подпись может любой у кого есть публичный ключ. Это и есть JWT RS256: любой API-сервер может проверить токен, но подписать новый — только auth-сервер. Именно поэтому RS256 предпочтительнее HS256 в микросервисах.
