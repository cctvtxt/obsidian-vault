[[SAML-Based Auth]]

**SAML Assertion** — это XML-токен безопасности, содержащий утверждения (claims) о субъекте (пользователе, сервисе), который передаётся от **Identity Provider (IdP)** к **Service Provider (SP)**. Это фундаментальный элемент федеративной аутентификации в стандарте SAML 2.0. Также **SAML Assertion** — это старший брат JWT: оба являются **токенами безопасности**, которые переносят утверждения о субъекте. 

# Формат и структура

Документ соответствует строгой XML-схеме. Корневым элементом выступает `Assertion`, который содержит метаданные выпуска, подпись и блоки утверждений.

**Основные элементы структуры**:

`Issuer` указывает идентификатор IdP, выпустившего токен.
`Signature` обеспечивает целостность и аутентичность с помощью стандарта XML Signature.
`Subject` содержит идентификатор пользователя (NameID) и параметры подтверждения.
`Conditions` задаёт временные рамки валидности и ограничения по аудитории.
`AuthnStatement` фиксирует факт аутентификации, время и метод входа.
`AttributeStatement` передаёт дополнительные данные, такие как роли, группы или отделы.

Корневой элемент обязан содержать уникальный идентификатор (`ID`), версию стандарта (2.0) и метку времени выпуска (`IssueInstant`).

# Передача данных

Assertion не передаётся изолированно. Он всегда инкапсулируется в протокольный контейнер, выбор которого зависит от типа интеграции.

| Сценарий | Контейнер | Способ передачи |
|----------|-----------|-----------------|
| Browser SSO (POST) | SAML Response | HTTP POST, скрытое поле формы SAMLResponse в формате Base64 |
| Browser SSO (Redirect) | SAML Response | HTTP GET, параметр URL после сжатия Deflate и кодирования Base64 |
| SOAP/WSDL сервисы | WS-Security Header | Встраивание в SOAP-заголовок как Security Token |
| Artifact Resolution | Ссылка (Artifact) | Back-channel SOAP-запрос для получения полного токена |

При взаимодействии через браузер Assertion всегда оборачивается в элемент Response. В сценариях машинного взаимодействия токен размещается непосредственно в заголовке SOAP-сообщения.

Хранение и жизненный цикл

Assertion имеет короткий срок жизни и не предназначен для долгосрочного хранения. После получения SP выполняет проверку и при успехе создаёт локальную сессию.

Жизненный цикл состоит из четырёх этапов. IdP выпускает токен с указанием времени создания и срока действия. Клиент передаёт его на SP. SP кратковременно кэширует идентификатор для защиты от повторного использования. После создания локальной сессии или истечения срока валидности документ отбрасывается.

Для защиты от replay-атак SP обязан хранить значения полей ID в активном кэше в течение всего окна валидности. Повторное появление того же идентификатора отклоняется. Аудит ведётся в виде хэшей идентификаторов и метаданных, без сохранения полных XML-документов.

Валидация

Проверка Assertion требует последовательного выполнения нескольких этапов. Пропуск любого из них создаёт уязвимости.

| Этап | Описание проверки |
|------|-------------------|
| Структурная | Соответствие XSD-схеме, наличие обязательных элементов, корректность пространств имён |
| Криптографическая | Верификация XML-подписи доверенным сертификатом IdP, проверка алгоритмов (рекомендуется SHA-256), подтверждение, что подпись покрывает именно этот документ |
| Временная | Текущее время находится в интервале NotBefore – NotOnOrAfter с учётом допустимого рассинхрона часов (обычно до 2 минут) |
| Аудитория | Элемент AudienceRestriction содержит идентификатор текущего SP |
| Уникальность | Значение Assertion.ID не встречалось ранее в активном кэше |
| SubjectConfirmation | Совпадение Recipient с адресом ACS, проверка InResponseTo на соответствие ID исходного запроса |

Особое внимание уделяется защите от атак типа XML Signature Wrapping. Валидация схемы выполняется до извлечения данных. Навигация по дереву XML осуществляется через строгие XPath-выражения. Использование методов поиска по имени тега без предварительной проверки структуры недопустимо.

Практические рекомендации

Срок действия токена рекомендуется ограничивать 5–15 минутами. Большие окна повышают риски компрометации. Шифрование Assertion обязательно при передаче конфиденциальных данных. Сертификаты подписи IdP требуют регулярной ротации и проверки статуса отзыва.

При интеграции с современными API и микросервисами чаще применяются JWT. SAML Assertion остаётся стандартом для корпоративных порталов, legacy-систем и SOAP-интерфейсов, где требуется строгая типизация атрибутов и интеграция с корпоративными каталогами.
