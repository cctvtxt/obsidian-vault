[[Authentification]]

**Federated**-аутентификация необходима для сценариев, где приложение делегирует проверку удостоверения IdP, что обеспечивает **_единую точку входа, снижение нагрузки на разработку и повышенную безопасность за счёт использования отраслевых стандартов и централизованного управления доступом_**. 

Федерация — это архитектурный подход, при котором доверие к пользователю устанавливается через доверенного посредника. Например: вход через корпоративный Active Directory, Google Workspace, Okta, Microsoft Entra ID или Auth0. Приложение не хранит учётные данные, а получает подтверждённый набор утверждений (claims) о пользователе после успешной проверки в IdP.

При этом **federated-аутентификация является стандартом де-факто для современных веб‑приложений и микросервисных архитектур**. По умолчанию REST API легко интегрирует её через Bearer-токены, проверяя подпись JWT без обращения к базе данных сессий. Для сценариев с глубоким управлением ролями или кастомными потоками предпочтительны расширения OIDC, SCIM для синхронизации профилей или интеграция с enterprise‑SSO.
____
Примером реализации данного подхода является аутентификация через OpenID Connect / OAuth 2.0, кроме этого, существуют различные flows: Authorization Code, PKCE, Client Credentials, Device Flow и интеграции с SAML 2.0.
______
**Federated-аутентификация подходит для enterprise‑решений, B2B‑платформ и публичных сервисов**, где критичны удобство пользователей (SSO), соответствие стандартам безопасности и скорость вывода на рынок. **Для изолированных или legacy‑систем** с жёсткими требованиями к автономности или отсутствием доступа к внешним IdP данный подход может быть избыточным или неприменимым.

**Federated**-аутентификация обычно используется в корпоративных порталах, SaaS‑платформах и мобильных приложениях, где требуется единый вход из разных источников, а управление жизненным циклом учётных записей централизовано. В таких случаях приложение делегирует проверку IdP и работает только с проверенными утверждениями.
