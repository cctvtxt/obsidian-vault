[[Session-Based Auth]]

Классическая **сессионная аутентификация** основана на сохранении состояния соединения на стороне сервера. После успешной проверки учетных данных сервер генерирует уникальный идентификатор сессии (Session ID), который передается клиенту и привязывается к данным пользователя в серверном хранилище. При последующих запросах клиент предъявляет этот идентификатор, позволяя бэкенду мгновенно восстановить контекст авторизации без повторной передачи пароля или обращения к внешним системам.
__________
Если архитектура соответствует стандартам веб-безопасности (OWASP ASVS, NIST SP 800-63B), Session ID должен передаваться исключительно по защищенным каналам и не раскрываться в URL-адресах, логах, реферерах или клиентских хранилищах, доступных скриптам. В отличие от stateless-токенов, Session ID является stateful-объектом: его валидность целиком зависит от живой записи в серверном кэше или БД. Защищать необходимо механизм его выдачи, привязку к клиентскому окружению и процесс принудительной инвалидации.

Во-вторых, Session ID необходимо **строго отделять от данных профиля и прав доступа**: он служит исключительно ключом для поиска сессии в хранилище, а не контейнером информации о пользователе. Это позволяет мгновенно отзывать доступ, обновлять привилегии без пересоздания токенов и избегать проблем с рассинхронизацией прав на стороне клиента.
# Требования к токену

- Уникальный в глобальном масштабе, вероятность коллизий стремится к нулю
- Минимальная энтропия 128 бит (16 байт), рекомендовано 256 бит для устойчивости к перебору
- Сгенерирован криптографически стойким ГПСЧ (CSPRNG), например `crypto.randomBytes`, `/dev/urandom`
- Не содержит предсказуемых паттернов, последовательностей, временных меток или идентификаторов пользователя
- Имеет жестко ограниченное время жизни (TTL) с механизмом скользящего продления (sliding expiration)
- Поддерживает абсолютное время истечения (absolute timeout) для снижения рисков компрометации
# Места внедрения на клиенте

Session ID передается клиенту в заголовке `Set-Cookie` и должен храниться **исключительно в cookie** с наборами защитных флагов. Это гарантирует автоматическую привязку к домену и блокировку доступа из клиентского JavaScript.
```http
HTTP/1.1 200 OK
Set-Cookie: session_id=v1.8f7a9b2c...; HttpOnly; Secure; SameSite=Strict; Path=/; Domain=app.example.com; Max-Age=3600
```
**HttpOnly** запрещает чтение через `document.cookie`, **Secure** требует HTTPS, **SameSite** защищает от CSRF, **Path/Domain** ограничивают область отправки, **Max-Age** задает срок жизни.

В нативных мобильных приложениях Session ID может храниться в защищенных хранилищах ОС (iOS Keychain, Android Keystore). В веб-архитектуре использование `localStorage`, `sessionStorage` или in-memory переменных для Session ID **категорически не рекомендуется** из-за критической уязвимости к XSS-атакам.

При SPA-архитектуре с разделением доменов (фронтенд `*.example.com`, бэкенд `api.example.com`) cookie отправляется с флагами `SameSite=None; Secure` и обрабатывается клиентом через `credentials: "include"` в `fetch`/`axios`.
# Передача на сервер с запросом 

Session ID передается автоматически браузером в заголовке `Cookie` при каждом HTTP-запросе к соответствующему домену и пути. Ручное добавление в заголовки не требуется и противоречит модели безопасности веб-платформы.
```http
GET /api/dashboard HTTP/1.1
Host: api.example.com
Cookie: session_id=v1.8f7a9b2c...; _csrf=xyz789
Authorization: Bearer <access_token> (опционально, если используется гибридная схема)
```

Для небраузерных клиентов (CLI, микросервисные вызовы, legacy SDK) Session ID может передаваться в кастомном заголовке `X-Session-ID` или в теле запроса, но требует явной настройки CORS, ограничения по IP и дополнительной защиты от перехвата.
# Обработка и хранение на сервере

Session ID генерируется сервером при успешной аутентификации или принудительной ротации. Результат CSPRNG кодируется в hex или base64url. Сервер создает запись в быстром хранилище (Redis, Memcached, специализированные session-провайдеры), связывая ID с `user_id`, `roles`, `iat`, `last_access`, `device_fingerprint` и метаданными безопасности. Данные в сессии должны быть минимальны; чувствительная информация (пароли, полные JWT, финансовые реквизиты) не сохраняется.

При **stateful**-подходе целесообразно использовать in-memory хранилище с TTL для автоматической очистки протухших сессий. Middleware при входящем запросе извлекает ID из cookie, проверяет наличие в хранилище, сверяет `exp` и `last_access`, обновляет TTL (sliding expiration) и инжектирует контекст пользователя в объект запроса. Отсутствие или истечение записи возвращает `401 Unauthorized`.

**Ротация сессии** обязательна при изменении уровня привилегий: вход в систему, повышение роли, смена пароля, включение MFA. Старый ID удаляется, генерируется новый, клиенту отправляется `Set-Cookie` с обновленным значением и прежними флагами безопасности.

**Защита от Session Fixation**: сервер никогда не принимает Session ID от клиента до аутентификации. При успешном входе всегда создается новая сессия с новым ID, а предыдущие идентификаторы инвалидируются.

**Привязка к окружению** (рекомендуется): сервер может сохранять хэши `User-Agent` и подсети IP в метаданных сессии. При существенном отклонении (смена браузера, геолокации, прокси) сессия помечается как подозрительная, требует повторной аутентификации или принудительно завершается.

**Инвалидация**: при вызове `logout()` запись удаляется из хранилища, cookie очищается (`Max-Age=0; Path=/`), инициируется очистка сессии на всех связанных устройствах при необходимости. Для корпоративных систем поддерживается механизм принудительного завершения сессий администратором или при обнаружении аномальной активности через SIEM/EDR.
