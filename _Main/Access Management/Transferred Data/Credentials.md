[[Basic Auth]]

Учетные данные в аутентификации — это информация, которую пользователь предоставляет для подтверждения своей личности. Обычно это логин (или email) и пароль

Открытый стандарт RFC 5802 описывает способ безопасной обработки данных для аутентификации пользователей 
# Обработка учетных данных на клиенте

Шифрование на клиенте практически бесполезно из-за публичности исходного кода сайта и значительно усложняет логику приложения. Полноценное хеширование с использованием соли и перца также бесполезно из-за публичности по тем же причинам. Хэширование паролей на клиенте не защищает от атак **Pass-the-Hash**, при которых злоумышленник перехватывает передаваемый хэш и использует его для входа. Сервер не отличает скомпрометированный хэш от оригинального. 

- Общепринятой практикой является отсутствие какой-либо обработки учетных данных пользователя на клиенте
- Как дополнительная мера защиты может быть применено **хеширование без соли и перца** (они могут будут видны на клиенте), только с тяжелым алгоритмом, это защитит учетные данные от записи в логи сервера и дополнительно защитит их от считывания при передаче
# Передача учетных данных

- Для того, чтобы учетные данные не передавались в открытом виде, необходимо обеспечение **TLS-шифрования** соединения. TLS достаточно надёжно защищает передаваемые данные между клиентом и сервером. Это подтверждено большим количеством исследований и проверено временем. Пароли передаются **в body** или в заголовках **Authorization, Proxy-Authorization или WWW-Authenticate** 
# Хранение и обработка учетных данных на сервере

- **Логины сохраняться в исходном виде** и служить идентификаторами аккаунтов, их вредно хешировать, так как их необходимо использовать для поиска данных
- В БД **пароли в явном виде нельзя хранить**, их необходимо **хешировать**, затем помещать в БД, хеширование в обратную сторону невозможно
- При хешировании необходимо **генерировать и использовать соль** для каждого уникального пользователя, ее удобно хранить рядом с хешами паролей
- **Перец** для хеширования необходимо создавать единожды и хранить **в переменных окружения**
