[[Token-Based Auth]]

В сфере кибербезопасности и цифровой идентификации токены играют ключевую роль в обеспечении безопасной связи и аутентификации пользователей. Среди множества доступных токенов чаще всего обсуждаются **opaque (непрозрачные)** и **структурированные** токены. Такие токены выдаются после аутентификации пользователя. Они служат подтверждением личности и включаются в запросы.

Структурированные токены, в отличие от непрозрачных, содержат внутри себя данные. Данные структурированы в определенном формате, который можно расшифровать, чтобы получить содержащуюся в них информацию.

В контексте аутентификации токен обозначается как Auth Token, это понятие используется в **RFC 6749, 6750, 7519** и других стандартах. **Аутентификационный токен** представляет собой подписанный криптографическим ключом набор символов, подтверждающий аутентификацию пользователя 
_____
В зависимости от архитектуры и логики веб-приложение может одновременно использовать несколько типов токенов:

- Access Token — токен доступа к информации, обычно имеет время жизни несколько минут.
- Refresh Token — токен обновления, по которому можно только получить новую пару токенов (Access и Refresh), срок жизни измеряется днями.

Совместное использование этих токенов позволяет реализовать безопасную и удобную систему аутентификации.

Токены генерируются и выдаются сервером. При отправке учетных данных для получения токенов необходимо придерживаться общих правил обработки credentials
# Хранение токенов на клиенте

1. Токен может быть помещен в веб-хранилище (**localStorage или sessionStorage**), в этом случае токен будет неуязвим для **CSRF**, так как браузер не будет автоматически включать что-либо из веб-хранилища в HTTP-запросы, доступ к данным может быть получен только с помощью JavaScript, работающего в том же домене, в то же время недоступность токена в веб-хранилище для поддоменов может являться проблемой. Токен в веб-хранилище уязвим для **XSS**, потому что вредоносный JavaScript, выполнившийся в контексте страницы, получает полный доступ ко всем данным браузера.
2. Токен может быть сохранен в **cookie**. Это рекомендуемый способ хранения токена на клиенте: браузер автоматически включает токен в любой запрос, который соответствует спецификации заголовка cookie (домен, путь и срок действия), файл cookie может может использоваться в поддоменах. Токен в cookie неуязвим для **XSS** при указании **HttpOnly**, но по умолчанию уязвим для **CSRF,** если не используется SameSite=strict или CSRF-токен
# Передача токенов на сервер

1. Через HTTP-заголовок **Authorization** со схемой аутентификации Bearer — это является наиболее безопасным и рекомендуемым способом передачи токена.
2. Через **Cookie** (с HttpOnly) — хранение и передача через cookie позволяет автоматически прикреплять токен к каждому запросу.
3. Возможно передавать токены через параметры URL или POST-данные, что **не рекомендуется**, как минимум, потому что попадет в логи сервера. 
# Генерация токенов на сервере

1. Opaque-токены создаются с целью однозначно идентифицировать данные, генерируются как случайные высокэнтропийные строки (минимум 128 бит, base64url из crypto.randomBytes)
2. Access-токены создаются с помещением всех необходимых данных payload, JWT создаются добавлением цифровой подписи из header и payload в signature 
3. Refresh-токены всегда opaque, генерируются случайной уникальной строкой 
# Валидация токенов на сервере

1. **JWT** и любые другие **access**-токены необходимо проверять на сервере авторизации для распределения вычислительных мощностей. Проверка представляет собой проверку подписи (**signature** - 3-ей части JWT, которая должна представлять себя хеш от **payload** и **header** с секретной солью и перцем сервера).  
2. Проверка подлинности **refresh**-токена для выдачи нового access-токена конкретному пользователю происходит **на authorization server** по lookup в хранилище (БД/Redis), сервер извлекает `refresh_token` из POST-запроса и ищет в хранилище по точному совпадению (или хэшу), проверяет параметры пользователя и запускает функцию генерации нового access-токена
```sql
SELECT user_id, client_id, scopes, expires_at, revoked FROM refresh_tokens WHERE token_hash = SHA256('abc123def456') AND revoked = false AND expires_at > NOW();
```
3. Для помещения в БД токенов, которые применяются для прямого доступа к сессиям, необходимо реализовывать хеширование 
4. Для **блэклиста токенов** на сервере авторизации рекомендуется использовать Redis как основное хранилище благодаря высокой скорости доступа и поддержке TTL (time-to-live). Это позволяет автоматически удалять записи о заблокированных токенах по истечении их срока действия, предотвращая бесконтрольный рост списка. Альтернативы вроде БД (SQL) подходят для низконагруженных систем, но уступают в производительности при частых проверках
# Хранение токенов на сервере

1. Аутентификационные **access**-токены не предназначены для хранения
2. **Refresh**- и **opaque**-токены необходимо хранить конфиденциально на **authorization server**, не передавать **resource server**. При этом лучшей практикой хранения конфиденциальных данных в БД является хеширование (SHA-256) данных при хранении. В зависимости от продолжительности хранения и интенсивности обращений к данным используются либо **реляционные БД**, либо **in-memory кеш**.
