[[OIDC-Based Auth]]

Классический **OAuth 2.0** предназначен для делегирования доступа к ресурсам (авторизация), однако не предоставляет информации о личности пользователя. Для аутентификации поверх OAuth 2.0 надстроен протокол **OpenID Connect (OIDC)**, который вводит понятие **ID токена**. ID токен позволяет клиентскому приложению убедиться, что пользователь действительно прошел проверку у доверенного Identity Provider (IdP), и получить базовые утверждения (claims) о его идентификаторе без обращения к защищенному API.
____
Если архитектура соответствует **OpenID Connect Core**, то ID токен возвращается при обмене `authorization_code` на токены. В отличие от непрозрачного Access Token, который используется для вызова ресурсов, ID токен предназначен исключительно для идентификации. Защищать необходимо процесс его выдачи и валидации, чтобы злоумышленник не мог подменить личность или провести replay-атаку.

Во-вторых, ID-токен необходимо **валидировать отдельно от Access Token** и не использовать его для доступа к API, так как это нарушает принцип разделения ответственности и усложняет отзыв прав пользователя.
_______
1. ID-токен является JWT (JSON Web Token) со стандартными заголовками и полезной нагрузкой
2. Содержит обязательные claims: `iss` (издатель), `sub` (уникальный ID пользователя), `aud` (аудитория), `exp` (время истечения), `iat` (время выдачи)
3. Подписан криптографически стойким алгоритмом (RS256, ES256, PS256)
4. Имеет ограниченное время жизни
5. Должен поддерживать механизм `nonce` для защиты от инъекции токена
# Внедрение на клиенте

ID токен не предназначен для повторной отправки при каждом запросе к API сервера. После первичной валидации он обменивается на сессию или Access Token. 

Если архитектура соответствует современным практикам безопасности, клиент должен хранить только тот минимум данных, который необходим для отображения интерфейса и принятия решений на фронтенде. Чувствительные атрибуты (роли, права, финансовые метки) **должны запрашиваться у сервера по требованию** или проверяться через Access Token. Хранение должно быть устойчиво к XSS, автоматически очищаться при завершении сессии и синхронизироваться между вкладками браузера.
# Обработка на сервере

ID токен валидируется сервером приложения (Relying Party) сразу после получения от Identity Provider. Проверка включает верификацию JWT-подписи через JWKS Endpoint (`/.well-known/openid-configuration/jwks`), совпадение `iss` и `aud` с доверенными значениями, проверку `exp`/`iat` и валидацию `nonce`, если он использовался

При **stateless**-аутентификации (микросервисы, API Gateway) валидация выполняется через middleware с кэшированием открытых ключей (JWKS). В некоторых архитектурах валидные claims из ID токена проксируются во внутренние сервисы в заголовках `X-User-Id` или `X-JWT-Payload`, но предпочтительнее использовать отдельный Access Token для внутренних вызовов

Повторное использование истекшего ID токена недопустимо. Для обновления сессии следует использовать механизм **Refresh Token** или инициировать повторный редирект на IdP.
