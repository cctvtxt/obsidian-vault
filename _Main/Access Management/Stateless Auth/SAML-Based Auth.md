[[Stateless Auth]]

Протокол описан в OASIS-стандарте "**Assertions and Protocols for the OASIS Security Assertion Markup Language (SAML) V2.0**". SAML использует не один транспорт, а набор **binding’ов** — способов доставить SAML-сообщение между IdP и SP. Основные варианты: HTTP Redirect, HTTP POST, HTTP Artifact и SOAP; сами SAML assertions остаются SAML/XML-сообщениями, а транспорт задаётся отдельно. То есть SAML определяет структуру запроса, ответа и assertion, а HTTP Redirect / POST / Artifact / SOAP определяют транспортный механизм
# <u><font color="#245bdb">Юзкейсы</font></u>

**SAML** остается надежным выбором для корпоративных сред с устоявшейся инфраструктурой федерации, для интеграции с Active Directory, ADFS или корпоративным IdP. **SAML** излишен для новых проектов, мобильных приложений, публичных API и сценариев, где важна гибкость и скорость интеграции, в этих сценариях необходим OIDC.

В гибридных сценариях возможна комбинация: **SAML** для федеративного входа в корпоративный портал, который затем выдает **OIDC**-токены для доступа к внутренним API. Такой подход позволяет использовать сильные стороны обоих протоколов, разделяя ответственность между аутентификацией на границе и авторизацией внутри системы.
# <u><font color="#245bdb">Преимущества SAML</font></u>

**Обеспечивает единый вход (SSO) через федерацию идентификации**, позволяя пользователям аутентифицироваться один раз и получать доступ к нескольким приложениям **без повторного ввода credentials**, здесь пользователь работает с SAML Assertion, являющимся аналогом OIDC ID Token

**Повышает безопасность**: credentials не передаются напрямую SP, используются подписанные и зашифрованные XML-ассершены, снижая риски фишинга и утечек.

**Централизует управление идентификацией в IdP**, упрощая администрирование, снижая нагрузку (меньше сбросов паролей) и обеспечивая контроль в распределенных средах
# <u><font color="#245bdb">Недостатки SAML</font></u>

**Сложность внедрения**: требует детальной настройки XML-конфигураций, сертификатов и синхронизации метаданных между IdP и SP, что трудоемко для небольших команд.

**Ограниченная поддержка мобильных и API-ориентированных приложений**: SAML заточен под веб, плохо интегрируется с REST/GraphQL API без прокси вроде SAML2Bearer

**Тяжеловесность протокола**: XML-ассершены большие по размеру, их передача медленнее, чем в JWT/OAuth

**XML Signature Wrapping** — это атака на XML-подписи, при которой злоумышленник меняет структуру XML-документа так, чтобы проверка подписи проходила успешно, но приложение при этом обрабатывало уже подменённые данные. Идея в том, что подпись валидирует один элемент XML, а логика приложения читает другой, уже подставленный злоумышленником элемент. Пример в SAML: можно оставить настоящую подписанную assertion в документе, но переместить её в другое место, а на её месте подставить фальшивую assertion с нужным атрибутом, например с ролью администратора
# <u><font color="#245bdb">SAML Single Sign-On</font></u>

**SAML SSO** - это механизм, использующий SAML в качестве инструмента, позволяющего пользователям авторизовываться в нескольких приложениях с одним логином-паролем через систему управления доступом. **SAML SSO** - это использование протокола SAML для реализации концепции SSO, что предоставляется возможным благодаря благодаря обмену криптографически подписанными утверждениями (assertions) между доверенными сторонами.

SAML фокусируется на федеративной аутентификации/авторизации между Identity Provider (IdP) и Service Provider (SP) через браузер: обмен SAML Request/Response в XML, подпись/шифрование, SSO без прямого API-вызова. В отличие от OAuth (grant types, token endpoint), здесь нет access/refresh-токенов — только утверждения для SSO

Таким образом, **SAML SSO остаётся «золотым стандартом» для корпоративной федерации идентификации**, обеспечивая безопасный, масштабируемый и удобный механизм единого входа для сложных экосистем приложений.
# <u><font color="#245bdb">Сравнение с OIDC-Baced-аутентификацией</font></u>

**SAML** и **OIDC (OpenID Connect)** решают схожую на первый взгляд задачу — федеративную аутентификацию, но делают это на разных архитектурных уровнях и с разными целями. SAML — это зрелый **протокол, основанный на XML, который передает подписанные утверждения между Identity Provider и Service Provider**. Его дизайн исторически ориентирован на корпоративные веб-приложения и браузерные сценарии с HTTP-редиректами. OIDC, в свою очередь, является стандартизированным слоем идентификации поверх OAuth 2.0 и оперирует легковесными JSON-структурами (JWT). Он создан для современных распределенных систем, мобильных клиентов и SPA, где важны скорость парсинга, однозначная семантика claims и бесшовная интеграция с REST-архитектурой. 
## Сценарии, где SAML предпочтительнее

SAML остается стандартом де-факто в крупных корпоративных, государственных и образовательных средах, где уже развернуты решения вроде ADFS, Shibboleth, PingFederate или SAML-коннекторы Azure AD/Okta. Протокол идеален для интеграции «старых» и «новых» веб-приложений без модификации кода: достаточно обменяться метаданными и настроить доверительные отношения. SAML-ассершены поддерживают сложные XML-схемы расширений, что позволяет передавать кастомные атрибуты, подписывать и шифровать отдельные элементы независимо, а также реализовывать строгие политики соответствия (eIDAS, ГОСТ, отраслевые стандарты). 
## Сценарии, где OIDC предпочтительнее

OIDC становится выбором по умолчанию при разработке современных клиентских приложений, мобильных платформ, SPA и микросервисных экосистем. Благодаря наследованию от OAuth 2.0, он естественным образом работает с CORS, мобильными deep-линками, desktop-клиентами и headless-архитектурами, где браузерные редиректы неудобны или невозможны. OIDC стандартизирует набор базовых claims (`sub`, `name`, `email`, `picture` и др.), что упрощает онбординг разработчиков и снижает потребность в кастомных маппингах на стороне приложения. 

**SAML assertion** обладает высокой семантической выразительностью благодаря строгой XML-схеме и поддержке сложных структур. Можно передавать вложенные атрибуты, множественные значения, атрибуты с типами и пространствами имен. Это удобно для корпоративных сценариев, где требуется точная передача организационной иерархии, ролевой модели или атрибутов из LDAP/Active Directory. Однако эта мощность имеет цену: объемные сообщения увеличивают нагрузку на сеть и парсинг, а валидация XML требует защиты от специфичных уязвимостей (XXE, signature wrapping). Также жесткая схема усложняет эволюцию: добавление новых атрибутов требует согласования между сторонами.

**OIDC ID Token** более лаконичен и гибок. Плоская структура JSON упрощает парсинг и интеграцию в современные языки программирования. Расширение через кастомные клеймы позволяет быстро адаптировать токен под нужды приложения без изменения схемы. Но эта гибкость требует дисциплины: отсутствие строгой схемы означает, что клиент и провайдер должны явно договориться о семантике кастомных claims. Также размер JWT ограничен рекомендациями для передачи в заголовках HTTP, что делает его менее подходящим для передачи очень больших наборов атрибутов. Большинство серверов (Nginx, Apache, Heroku) ограничивают **общий размер всех заголовков** примерно **8 КБ**. Рекомендуется держать JWT **до 7 КБ**, чтобы оставить место для других заголовков (`Content-Type`, `User-Agent`, cookies)
# <font color="#245bdb"><u>Архитектура и разделение ответственности</u></font>

В SAML участвуют три стороны, и конфигурация каждой из них решает свою задачу. **Identity Provider (IdP)** — это источник истины об аутентификации: он хранит учётные записи, проверяет пароли, выдаёт подписанные утверждения. **Service Provider (SP)** — это приложение, которое хочет делегировать аутентификацию: оно доверяет IdP, но не видит паролей пользователей. **Клиент** — это приложение, которое инициирует поток и переносит сообщения между сторонами. Конфигурация строится по принципу «контракта»: каждая сторона публикует свои возможности в метаданных, а контрагент загружает их и настраивает локальную обработку.

Конфигурация SAML — это настройка доверия через метаданные и криптографию. **IdP** настраивается на выдачу утверждений определённым **SP**, с определёнными атрибутами и через определённые биндинги. **SP** настраивается на доверие конкретному **IdP**, валидацию его подписей и маппинг атрибутов. Клиент в браузерном сценарии почти не требует настройки, но толстые клиенты нуждаются в конфигурации эндпоинтов и сертификатов. 
# Конфигурация Identity Provider

Настройка Identity Provider сводится к регистрации доверенных приложений (Service Providers) и определению правил выдачи данных. Вы загружаете метаданные приложения, чтобы система «узнавала» его запросы, и настраиваете маппинг атрибутов — связываете внутренние данные каталога пользователей (например, почту или отдел) с полями, которые ожидает получить приложение. Критически важный этап — криптографическая привязка: обмен сертификатами для подписи и шифрования, что гарантирует подлинность сообщений и защиту данных при передаче. После этого провайдер автоматически формирует подписанные ответы с нужными атрибутами для каждого зарегистрированного приложения, соблюдая согласованные протоколы безопасности.
# Конфигурация Service Provider

Настройка Service Provider фокусируется на безопасном приеме и проверке входящих данных. Вы импортируете метаданные Identity Provider, чтобы приложение доверяло источнику аутентификации и знало, куда перенаправлять пользователей для входа. Затем настраивается конечная точка (ACS URL), которая принимает ответы от провайдера, и включается строгая проверка подписи входящих сообщений и сертификатов, чтобы исключить подделку. Финальный шаг — преобразование полученных стандартных атрибутов (например, email или роли) во внутренний формат вашей системы, чтобы пользователь получил корректный доступ к функционалу сразу после успешной аутентификации.
# Конфигурация клиента

Клиент в классическом браузерном сценарии SAML практически не требует явной конфигурации — его роль сводится к перенаправлению запросов и пересылке данных. Пользователь переходит на защищённый ресурс приложения (SP), приложение генерирует `AuthnRequest` и перенаправляет браузер на `entryPoint` IdP (через Redirect или POST). Браузер просто следует за редиректами, отправляет куки сессии, отображает форму логина.
# Обмен метаданными

Ключевой механизм согласования конфигурации — обмен метаданными в формате XML, определённом спецификацией SAML. Каждая сторона генерирует свой `EntityDescriptor`, который содержит: `entityID`, список поддерживаемых ролей (IdP/SP), конечные точки с привязками к биндингам, публичные сертификаты, контактную информацию и сроки действия документа. Обмен происходит одним из трёх способов: **ручная передача файла** (администратор скачивает `metadata.xml` и загружает в панель управления контрагента), **публикация по защищённому URL** (сторона размещает метаданные на `https://.../metadata`, контрагент настраивает периодический опрос с валидацией подписи), или **через агрегатор/федерацию** (например, InCommon, eduGAIN), где метаданные централизованно подписываются и распространяются между участниками.

После загрузки метаданных контрагента каждая сторона кэширует их локально и использует для: валидации входящих сообщений (проверка `issuer`, `destination`, подписи), построения исходящих запросов (выбор эндпоинта и биндинга), и ротации ключей (отслеживание `validUntil` и автоматическое переключение на новый сертификат). Важно: метаданные сами могут быть подписаны, и продвинутые реализации проверяют эту подпись перед импортом, чтобы исключить подмену.
# <u><font color="#245bdb">SAML Bindings</font></u>

**Биндинги в SAML** - это декларативный контракт в метаданных. Серверная реализация сводится к регистрации эндпоинтов, подключению крипто-материала и использованию проверенной библиотеки для сериализации/валидации. Динамический выбор происходит автоматически на основе метаданных, типа сообщения и политик безопасности. Физическое развёртывание требует точной синхронизации времени, валидации цепочек сертификатов и строгой обработки XML без ручной парсинг-логики

Они конфигурируются **на уровне каждого типа сообщения и каждой конечной точки** через механизм метаданных. В одном и том же `EntityDescriptor` можно и нужно объявлять **несколько биндингов одновременно**. Каждый из них получает:

- Уникальный `Location` (URL эндпоинта)
- Атрибут `Binding` с стандартным URN (например, `urn:oasis:names:tc:SAML:2.0:bindings:HTTP-POST`)
- `index` (целое число для упорядочивания)
- `isDefault="true"` (рекомендуемый выбор, если нет явных политик)

Пример фрагмента метаданных SP:
```xml
<md:AssertionConsumerService Binding="urn:oasis:names:tc:SAML:2.0:bindings:HTTP-POST"
Location="https://sp.example.com/saml/acs" index="0" isDefault="true"/>
<md:AssertionConsumerService Binding="urn:oasis:names:tc:SAML:2.0:bindings:HTTP-Artifact"
Location="https://sp.example.com/saml/artifact" index="1"/>
<md:AssertionConsumerService Binding="urn:oasis:names:tc:SAML:2.0:bindings:HTTP-Redirect"
Location="https://sp.example.com/saml/acs-redirect" index="2"/>
```

IdP при построении ответа читает эти записи, фильтрует по биндингам, которые он умеет генерировать, и выбирает целевой эндпоинт. Если совпадений несколько → берётся `isDefault="true"` или первый. Таким образом, **поддержка нескольких биндингов — это штатный механизм отказоустойчивости и миграции**

**HTTP Redirect Binding**

Использует обычное HTTP-перенаправление (GET-запрос).  
SAML-сообщение помещается прямо в строку запроса, поэтому его сначала сжимают и кодируют, чтобы оно поместилось:  
**XML → DEFLATE → Base64 → URL encoding.** Из-за лимита длины URL (примерно 2–8 КБ) этот способ годится только для коротких сообщений — обычно `AuthnRequest` и `LogoutRequest`.

**HTTP POST Binding**

Передача сообщения происходит через HTTP POST — данные находятся в теле запроса (HTML-форма), а не в строке URL.  
Сериализация проще: **XML → Base64**. Результат записывается в скрытое поле формы (`SAMLResponse` или `SAMLRequest`). Ограничений по размеру тела запроса нет, поэтому этот метод — стандарт для передачи `SAMLResponse`. Он подходит, когда нужно передавать большие данные, например группы, атрибуты и условия доступа.

**HTTP Artifact Binding**

Здесь браузер не получает само сообщение SAML, а только короткий «артефакт» — маленький токен (около 54 байт после Base64). Они действуют как ссылка: SP по нему запрашивает у IdP полное сообщение по защищённому каналу (через SOAP-запрос `ArtifactResolve`). IdP отвечает `ArtifactResponse` со всем SAML-документом. Такой подход позволяет избежать утечки чувствительных данных в браузер и обходит лимиты размера URL. Вся полезная информация передаётся между серверами напрямую.

**SOAP Binding**

Используется для обмена сообщениями напрямую между серверами — без участия браузера. SAML-запрос и ответ помещаются в **SOAP-конверт** (`<soap:Envelope>`), где всё оформлено как обычное XML-сообщение. Этот метод применяют там, где браузерная модель неудобна — при динамическом запросе атрибутов (`AttributeQuery`, `AuthnQuery`) или в профиле **ECP** для мобильных и десктопных клиентов. SOAP позволяет надёжно обмениваться данными между системами, где требуется строгий контроль безопасности, поддержка транзакций, передача бинарных вложений и верификация сертификатов. Минус — высокая сложность реализации.
