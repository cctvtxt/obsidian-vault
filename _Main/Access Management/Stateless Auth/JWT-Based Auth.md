[[Stateless Auth]]

**JSON Web Token (JWT)** — стандарт RFC 7519 для передачи данных между сторонами в виде JSON‑объектов. Токены содержат утверждения о пользователе и подписываются секретным ключом. В отличие от Session Authentication, где каждый запрос требует проверки в БД, JWT содержит всю информацию внутри себя и может быть верифицирован локально на любом сервере без дополнительных запросов к БД. Такой токен по умолчанию не шифрует данные, но гарантирует их целостность, поэтому удобно хранить в нем идентификатор пользователя, срок жизни и роли, и использовать в stateless‑аутентификации: каждый запрос несет `Authorization: Bearer <jwt>`, серверу не нужно поддерживать сессию, но отзыв токенов требует дополнительных механизмов (малый TTL, blacklist), также **чувствительные данные внутрь JWT класть нельзя**. Чувствительные данные — это информация, утечка которой может нанести ущерб безопасности, конфиденциальности или привести к компрометации пользователя: пароли, хэши паролей или ключи API, персональные данные, профили пользователей.
## Принцип работы

Взаимодействие между клиентом и сервером с использованием JWT (JSON Web Token) происходит по следующему алгоритму:

1. **Клиент выполняет вход,** используя учетные данные и отправляя запрос на сервер.
2. **Сервер проверяет** эти учетные данные. Если они действительны, сервер генерирует JWT и отправляет его обратно клиенту. При генерации JWT-токена сервер ставит подпись секретным ключом, который хранится только в веб-приложении.
3. **Клиент сохраняет** JWT, обычно в локальном хранилище, и включает его в заголовок каждого последующего HTTP-запроса.
4. **Сервер,** получая эти запросы, проверяет JWT. Если он действителен, клиент аутентифицирован и авторизован.
## Юзкейсы

JWT аутентификация лучше подходит для приложений, которые дают приоритет масштабируемости, эффективности и распределенности: **REST API, одностраничные приложения (SPA), микросервисы**. JWT аутентификация полезна для stateless взаимодействий между клиентами и серверами

1. JWT идеально подходят для Single Sign-On, позволяя пользователям аутентифицироваться один раз и бесшовно получать доступ к нескольким сервисам 
2. Мобильные приложения, RESTful API и веб-сервисы обычно используют JWT для аутентификации, так как они легковесны, портативны и устраняют необходимость в управлении сессиями на стороне сервера
3. В микросервисных средах JWT позволяют каждой службе самостоятельно проверять токен без обращения к центральному хранилищу сессий, обеспечивая масштабируемость и эффективность
4. JWT превосходят в сценариях, включающих несколько доменов или поддоменов (например, `api.example.com`, `dashboard.example.com` и `docs.example.com`). В отличие от cookies, JWT позволяют аутентификацию между доменами без дополнительных зависимостей
## Преимущества

1. Фундаментальное преимущество JWT (JSON Web Token) для аутентификации — **stateless-архитектура** (отсутствие необходимости хранить состояние на сервере). Сервер **не обращается к базе данных или кэшу** для проверки пользователя — он просто валидирует подпись токена и декодирует его содержимое
2. **Эффективность в расширенных системах** — разные сервисы могут проверять токен без единого централизованного хранилища сессий
3. **Масштабируемость** — сервисы работают независимо, можно добавлять узлы без синхронизации сессий
4. **Отказоустойчивость** — сбой сервиса аутентификации не ломает работу других сервисов (они уже имеют все данные в токене)
5. **Гибкость** — сервисы могут быть написаны на разных технологиях, но одинаково работать с JWT
## Недостатки

- сложность управления жизненным циклом токенов;
- уязвимость системы при компрометации секретного ключа.
## <u><font color="#c00000">Решение проблемы перехвата JWT</font></u>

1. Применение TLS-шифрования, для для TLS расшифровка практически невозможна: AES‑128/256 + ECDHE дают уровень стойкости порядка 21282^128 и выше, что физически не подбирается. Атака для расшифровки может быть выполнена только на устройства
2. Применение короткого времени жизни (~15 минут), это нужно, чтобы минимизировать последствия утечки, так как **JWT-токен нельзя сразу отозвать**
3. Ротация refresh‑токенов помогает защитить JWT, потому что делает его одноразовым, а не вечным ключом к сессии тем, что при обновлении access-токена обновляется и refresh-токен
## <u><font color="#c00000">Решение проблемы досрочного отзыва JWT</font></u>

1. Вместо хранения всех токенов, **что убивает все преимущества stateless-аутентификации**, создается black_list в in-memoiry DB. При проверке токена мы сначала проверяем, не добавлен ли он в список, затем валидируем его. Если токен в черном списке, возвращаем 401. Чтобы токены не накапливались в черном списке, прописывается достаточный TTL
## <u><font color="#c00000">Решение проблемы realtime-обновления JWT</font></u>

1. После подписания **payload существующего JWT не может быть обновлен** и прошлый токен будет жить, пока подпись не истечет. Реальное обновление `claims` всегда требует выпуска нового JWT, что решается добавлением старого токена в blacklist
## <u><font color="#c00000">Решение проблемы работы с множеством устройств</font></u>

1. Простым решением является выдача общей пары токенов на аккаунт. Тогда при выходе будут разлогинены все девайсы, **и это опасно при утечке**
2. Другим решением является выпуск отдельного токена на каждое устройство/сессию, тогда реальным будет создавать приложения с разными правами доступа к ресурсам сервера. 

Невозможно проверить все выданные JWT до их истечения для реализации аннулирования пользователя на всех устройствах. Хотя теоретически возможно отозвать ключ подписи, чтобы отменить JWT, это также аннулирует все JWT, использующие этот ключ, а процесс обработки кэшированных ключей сделает этот подход непрактичным для простых операций аннулирования пользователей.
