[[Stateless Auth]]

RFC 6749 - протокол реализации системы авторизации, также стандартизовано бесконечность OAuth-flow для интеграции различных систем посредством данной технологии

Oauth 2.0 решает проблему с тем, что ввод длинной последовательности случайных символов в поле формы делает пользовательский опыт плохим, ровно как и использование API-ключей для доступа к системам

**Автоматизация обмена ключами** - одна из основных проблем, решаемых с помощью OAuth. Она предоставляет клиенту стандартный способ получения ключа с сервера путем выполнения пользователем простого набора действий. Все, что нужно сделать пользователю, это быть авторизованным на внешнем сервисе, предоставляющему доступ. 

**OAuth 1.0** — это первоначальная спецификация протокола делегированного доступа (опубликована в 2007 году, стандартизирована в RFC 5849), которая обеспечивала безопасность запросов за счёт сложной криптографической подписи каждого HTTP-запроса, требующей генерации уникальных `nonce`, как Digest Auth, временных меток и строгой сортировки параметров. Протокол устарел и был заменён на OAuth 2.0 по нескольким ключевым причинам: он был чрезвычайно сложен в реализации и отладке (малейшая ошибка в подписи ломала запрос), не поддерживал современные сценарии использования (мобильные приложения, нативные клиенты, сервисы без браузера), а также не масштабировался для работы с микросервисами.

**OAuth 2.0** - это протокол **авторизации**, позволяющий сторонним приложениям (клиентам) получать ограниченный доступ к защищённым ресурсам пользователя на сервере без передачи им логина и пароля. Вместо этого используются временные токены доступа, выдаваемые сервером авторизации. В авторизации по протоколу участвуют: **пользователь**, **клиент**, **сервер** и **OAuth-сервис**.

Пользователь проходит логин у авторизационного сервера, тот выдает приложению access token, дальше API доверяет этому токену, не видя пароля пользователя, что позволяет реализовать вход через сторонние провайдеры, тонкую грануляцию прав (scopes) и сценарии как user‑to‑service, так и service‑to‑service, ценой усложненной архитектуры и необходимости во введении отдельного auth‑сервера.
## Federated Authentication

**Федеративная аутентификация (Federated Authentication)** — это архитектурный паттерн, при котором аутентификация пользователя происходит **не в вашем приложении**, а во внешней доверенной системе (провайдере идентичности, **Identity Provider — IdP**), после чего ваш сервис (**Service Provider — SP** или **Relying Party — RP**) получает подтверждение личности и доверяет этому подтверждению.

Подход представлен в OIDC, SAML 2.0, WS-Federation
# Достоинства

1. нивелирует необходимость в передаче учетных данных
2. стандартизирует взаимодействие между клиентами, провайдерами идентификации и ресурс-серверами
3. обеспечивает гибкое управление правами через `scopes`
4. поддержка множество проработанных Flow
5. простота масштабирования системы авторизации
# Недостатки

1. OAuth 2.0 — это **протокол авторизации, а не аутентификации**, поэтому его часто misuse'ят для «входа через соцсети» без наложения OIDC
2. зависимость от внешних IdP (изменение политик, downtime, vendor lock-in)
3. дополнительный сетевой overhead и задержки на валидацию/обмен токенов
4. высокий риск ошибок внедрения и сложный жизненный цикл токенов (обновление, отзыв, интроспекция, обработка ошибок `invalid_grant`)
# OAuth 2.0 Authorization Code Flow

1. Разработчик регистрируется в гугле и создает нового клиента, прописывает адреса своего сайта: гугл дает **client ID** и **client secret**. При этом разработчик на своем сервере генерирует функцию для выдачи гиперссылки для редиректа пользователей, прописывает, доступ к чему именно должен давать нам сервис: например, доступ к диску, а также генерирует на сессию входа CSRF-токен `state` для защиты 
```javascript
// .env file
// GOOGLE_CLIENT_ID=your-google-client-id.googleusercontent.com
// GOOGLE_CLIENT_SECRET=your-google-client-secret
// GOOGLE_REDIRECT_URI=http://localhost:3000/auth/google/callback

const GOOGLE_OAUTH_URL = 'https://accounts.google.com/o/oauth2/v2/auth';

function generateGoogleAuthUrl(state = null) {
  const params = new URLSearchParams({
    client_id: process.env.GOOGLE_CLIENT_ID,
    redirect_uri: process.env.GOOGLE_REDIRECT_URI,
    scope: 'drive', // Запрашиваемые scopes
    response_type: 'code',         // Authorization Code Flow
    access_type: 'offline',        // Для получения refresh token
    prompt: 'consent',             // Показывать экран согласия
    state: crypto.randomBytes(32).toString('hex'); // CSRF‑токен
    // CSRF‑токен должен затем храниться в БД для проверки
  });

  // Добавляем state для CSRF защиты (опционально)
  if (state) params.append('state', state);
  return `${GOOGLE_OAUTH_URL}?${params.toString()}`;
}
```
2. Клиент предоставляет по client.com/login пользователю кнопку для аутентификации по гуглу, пользователь кликает.
3. На кнопке стоит редирект на страницу гугла accounts.google.com/v3/signin, (если **prompt: 'consent'**) пользователь видит страницу с выбором гугл-аккаунта, простановкой галочек для выдачи доступа к ресурсам, вместе с запросом отправляются куки сервиса: `SID`, `HSID`, `SSID`, `APISID`, гугл считывает эти куки, показывает список ваших аккаунтов для выбора
4. Сервис гугла генерирует код пользователя, редиректит нас на сайт, с которого мы пришли, но добавляет код в виде client.com/login/google?code=23846278364t276346239867
5. **Клиент** делает `params.get('code')` и `params.get('state')`, затем шлет оба на **бек**
```javascript
fetch('/api/auth/google/callback', {
  method: 'POST',
  body: JSON.stringify({ code, state })
});
```
6. **На сервере** прописан эндпоинт для возвращения кодов вида `POST /api/auth/google/callback`. Сервер проверяет, чтобы `state` был корректным и использовался впервые, забирает `code` и шлет POST в `oauth2.googleapis.com/token`
```javascript
url = 'https://oauth2.googleapis.com/token'
// client_id и client_secret получены на первом этапе
body = {
  'client_id': process.env.GOOGLE_CLIENT_ID,
  'client_secret': process.env.GOOGLE_CLIENT_SECRET,
  'grant_type': 'authorization_code',
  'redirect_uri': process.env.GOOGLE_REDIRECT_URI,
  'code': code
}
```
7. Сервис гугла возвращает **access_token** (с ним можно ходить в гугл и взаимодействовать с ресурсами, короткоживущий токен), **refresh_token** (живет долго, хранится на бэкенде), **scope** (наши возможности)
8. Сервер производит свою авторизацию пользователя, ищет пользователя в своей БД. Если нет → создаёт. Проверяет статус (активен, забанен, требуется доп. верификация) и прочее, выдает собственные токены доступа, отправляет на клиент подтверждение
9. Далее с сервером, предоставляющим доступ к своим ресурсам, взаимодействуют только сервер, отправляет в гугл запросы вида:
```javascript
url='https://googleapis.com/drive/files'
headers={
	'Authorization: Bearer ${access_token}'
}
```
# Proof Key for Code Exchange

PKCE (RFC 7636) — расширение протокола OAuth 2.0, которое добавляет дополнительный уровень безопасности в процесс авторизации, где, несмотря на наличие `client_secret` на сервере и `state` для CSRF-защиты, в исходном алгоритме остаётся критическая уязвимость перехвата **authorization‑code interception attack**. Проблема стандартного флоу в том, что `client_secret` является **глобальным для сервера**, а `code_verifier` генерируется **на каждую сессию**, что повышает безопасность

На этапе обмена токена кода на токен доступа, код `state` защищает только от CSRF-атак, то есть от подделки запроса на начало авторизации, но **не предотвращает кражу уже выпущенного кода**. А `client_secret`, хотя и хранится на бэкенде, не является криптостойким аутентификатором сессии: при его утечке или компрометации сервера злоумышленник получает полную возможность обменять перехваченный код на токены доступа. Без дополнительной привязки код остаётся **самодостаточным ключом**. PKCE решает эту проблему, вводя криптографическую связку между запросом авторизации и запросом на получение токена. С ней потеря и кода авторизации и гнлобалвьного идентификатора сервера `client_secret` не является проблемой

На старте сервер генерирует **случайный секрет** (`code_verifier`) и отправляет в Google только **его хеш** (`code_challenge`). Сам секрет остаётся исключительно на бэкенде. При возврате пользователя сервер предъявляет сессионный `code_verifier` вместе с кодом. Google вычисляет хеш от полученного секрета и сверяет его с сохранённым ранее. Если значения не совпадают, обмен кода отклоняется. Даже при полном перехвате кода атакующий не сможет получить токены, так как не знает `code_verifier`

1. На первом шаге предыдущего алгоритма создаем `codeChallenge` и `codeVerifier`, `codeChallenge` прикрепляем к параметрам пользовательского запроса
```javascript
function generateCodeChallenge(verifier) {
  return crypto
    .createHash('sha256')
    .update(verifier)
    .digest('base64url');
}

const codeVerifier = generateCodeVerifier();
const codeChallenge = generateCodeChallenge(codeVerifier);

params.append('codeChallenge', codeChallenge);
params.append('code_challenge_method', 'S256');
```
2. Этапы 2-5 остаются без изменения
3. На 6 этапе в процессе обмена кода на токен 
```javascript
body: new URLSearchParams({
      client_id: process.env.GOOGLE_CLIENT_ID,
      client_secret: process.env.GOOGLE_CLIENT_SECRET,
      grant_type: 'authorization_code',
      redirect_uri: process.env.GOOGLE_REDIRECT_URI,
      code: code,
      // PKCE-параметр — обязателен, если использовался code_challenge

      code_verifier: codeVerifier
    }
```
4. Дополнительным этапом можно указать то, что code_verifier должен использоваться и обновляться при каждом повторном запросе токена доступа гугл
5. Остальные этапы остаются без изменений
# OAuth 2.1

Сегодня PKCE стал отраслевым стандартом. Спецификация OAuth 2.1 делает PKCE обязательным для всех типов клиентов, отходя от практики, где `client_secret` считался достаточной защитой. Крупные провайдеры, включая Google, уже требуют или настоятельно рекомендуют внедрение PKCE при ревью приложений и выдаче прав доступа. Добавление этого шага превращает ваш флоу из функционального в безопасный, закрывая вектор перехвата сессий, инъекций кода и последствия компрометации учётных данных OAuth-клиента.
