[[Stateless Auth]]


В таком подходе вся информация о пользователе кодируется в токене, который клиент отправляет в заголовках запросов; сервер только верифицирует подпись



| Критерий               | Token                                                             |
| ---------------------- | ----------------------------------------------------------------- |
| **Хранение состояния** | Клиент хранит токен; сервер не хранит состояние                   |
| **Масштабируемость**   | Выше (сервер не зависит от хранилища сессий)                      |
| **Безопасность**       | Уязвимы к XSS (если токен в localStorage); требуют защиты подписи |
| **Срок жизни**         | Короткие (например, 15–60 мин), требуют refresh token             |
| **Отзыв доступа**      | Требует чёрного списка токенов или проверки на сервере            |
| **Объём данных**       | Токен ограничен по размеру (JWT ~4 KB)                            |
|                        |                                                                   |

1. Пользователь вводит логин/пароль.
2. Сервер проверяет данные, генерирует _токен_ (с payload: ID пользователя, роли, срок действия).
3. Токен подписывается секретным ключом и отправляется клиенту (в cookie, localStorage или заголовке `Authorization`).
4. Клиент включает токен в каждый запрос (например, `Authorization: Bearer <token>`).
5. Сервер проверяет подпись токена и извлекает данные без обращения к БД.

**Плюсы (на примере JWT):**

- Stateless: легко масштабировать.
- Подходит для микросервисов (токен читается всеми сервисами).
- Явный срок действия (expiry).

**Минусы (на примере JWT):**

- Сложность отзыва токена до истечения срока.
- Риск XSS при хранении в localStorage.
- Ограниченный размер payload.
- Необходимость защиты секретного ключа.
