[[Stateless Auth]]

TLS-протокол также позволяет проводить аутентификацию клиента. При двусторонней аутентификации сервер посылает сообщение **CertificateRequest**, чтобы получить сертификат пользователя. Причем, для большей надежности, у пользователя закрытый ключ, который соответствует сертификату, может храниться на аппаратном токене. Такой подход часто применяется на практике в решениях, где особенно строгие требования по безопасности. Сертификаты клиентов и серверов в таких решениях зачастую выпускаются внутренним удостоверяющим центром.

В таком случае не надо делать страницу авторизации, пользователю не надо придумывать отдельный пароль, мы имеем возможность быстрого импорта пользователей (просто файлы с сертификатами) и знаем имя пользователя, так как в mTLS имя пользователя извлекается сервером из **Distinguished Name** клиентского сертификата, который клиент автоматически отправляет в ответ на сообщение **CertificateRequest** сервера. DN заранее известен, поскольку сертификаты выдаются внутренним **CA** с предопределёнными уникальными именами для каждого пользователя — их можно импортировать на сервер как файлы .pem/.crt для проверки и маппинга на учётные записи.


Но, к сожалению, есть **недостатки**: клиентские сертификаты более требовательны к качеству криптографии (то Microsoft с апдейтами начудит и всё сломает, то Chrome решит истерить), для Firefox приходится вручную импортировать сертификат, с мобильными браузерами множество проблем (кто-то не поддерживает, импорт сертификатов приводит к некоторому геморрою). 






Если клиент не просто пользуется защищенным каналом (где только сервер предоставляет свой сертификат), а сам отправляет собственный X.509‑сертификат в TLS‑рукопожатии, а сервер его криптографически проверяет (цепочка доверия + подпись, владение приватным ключом), это и называется mutual TLS (**mTLS**) или аутентификация клиента через TLS.

При аутентификации клиентов с mutual TLS:

- Сервер предъявляет свой сертификат.
- Клиент предъявляет свой сертификат.
- Обе стороны проверяют сертификаты друг друга.

При TLS-аутентификации **часто не требуются дополнительные данные** — идентичность проверяется исключительно сертификатом и приватным ключом в handshake. 




TLS уникален **PKI-доверием** и интеграцией с транспортным шифрованием — stateless, не требует сессий или баз данных для авторизации
