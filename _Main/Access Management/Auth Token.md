[[Token-Based Auth]]

В сфере кибербезопасности и цифровой идентификации токены играют ключевую роль в обеспечении безопасной связи и аутентификации пользователей. Среди множества доступных токенов чаще всего обсуждаются **opaque (непрозрачные)** и **структурированные** токены. Такие токены выдаются после аутентификации пользователя. Они служат подтверждением личности и включаются в запросы.

Структурированные токены, в отличие от непрозрачных, содержат внутри себя данные. Данные структурированы в определенном формате, который можно расшифровать, чтобы получить содержащуюся в них информацию.

В контексте аутентификации токен обозначается как Auth Token, такое понятие используется в **RFC 6749, 6750, 7519** и других стандартах. **Аутентификационный токен** представляет собой подписанный криптографическим ключом набор символов, подтверждающий собой то, что пользователь подтвердил свою личность 
_________
В зависимости от архитектуры и логики веб-приложение может одновременно использовать несколько типов токенов:

- Access Token — токен доступа к информации, обычно имеет время жизни несколько минут.
- Refresh Token — токен обновления, по которому можно только получить новую пару токенов (Access и Refresh), срок жизни измеряется днями.

Совместное использование этих токенов позволяет реализовать безопасную и удобную систему аутентификации, при которой пользователю не требуется проходить повторную авторизацию после истечения срока действия Access Token.


















При выборе между структурированным токеном и opaque нужно исходить не из того, какой из них больше подходит для **архитектуры приложения, модели безопасности и операционных потребностей**:

**Нужна ли вам функция отзыва?**

- **Да** → Используйте **непрозрачные токены**. Сервер может мгновенно отозвать их, что делает их идеальным решением для длительных сессий, токенов обновления или сред с высоким уровнем безопасности.
- **Нет (короткие токены вполне подходят)** → JWT хорошо себя зарекомендовали, особенно в сочетании с коротким сроком действия для минимизации рисков.

**Вас беспокоит видимость токенов?**

- **Да** → Используйте **непрозрачные токены**. Они не раскрывают никаких пользовательских данных или утверждений, что делает их более безопасными по умолчанию, особенно в средах с нулевым доверием или регулируемых средах.
- **Нет** → JWT вполне подходят, если вас не смущают раскрываемые метаданные (например, роли, имена пользователей) и вы не хотите включать в токен конфиденциальные данные.

**Работаете ли вы в среде микросервисов?**

- **Да** → Используйте **JWT**. Благодаря отсутствию отслеживания состояния и локальной проверке они идеально подходят для распределенных систем, где скорость и масштабируемость имеют решающее значение.
- **Нет (централизованные приложения или монолитные системы)** → Непрозрачные токены зачастую проще в управлении и обеспечивают лучший контроль для традиционных приложений.

**Ожидаете ли вы частых изменений в системе разрешений?**

- **Да** → Используйте **непрозрачные токены**. Поскольку они проверяются на стороне сервера, любые изменения в ролях пользователей или политиках доступа вступают в силу немедленно.
- **Нет** → JWT по-прежнему могут использоваться, но следует учитывать, что срок их действия может быть слишком долгим, из-за чего могут сохраняться устаревшие разрешения.

### **Гибридные подходы (лучшее из двух миров)**

На практике многие производственные системы используют **комбинацию** JWT и непрозрачных токенов, чтобы обеспечить баланс между скоростью, безопасностью и возможностью отзыва:

**JWT для доступа + непрозрачный токен для обновления**

- **Токены доступа**: используйте JWT с коротким сроком действия для быстрого доступа к API без сохранения состояния.
- **Токены обновления**: используйте непрозрачные токены для безопасного продления сеанса с возможностью отзыва.

Эта модель широко используется в протоколах OAuth 2.0 и поддерживается большинством современных провайдеров идентификации. Она позволяет:

- **Скорость** и **масштабируемость** JWT для большинства запросов
- **Контроль** и **возможность отзыва** непрозрачных токенов при обновлении сессий

**JWT для внешних/публичных API + Opaque для внутренних сервисов**

- JWT легко проверять в разных командах или сторонних клиентах.
- Непрозрачные токены обеспечивают более строгий контроль в рамках вашей доверенной инфраструктуры.

# Обработка и хранение токенов на клиенте



Веб-хранилище (localStorage или sessionStorage)

**+** Браузер не будет автоматически включать что-либо из веб-хранилища в HTTP-запросы, что делает его неуязвимым для CSRF.  
  
**+** Доступ к данным может быть получен только с помощью JavaScript, работающего в том же домене, который создал эти данные.

**—** Недоступен для JavaScript, работающего в поддомене, который создал эти данные (значение, записанное example.com, не может быть прочитано как sub.example.com).  
  
**—** Уязвим для XSS.

Заголовок cookie

**+** Неуязвим для XSS.  
  
**+** Браузер автоматически включает токен в любой запрос, который соответствует спецификации заголовка cookie (домен, путь и срок действия).  
  
**+** Файл cookie может быть создан в домене верхнего уровня и использоваться в запросах, выполняемых поддоменами.

**—** Уязвим для CSRF.  
  
**—** Необходимо знать и всегда учитывать возможное использование файлов cookie в поддоменах.  
  
**—** Выбор запросов, которые должны включать файл cookie, выполним, но сложнее.  
  
**—** Можно (по-прежнему) столкнуться с некоторыми проблемами из-за небольших различий в том, как браузеры работают с файлами cookie.  
  
**—** Если не быть осторожным, то можно реализовать стратегию защиты от CSRF, которая уязвима для XSS.









# Передача токенов на сервер

Существует три популярных способа передачи JWT между клиентом и сервером. 

1. Через HTTP-заголовок Authorization со схемой аутентификации Bearer — является наиболее безопасным и рекомендуемым способом передачи токена.
- Удобно использовать с fetch, axios, curl и другими HTTP-клиентами.
- Можно настроить сервер для автоматической проверки заголовка.

**Безопасность:**

- Защита от CSRF, если **токен не хранится в cookie.**
- Нужно защищать от **XSS,** если токен в localStorage/sessionStorage.

2. В Cookie (обычно HttpOnly) — хранение и передача через cookie позволяет автоматически прикреплять токен к каждому запросу.

3. В Cookie (обычно HttpOnly - XSS-защита) — хранение и передача через cookie позволяет автоматически прикреплять токен к каждому запросу.
Уязвим для **CSRF,** если не используется SameSite=strict или CSRF-токен.
- Меньший контроль клиента над отправкой





ерез параметры URL или POST-данные — используется **редко** и **не рекомендуется.**

**Примеры:**

- https://example.com/callback?token=eyJhbGci…
- В теле формы или JSON-объекта в POST-запросе.

**Недостатки:**

- Может попасть в логи сервера или историю браузера.
- Повышенный риск утечки токена
