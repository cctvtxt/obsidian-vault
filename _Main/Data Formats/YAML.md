YAML (YAML Ain’t Markup Language) разработан как более гибкий и читаемый формат данных, который **включает в себя всю функциональность JSON, но предоставляет дополнительные возможности**. Это означает, что:

1. **Любой JSON-файл можно парсить как YAML** — синтаксис JSON является подмножеством синтаксиса YAML.
2. **YAML поддерживает больше возможностей, чем JSON:**

- **Комментарии** (с символом `#`), отсутствующие в JSON.
- **Отсутствие кавычек** для ключей и строк (в отличие от JSON, где всё заключено в двойные кавычки).
- **Система отступов** (как в Python) для обозначения структуры вместо фигурных скобок `{}` и квадратных скобок `[]`.
- **Поддержка расширенных типов данных** (например, `!!int`, `!!bool`).
- **Якоря и шаблоны** для повторного использования фрагментов конфигурации.
- **Поддержка многострочных строк** и других удобных синтаксических конструкций.

---

Пример Создан Кларком Эвансом, Инго Штольте и Ореном Бен-Хуримом в 2001 году как человеческий формат сериализации данных. Цель — простота чтения и записи, совместимость с XML/JSON, с акцентом на indentation-based структуру. Версии эволюционировали: 1.0 в 2004 (YAML 1.1), 1.2 в 2009 (текущая стабильная). YAML использует отступы для иерархии, поддерживает базовые типы: строки, числа, булевы значения, даты, массивы, словари и теги для типов. Комментарии начинаются с #, строки многострочные с | или >.

```yaml
dependencies:
	serde:
		version: "1.0"
		features: ["derive"]
	toml: "0.8"
build:
	version: "0.1.0"
	date: 2026-03-04T20:39:00Z
	authors: ["dev@example.com"]
```

Формат позиционируется как "человеческий" (Yet Another Markup Language), альтернатива XML (проще, без тегов), JSON (добавляет комментарии, даты), TOML (больше гибкости для сложных структур). Часто сравнивают с `Python dict`.

## Преимущества

Высокая читаемость благодаря отступам, как в Python-коде.  
Поддержка сложной иерархии, слияния файлов (merge keys) и якоря/алиасы для DRY.  
Гибкость: многострочные строки, теги типов, нумерация массивов.  
Широкая поддержка языков (Python PyYAML, Rust serde_yaml, Go и др.).  
Идеален для конфигов с динамическими данными.

## Недостатки

Неоднозначности парсинга (off-side rule, разные интерпретации отступов).  
Уязвимости (YAML-бомбы, экспоненциальное раздувание памяти).  
Меньшая строгая типизация, приводит к ошибкам в датах/числах.  
Сложность для машинного чтения без схем в больших файлах.  
Кривая обучения для продвинутых фич (теги, якоря).

## Область применения

YAML доминирует в конфигурациях: Docker Compose, Kubernetes manifests, Ansible, GitHub Actions, Helm charts. Идеален для DevOps, где важна читаемость и иерархия без лишнего boilerplate.
