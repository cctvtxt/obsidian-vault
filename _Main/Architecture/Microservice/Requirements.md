## 1. Границы микросервисов (Service Boundaries)

* **Доменные области** (Domain‑Driven Design): каждый сервис должен соответствовать ограниченной доменной области (Bounded Context).
* **Принцип единственной ответственности** (SRP): один сервис — одна бизнес‑функция.
* **Размер сервиса**: избегайте «наносервисов», но и не делайте слишком крупные сервисы.
* **Связность данных**: данные внутри сервиса должны быть логически согласованы.

## 2. Контракты взаимодействия

* **Форматы данных**: JSON, XML, Protocol Buffers, Avro и т. д.
* **Протоколы коммуникации**:
    * синхронные (REST/HTTP, gRPC);
    * асинхронные (AMQP, Kafka, RabbitMQ).
* **Спецификация API**: OpenAPI/Swagger для REST, Protobuf для gRPC.
* **Версионирование API**: стратегии версионирования (URL, заголовки, семантическое версионирование).
* **Контракты между сервисами**: чёткое определение входных/выходных данных, кодов ошибок.
* **Обратная совместимость**: правила внесения изменений без нарушения работы клиентов.

## 3. Коммуникационные паттерны

* **Синхронная коммуникация** (REST, gRPC): подходит для операций с немедленным ответом.
* **Асинхронная коммуникация** (очереди сообщений, event‑driven): для фоновых задач и слабой связности.
* **API Gateway**: единая точка входа, маршрутизация, аутентификация.
* **Service Mesh** (Istio, Linkerd): управление трафиком, наблюдаемость, безопасность на уровне сети.

## 4. Управление данными

* **База данных на сервис** (Database per Service): изоляция данных, независимость развёртывания.
* **Стратегии согласованности**:
    * ACID (для транзакционных систем);
    * BASE (для высокой доступности).
* **Паттерны управления данными**:
    * Event Sourcing;
    * CQRS;
    * Saga (для распределённых транзакций).

* **Кэширование**: локальное и распределённое (Redis, Memcached).
* **Синхронизация данных**: механизмы синхронизации между сервисами.

## 5. Производительность и масштабируемость

  

* **Горизонтальное масштабирование**: возможность добавления экземпляров сервиса.

* **Балансировка нагрузки**: выбор алгоритма (round‑robin, least connections).

* **Ограничение частоты запросов** (Rate Limiting): защита от перегрузки.

* **Таймауты и повторные попытки** (Retries): стратегии обработки временных сбоев.

* **Circuit Breaker**: предотвращение каскадных сбоев.

* **Буферизация и пакетная обработка**: оптимизация нагрузки на сервисы.

  

## 6. Отказоустойчивость и надёжность

  

* **Health Checks**: endpoints для мониторинга состояния сервиса.

* **Graceful Degradation**: работа в условиях частичной неисправности.

* **Автоматическое восстановление**: перезапуск упавших экземпляров.

* **Мониторинг и алертинг**: метрики (CPU, память, задержки), логи, трейсинг.

  

## 7. Безопасность

  

* **Аутентификация**: JWT, OAuth2, OpenID Connect.

* **Авторизация**: RBAC, ABAC.

* **Шифрование**: TLS для трафика, шифрование данных в покое.

* **Защита API**: валидация входных данных, ограничение payload.

* **Secrets Management**: безопасное хранение паролей, ключей API.

  

## 8. Инфраструктура и развёртывание

  

* **Контейнеризация**: Docker для упаковки сервисов.

* **Оркестрация**: Kubernetes, Docker Swarm.

* **CI/CD**: автоматизация сборки, тестирования, развёртывания.

* **Инфраструктура как код** (IaC): Terraform, Ansible.

* **Blue‑Green или Canary Deployments**: стратегии безопасного развёртывания.

  

## 9. Наблюдаемость (Observability)

  

* **Логирование**: централизованный сбор логов (ELK Stack, Graylog).

* **Метрики**: Prometheus, Grafana для мониторинга.

* **Распределённый трейсинг**: Jaeger, Zipkin для отслеживания запросов.

* **Аудит**: отслеживание действий пользователей и изменений.

  

## 10. Тестирование

  

* **Юнит‑тесты**: проверка отдельных компонентов.

* **Интеграционные тесты**: проверка взаимодействия между сервисами.

* **Контрактные тесты**: проверка соответствия API контрактам.

* **Нагрузочное тестирование**: оценка производительности под нагрузкой.

* **Chaos Engineering**: тестирование устойчивости к сбоям.

  

## 11. Документация

  

* **API Documentation**: автоматическая генерация (Swagger UI, ReDoc).

* **Архитектура системы**: диаграммы компонентов, последовательности.

* **Руководства**: для разработчиков, операторов, администраторов.

* **Change Log**: история изменений API и сервисов.

  

## 12. Организационные аспекты

  

* **Команды**: структура команд (по сервисам или функциям).

* **Коммуникация**: процессы взаимодействия между командами.

* **Стандарты разработки**: coding style, правила именования.

* **Управление зависимостями**: чёткие правила использования общих библиотек.

## 📘 СТРУКТУРА ТЗ МИКРОСЕРВИСА АВТОРИЗАЦИИ

### 1. Назначение и границы ответственности

- Централизованная выдача, валидация и отзыв токенов доступа
- Управление ролями, разрешениями, атрибутами пользователей и ресурсами
- Внешняя оценка политик доступа (RBAC / ABAC / ReBAC)
- Аудит событий доступа и изменений политик
- **Не входит:** бизнес-логика доменных сервисов, UI логина/регистрации (отдаётся на фронтенд или отдельный сервис Identity UI)

### 2. Функциональные требования

|Категория|Требования|
|---|---|
|**Протоколы**|OAuth 2.1 / OIDC Core 1.0. Поддержка `authorization_code + PKCE`, `client_credentials`, `refresh_token`. Implicit flow запрещён.|
|**Токены**|JWT (RFC 7519) для access token. Поддержка introspection (RFC 7662) и token revocation. Ротация signing keys без downtime.|
|**Авторизация**|Поддержка RBAC и/или ABAC. Вынос политик в движок (OPA, Casbin, Cedar). Claims в токене минимальны, детали подгружаются по необходимости.|
|**Управление доступом**|REST/gRPC API для CRUD ролей, групп, политик, маппинга `user ↔ resource ↔ action`. Поддержка версионирования политик.|
|**Сессии**|Stateless access tokens. Хранение refresh token/sessions в защищённом хранилище с TTL, поддержкой принудительного отзыва.|
|**События**|Webhook / Kafka / CloudEvents при: входе, выходе, смене роли, отзыве токена, превышении лимитов.|

### 3. Нефункциональные требования

|Параметр|Требование|
|---|---|
|**Доступность**|≥ 99.9% (SLA). Active-Active репликация, graceful degradation при падении зависимостей.|
|**Производительность**|Валидация токена ≤ 50ms (p95). Горизонтальное масштабирование без stateful зависимостей в hot path.|
|**Хранение**|PostgreSQL/MySQL для политик и аудитов. Redis/Memcached для кэша JWK, blacklist токенов, rate limits.|
|**Мультиарендность**|(опционально) `tenant_id` во всех запросах и токенах. Изоляция данных на уровне БД или схемы.|
|**Совместимость**|Kubernetes-native, readiness/liveness probes, graceful shutdown, config via env/ConfigMap/Secrets.|

### 4. Требования безопасности

- TLS 1.3 везде (in transit), шифрование чувствительных данных at rest (AES-256-GCM или аналог)
- Обязательный PKCE для публичных клиентов
- Rate limiting + CAPTCHA/2FA после N неудачных попыток
- Защита от replay, token fixation, privilege escalation
- Ротация секретов и ключей подписи каждые 30-90 дней (автоматизировано)
- Соответствие 152-ФЗ / GDPR / PCI-DSS (при обработке платежей)
- Immutable audit log (WORM или append-only таблица)

### 5. Интеграционные требования

- OpenAPI 3.1 + AsyncAPI для событий
- Поддержка API Gateway (Kong, Traefik, Apigee, NGINX) через JWT validation plugin
- SDK на Go/Java/Python/Node.js для валидации токенов и работы с политиками
- Совместимость с Service Mesh (Istio/Linkerd) через JWT request authentication

### 6. Эксплуатация и наблюдаемость

- Метрики (Prometheus): `auth_token_validation_duration_seconds`, `auth_policy_evaluation_total`, `active_sessions`, `error_rate`, `key_rotation_status`
- Логи: структурированные (JSON), correlation ID, без PII в plaintext
- Tracing: OpenTelemetry, сквозной trace_id через все сервисы
- Healthchecks: `/healthz`, `/readyz`, `/debug/vars`
- Автоматические бэкапы политик и аудитов, план восстановления (RPO ≤ 15min, RTO ≤ 1h)

## 🔄 2. Фундамент CI/CD (что заложить сразу)
1. **Pipeline-per-service**: триггер только при изменении файлов в папке сервиса + `infra/` + `packages/`
2. **Этапы пайплайна:**
   ```yaml
   lint → test (unit + integration) → build (multi-stage) → scan (trivy/snyk) → push (registry) → deploy (staging)
   ```
3. **Детекция изменений:** используйте `actions/changed-files` (GitHub) или `git diff --name-only` в GitLab CI, чтобы не пересобирать всё при правке одного сервиса.
4. **Артефакты:** образы в приватном registry (GHCR, GitLab, Harbor, Yandex Container Registry). Теги: `git-sha` + `semver`.
5. **Секреты:** GitHub Secrets / GitLab CI Variables / HashiCorp Vault. Никогда в коде.
6. **Шаблоны сервисов:** `cookiecutter` или GitHub Template Repo с готовым `Dockerfile`, healthchecks, CI-шаблоном, `.env`, `Makefile`.

## 🐳 3. Docker Compose vs Прод-оркестрация
| Сценарий | Инструмент | Что настраивать |
|----------|------------|-----------------|
| **Локальная разработка** | `docker-compose.yml` | `depends_on`, `healthcheck`, `restart: unless-stopped`, volumes для БД, `.env`, `docker compose watch` для hot-reload |
| **Интеграционные тесты** | Compose + `testcontainers` | Временные БД, мок-шлюз, мок-очереди, автоподнятие/удаление |
| **Продакшен** | Kubernetes + Helm / ArgoCD / Flux | `livenessProbe`, `readinessProbe`, `resources.limits/requests`, `PodDisruptionBudget`, `HorizontalPodAutoscaler`, `Ingress/Gateway API` |

> ✅ **Docker Compose — только для dev.** Для prod-восстановления используйте оркестратор с healthchecks и политиками рестарта.

## 🛡️ 4. Как заложить восстановление сервисов "на случай падения"
1. **Health Endpoints в каждом сервисе:**
   - `GET /healthz` → liveness (жив ли процесс)
   - `GET /readyz` → readiness (готов ли принимать трафик: БД подключена, кэш прогрет, зависимости доступны)
2. **Graceful Shutdown:**
   - Обработка `SIGTERM`
   - Завершение текущих запросов (drain)
   - Отписка от очередей/Kafka, закрытие пулов соединений
   - Timeout: 15-30 сек (настраивается в оркестраторе)
3. **Оркестратор реагирует:**
   - `liveness` упал → контейнер убивается и перезапускается
   - `readiness` упал → сервис исключается из балансировки шлюза, трафик идёт на живые реплики
   - `restartPolicy: Always` / `on-failure` в Compose (для dev)
4. **API Gateway роль:**
   - Читает readiness-статус (через service discovery или health-check polling)
   - Прекращает маршрутизацию на недоступные инстансы
   - Возвращает `503 Service Unavailable` или включает circuit breaker
