[[HTTP 1.1]]

Когда клиент отправляет заголовок **Content-Type** в запросе, он сообщает серверу, что данные в теле запроса отформатированы определенным образом. Если клиент хочет отправить серверу данные **JSON**, он установит **Content-Type** на "**application**/**json**". Клиент может установить заголовок **Accept**, чтобы сообщить серверу, какие форматы данных он может принимать. Благодаря этим двум заголовкам, **Content-Type** и **Accept**, клиент и сервер могут работать с форматами данных, которые они понимают и которые им необходимы для корректной работы.

Типичные значения, которые «имеет смысл» прописывать:

- HTML: `text/html`
- XML: `application/xml` (или реже `text/xml`)
- JSON: `application/json`
- CSS: `text/css`
- JavaScript: `text/javascript` или `application/javascript` (зависит от рекомендаций и стека)
- Картинки: `image/png`, `image/jpeg`, `image/svg+xml` и т.п.
- Файлы для скачивания: например, `application/pdf`, `application/zip` и т.д.

Для обычных веб‑страниц указывается тип `Content-Type: text/html; charset=utf-8`. **HTML (HyperText Markup Language)** — стандартизированный язык гипертекстовой разметки документов для веба, основной задачей которого является описание структуры и представления страниц в браузере, а не абстрактного обмена произвольными данными между системами. Его спецификации исторически разрабатываются консорциумом W3C, а в последние годы — совместно с группой WHATWG, и эволюция HTML5 шла именно в направлении богатого пользовательского интерфейса, мультимедиа и семантики для веб‑страниц. JSON и XML, напротив, проектировались в первую очередь как переносимые форматы структурированных данных, которые удобно парсить и валидировать в приложениях, поэтому в технических описаниях их обычно сравнивают между собой, а HTML выделяют отдельно как язык разметки документов, хотя формально все три — текстовые форматы с тегами/структурами и строгим синтаксисом

Наиболее распространенными форматами, встречающимися в современных API, являются JSON (JavaScript Object Notation) и XML (Extensible Markup Language).

**JSON (JavaScript Object Notation)** — текстовый формат представления структурированных данных, основанный на подмножестве синтаксиса объектов JavaScript, но ставший языконезависимым стандартом обмена данными между системами. Он оперирует всего двумя базовыми структурами — объектами (наборы пар «ключ–значение») и массивами (упорядоченные списки значений), поддерживая примитивы вроде строк, чисел, булевых значений и null, что делает формат компактным, легко читаемым человеком и очень удобным для веб‑API и конфигураций. Создателем JSON принято считать американского программиста **Дугласа Крокфорда**, который в начале 2000‑х, работая над веб‑проектами (включая компанию State Software), продвигал идею использования простого текстового формата поверх JavaScript для асинхронного обмена данными между браузером и сервером; в 2010‑е JSON был стандартизован ECMA и окончательно закрепился как де‑факто стандарт формата для REST‑API и фронтенд–бекенд взаимодействия. Популярность JSON во многом связана с его минимализмом по сравнению с более громоздкими форматами, простотой парсинга и тем, что структуры данных напрямую отображаются на объекты в большинстве современных языков программирования.

**XML (eXtensible Markup Language)** — расширяемый язык разметки, предназначенный для хранения и обмена структурированными данными в текстовом виде, в котором информация описывается вложенными элементами с тегами и атрибутами. Он возник как упрощённое и более практичное подмножество SGML (Standard Generalized Markup Language), мощного, но сложного метаязыка разметки, который использовался в крупных документационных и правительственных системах и оказался избыточным для массового интернета. Разработку XML в середине 1990‑х вёл консорциум W3C: рабочая группа по XML сформулировала основные принципы в 1996 году, а в феврале 1998 года первая спецификация получила статус рекомендации, открыв путь к широкому применению формата в веб‑сервисах, документации, конфигурациях и корпоративных интеграционных решениях. Вокруг XML со временем сформировалась целая экосистема стандартов и технологий — схемы XML Schema для описания структуры данных, XSLT для трансформации, XPath и XQuery для выборок и запросов, специализированные форматы наподобие DocBook или DITA для технической документации, а также отраслевые стандарты обмена, включая электронную коммерцию и интеграцию корпоративных систем. Исторически XML был основным универсальным форматом для сложных, строго типизированных структур данных и до сих пор активно используется там, где важны формальная схема, валидация и богатые средства обработки, хотя в веб‑разработке на прикладном уровне его в значительной степени вытеснил более лёгкий JSON.

Форматы передачи данных в API — это способы сериализовать структурированные данные (объекты, массивы, примитивы) в строку, которую можно передать по сети и потом точно восстановить обратно. Они различаются читаемостью, компактностью, возможностями типизации и поддержкой инструментами.

Для API выбор формата обычно сводится к компромиссу между универсальностью и скоростью. JSON остаётся основным форматом для публичных и внутренних HTTP‑API благодаря простоте, кросс‑языковой поддержке и хорошей интеграции с инструментами и браузерами. YAML и TOML чаще отдают конфигурациям и спецификациям, где первична читаемость человеком. XML используется, когда важно строгое контрактное описание, совместимость с наследием и богатый набор стандартов вокруг схем, валидации и трансформаций. Бинарные форматы вроде Protobuf, Avro, MessagePack и CBOR выбирают там, где критичны производительность, компактность и строгие схемы — в первую очередь во внутренних микросервисах, стриминге и высоконагруженных распределённых системах.

**JSON (JavaScript Object Notation)** — де‑факто стандарт для web‑API. Это текстовый формат на основе пар «ключ–значение» и массивов, синтаксически близкий к подмножеству JavaScript: объекты в фигурных скобках, массивы в квадратных, строки в кавычках, двоеточия и запятые как разделители. JSON поддерживает ограниченный набор типов (числа, строки, булевы, null, массивы и объекты), зато максимально прост для машинной обработки, хорошо парсится почти на любом языке и отлично подходит для REST и GraphQL. Документы обычно компактнее XML, но не содержат комментариев и встроенных ссылок/метаданных, а строгая структура облегчает валидацию по JSON Schema и генерацию кода.

**YAML (YAML Ain’t Markup Language)** — человеко‑ориентированный формат сериализации, в котором структура задаётся отступами, а не скобками. Он также опирается на концепцию словарей и списков, но позволяет писать более «естественно» и добавлять комментарии через `#`, поддерживает более широкий набор типов (даты, многострочные строки, алиасы и т.д.), что делает его удобным для конфигураций и описаний схем. YAML часто используют для конфигов в DevOps (Kubernetes, Ansible, CI/CD, docker‑compose), а также для описания OpenAPI‑спецификаций, но как основной формат передачи данных между сервисами применяется реже из‑за сложности парсинга, неоднозначностей с отступами и меньшей предсказуемости по сравнению с JSON.

**XML (eXtensible Markup Language)** — текстовый формат с древовидной структурой на основе тегов `<tag>...</tag>` и атрибутов. Он исторически широко применялся в веб‑сервисах (SOAP, старые интеграции) и по-прежнему силён там, где важно строгое контрактное описание (XSD‑схемы), неймспейсы, валидируемый документооборот и расширяемость. XML легче аннотировать метаданными, он поддерживает смешанный контент (текст с вложенными элементами), но более «тяжеловесен» по размеру и шумному синтаксису, что делает его менее удобным для современных легковесных API, хотя в enterprise‑интеграции и B2B‑обмене XML до сих пор широко используется.

**Form‑encoded (application/x-www-form-urlencoded)** — классический формат HTTP‑форм: пары «ключ=значение», закодированные как query‑строка (`key1=value1&key2=value2`), с URL‑кодированием спецсимволов. Такой формат часто применяют для простых HTML‑форм и некоторых OAuth‑эндпоинтов (например, получение токена), а также в старых API. Он прост, хорошо поддерживается в стандартных библиотеках, но плохо подходит для сложных вложенных структур и менее удобен для явной схемы данных.

**Multipart/form‑data** — формат для отправки «многокомпонентных» запросов, где тело разбито на части с собственными заголовками (часто файл + JSON/поля формы). Его использует браузер при загрузке файлов через `<form enctype="multipart/form-data">`, а в API его выбирают, когда нужно передать одновременно бинарные данные и структурированное описание. Основное отличие от обычного form‑encoded — поддержка файлов и больших бинарных блоков, хотя разбор и сборка таких сообщений сложнее, а сама структура менее очевидна, чем у чистого JSON.

**CSV (Comma-Separated Values)** — простой табличный формат, где строки представляют записи, а столбцы разделяются запятыми (или другими разделителями). CSV популярен для экспорта/импорта, аналитики, интеграции с BI и таблицами, а в API его иногда используют для выдачи больших табличных выборок (отчеты, выгрузки), потому что он компактный и легко читается инструментами анализа. При этом CSV фактически не поддерживает вложенные структуры, типы и строгую схему, поэтому плохо подходит для сложных объектных моделей и однозначной валидации.

**TOML (Tom’s Obvious, Minimal Language)** — конфигурационный формат, ориентированный на простоту и однозначность парсинга, с явной типизацией (даты, массивы, таблицы), который удобно читать и писать человеком. Его преимущественно используют для конфигов (например, в экосистеме Rust, некоторых сервисах и приложениях), а не как основной формат передачи данных между сервисами, но теоретически он может использоваться и в API, если требуется чёткая, но человеко‑ориентированная конфигурация.

**Protocol Buffers (Protobuf)** — бинарный формат сериализации от Google, основанный на заранее определённых .proto‑схемах, по которым генерируется код для разных языков. Он обеспечивает компактные сообщения и быструю (де)сериализацию, строгую типизацию и поддержку эволюции схемы без поломки старых клиентов. Protobuf является базой для gRPC и отлично подходит для высоконагруженных микросервисных взаимодействий, но плохо читается человеком без инструментов, сложнее отлаживается «на глаз», и требует согласованного управления схемами.

**Avro** — бинарный формат с сильной поддержкой эволюции схемы, популярный в экосистеме Hadoop, Kafka и систем потоковой обработки данных. В Avro схема может передаваться вместе с сообщением или храниться отдельно (например, в Schema Registry), а сериализация ориентирована на компактность и быстроту, что делает этот формат удобным для логов, событий и больших потоков данных. В HTTP‑API Avro используется реже, но часто применяется в внутренних event‑driven‑архитектурах и для межсервисных сообщений.

**MessagePack** — бинарный формат, логически очень близкий к JSON, но оптимизированный по размеру и скорости (можно воспринимать как «бинарный JSON»). Он сохраняет привычную структуру объектов и массивов, но кодирует данные более компактно, благодаря чему лучше подходит для систем с ограниченным трафиком или высокой нагрузкой. Однако меньшая распространенность в веб‑инструментах и необходимость специальных библиотек делают его менее универсальным, чем обычный JSON.

**CBOR (Concise Binary Object Representation)** — еще один бинарный формат для структур данных, совместимый по модели с JSON, но расширяющий типы (например, теги, бинарные строки, даты) и ориентированный на компактность. CBOR часто используют в IoT и ограниченных устройствах, где важно минимизировать размер сообщений, а также в протоколах с жесткими требованиями к эффективности, но в классических REST‑API он встречается заметно реже.

**BSON (Binary JSON)** — бинарное представление JSON, активно используемое, например, MongoDB для внутреннего хранения. BSON поддерживает дополнительные типы (даты, бинарные данные, ObjectId) и позволяет быстро обходить дерево документов, что удобно для баз данных и некоторых RPC‑схем. В веб‑API BSON встречается не так часто, так как требует специфической поддержки клиентами, но может использоваться в системах, тесно связанных с MongoDB или в собственных бинарных протоколах.

**Plain text / custom formats** — иногда API просто возвращают или принимают неструктурированный текст (лог, конфиг, скрипт) или самописный формат с минимальными разделителями. Такой подход удобен для очень специализированных задач или старых протоколов, но почти не дает универсальности и требует отдельных парсеров для каждого клиента, поэтому в современных системах его стараются заменять стандартными текстовыми или бинарными форматами.
