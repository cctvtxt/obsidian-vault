[[HTTP 1.1]]

Кэширование в целом есть механизм оптимизации производительности ИТ-систем. Суть кэширования заключается в добавлении буфера между приложением и БД или приложением и сервисом для хранения часто используемых данных.

Сервер **должен явно указывать**, можно ли кэшировать ответ и на какой срок.

**Без этих заголовков** клиент/прокси обычно не кэшируют ответ или применяют эвристики (что ненадёжно). Клиент (браузер, мобильное приложение) или прокси/CDN **реализует логику кэширования**.

1. **Сохраняет** ответ вместе с заголовками кэширования.
2. **Проверяет свежесть** перед использованием:
3. **Обновляет кэш** при получении ответа `304 Not Modified` (данные те же) или загружает новую версию при `200 OK`.

Прокси и CDN действуют аналогично, но кэшируют для множества клиентов.

---

Каждая запись в любом кэше содержит 3 поля:

- **Ключ** (Key). Обычно ключом является кортеж из параметров запроса или параметров БД;
- **Значение** (Value). Это сами данные;
- **Время жизни** (Time to Live). Это «срок годности» записи — время, через которое запись нужно обновить или удалить.

- Свежесть кэша помогает понять, может ли элемент в кэше быть кандидатом на обслуживание клиенту. Контент в кэше будет использоваться для обслуживания запроса только в том случае, если он находится в пределах времени свежести, указанного политикой кэширования.
- Валидация – это проверка устаревшего контента на исходном сервере с целью уточнить последнюю версию элемента.
- Инвалидация – это процесс удаления контента из кэша раньше срока истечения его свежести. Это необходимо, если элемент был изменен на исходном сервере, а в кэше хранится его устаревший элемент, что вызовет значительные проблемы для клиента.

**Оценка эффективности кэширования:**

1. **Процент попадания (hitrate, hit ratio)** — показывает, сколько запросов в процентном выражении получают ответы из кэша, а какой — из БД: при этом **_Hitrate = кол-во попаданий в кэш / (кол. попаданий в кэш + кол. промахов)_**
2. \*\*Сравнение количества элементов в кэше с количеством данных в БД
3. **Размер записи кэша в байтах** (средний размер, расчёт перцентилей размера)
4. **Количество записей, вытесняемых из кэша в секунду (evictions)**
5. **Задержка получения данных из кэша (latency)** — требование: задержка получения данных из кэша должна быть меньше задержки получения данных из оригинального источника
6. \*\*Количество ошибок получения данных из кэша в секунду

**Классификация подходов к кэшированию**

**По порядку размещения кэша:**

1. **В браузере (browser cache).** Кэш будет сохраняться на уровне фронта и храниться на устройстве пользователя.
2. **В прокси (proxy cache).** Прокси может быть размещён между фронтом и приложением или между приложением и бэком. Прокси будет фильтровать запросы: часть направлять по исходному пути, а на часть отвечать кэшированными данными.
3. **В базе данных (database cache).** Может быть реализован с помощью плагинов или in-memory индексов некоторых СУБД.
4. **В памяти приложения (in-memory cache).** Кэш сохраняется только в рамках выполняемого процесса и одного сервера. При перезапуске приложения кэш будет утерян. Если экземпляры приложения развёрнуты на нескольких серверах, на каждом сервере будет свой кэш.
5. **Распределённый кэш (distributed cache).** Кэш является общим для нескольких экземпляров приложения. Часто реализуется внешним сервисом (Memcached или Redis), но также может быть представлен библиотекой внутри приложения.

**По порядку взаимодействия с кэшем:**

1. **Кэширование на стороне (Cache-Aside).** Приложение обращается в кэш, и если данные есть в кэше, то используются данные из него. Если данных в кэше нет, то приложение делает запрос к основному хранилищу.
2. **Сквозная запись (Write-Through).** Данные записываются одновременно в кэш и в основное хранилище данных.
3. **Write-Behind.** Сначала обновляется кэш, затем — основное хранилище данных.
4. **Сквозное чтение (Read-Through).** Похож на кэширование на стороне: если данных в кэше нет, то данные берутся из БД. Важное отличие: координация запросов в БД происходит на стороне кэша, а не приложения.
5. **Упреждающее чтение (Read-Ahead).** Данные помещаются в кэш на основе предположения о том, какие данные будут запрошены следующими.

---

Кэш, применительно к браузеру, это каталог с файлами на жестком диске, куда сохраняются элементы от просматриваемых вами интернет-страниц. Например, там хранятся рисунки, скрипты. При обновлении страницы браузер больше не загружает эти данные из интернета, а просто берет их с вашего компьютера, что значительно быстрее, чем загружать их повторно из сети. Поэтому технология кэширования данных браузером эффективна и полезна. Браузер сохраняет кеш, который считается частным кешем, относящимся к конкретному пользователю.

## Заголовки кэширования

Благодаря различным версиям протокола HTTP появилось несколько разных заголовков кэширования с различными уровнями сложности:

- **Expires** очень простой. Он устанавливает время в будущем, когда истекает срок действия контента. На этом этапе любые запросы на контент будут возвращены на исходный сервер. Этот заголовок, вероятно, лучше всего использовать только как запасной.
- **Cache-Control** – современная замена **Expires**. Он хорошо поддерживается и реализует гораздо более гибкий дизайн. Почти во всех случаях он предпочтительнее **Expires**, но иногда может не помешать установить оба заголовка. Особенности параметров, которые вы можете установить с помощью **Cache-Control**, мы обсудим чуть позже.
- Заголовок **Etag** используется валидацией кэша. Сервер может предоставить уникальный **Etag** для элемента. Когда придет время проверить кэшированный контент, кэширующий сервер может отправить обратно **Etag** контента. Исходный сервер либо сообщит ему, что контент не изменился, либо отправит обновленный контент (с новым **Etag**).
- **Last-Modified** – этот заголовок указывает, когда элемент был изменен в последний раз. Он может использоваться как часть стратегии валидации для обеспечения свежего контента.
- **Content-Length** – этот заголовок имеет большое значение при определении политик кэширования. Некоторое программное обеспечение отказывается кэшировать контент, если оно заранее не знает о размере кэша.
- **Vary** – это заголовок, который сообщает системе, по каким заголовкам кэширования она может понять, что в кэше находится правильный контент. Чаще всего он используется для указания ключа заголовка **Accept-Encoding**, чтобы кэш мог различать сжатый и несжатый контент.

Cache-Control используется для определения политики кэширования. С помощью этого заголовка можно установить несколько инструкций политики:

- **no-cache**: указывает, что любой кэшированный контент нужно повторно проверить на каждом запросе, прежде чем передавать клиенту. Это, по сути, сразу же помечает контент как устаревший, но позволяет использовать методы валидации, чтобы избежать повторной загрузки всего элемента.
- **no-store**: указывает, что содержимое не может быть кэшировано. Этот флаг уместно установить на конфиденциальные данные.
- **public**: указывает, что контент является общедоступным, что означает, что он может быть кэширован браузером и любыми промежуточными кэшами. Для запросов, использующих HTTP-аутентификацию, ответы по умолчанию помечены как **private**. Этот заголовок отменяет эту настройку.
- **private**: означает, что контент является конфиденциальным. Такой контент может храниться в браузере пользователя, но не должен кэшироваться промежуточными сторонами. Этот флаг часто используется для пользовательских данных.
- **max-age**: настраивает максимальный срок хранения кэшированного контента, прежде чем его нужно будет повторно загрузить с исходного сервера. По сути, он заменяет **Expires** и является основой для определения свежести контента. Этот параметр принимает значение в секундах; максимально допустимый срок свежести – 1 год (31536000 секунд).
- **s-maxage**: очень похож на max-age , поскольку указывает время, в течение которого можно хранить кэш. Разница в том, что этот параметр применяется только к промежуточным кэшам. Вместе с max-age он позволяет более гибко разрабатывать политику.
- **must-revalidate**: указывает на то, что информацию о свежести в **max-age**, **s-maxage** или **Expires** следует строго соблюдать. Нельзя обслуживать устаревший контент ни при каких обстоятельствах. Этот флаг предотвращает использование кэшированного контента в случае перебоев в сети и подобных сценариях.
- **proxy-revalidate**: работает так же, как и предыдущий, но применяется только к промежуточным прокси. В этом случае браузер пользователя можно использовать для обслуживания устаревшего контента в случае прерывания сети, но промежуточные кэши не могут использоваться для этой цели.
- **no-transform**: указывает кэшам, что при любых обстоятельствах им не разрешается изменять полученный контент по соображениям производительности. Это означает, например, что кэш не может отправлять сжатые версии содержимого, если он не получил их с исходного сервера.

Заголовки кэширования **могут прописывать и сервер, и клиент** — но роли и механизмы у них принципиально разные.

Сервер **задаёт правила кэширования** для ответов (ресурсов: HTML, CSS, JS, изображений и т. п.). Он включает в HTTP‑ответ специальные заголовки, которые говорят клиентам и промежуточным прокси/CDN

Клиент (браузер, приложение) **отправляет заголовки в запросе**, чтобы:

- сообщить, какой ресурс у него уже есть в кэше;
- попросить сервер проверить, актуален ли он;
- повлиять на поведение промежуточного кэша.

Таким образом, кэширование — это **двусторонний диалог**, где сервер задаёт политику, а клиент и промежуточные системы ей следуют, используя стандартизированные заголовки.

При этом **HTTP‑заголовки** служат сигналами для приложения, которое координирует работу с внешними кэшами (Redis/Memcached). **Redis и Memcached** глухие к HTTP‑заголовкам. Они хранят данные с TTL, который задаёт приложение, при этом **Node.js** как раз анализирует заголовки (`Cache-Control`, `ETag` и др.) и принимает решения о кэшировании на уровне кода

### Проблемы кеширования и способы их решения

---

**Дублирование запросов в полёте при кэшировании**

Дублирование запросов возникает, когда:

1. С минимальной разницей во времени поступают одинаковые запросы.
2. Нужных данных нет в кэше.

В результате одинаковые запросы одновременно начинают нагружать базу данных (БД). Решение: паттерн **Singleflight** — это паттерн, который предотвращает дублирование одновременных запросов к ресурсу (HTTP-запросы, запросы к БД) благодаря **механизму объединения (коалиции) идентичных одновременных запросов** в один фактический вызов к ресурсу, а затем распространению полученного результата среди всех ожидающих клиентов.

---

**Недоступность данных во время сбоя**

Здесь пользователь видит в приложении ошибку — хотя во многих случаях **не обязательно показывать самые свежие данные**. Можно отобразить устаревшие данные с пояснением. Решение: паттерн **fallback** (запасной вариант). Чтобы избежать ошибок, используют **альтернативный источник данных** — это суть паттерна **fallback**. Он позволяет системе «переключаться» на резервные данные при сбое основного источника.

---

**Проблемы с таймаутами при реализации кэширования**

**Суть проблемы:** Таймауты мешают эффективному наполнению кэша, что приводит к следующим ситуациям:

- Данные в кэше постепенно устаревают, так как не обновляются вовремя.
- Нагрузка на бэкенд растёт, поскольку приложение постоянно обращается к нему из-за отсутствия актуальных данных в кэше.
- Увеличивается время ответа системы, что усугубляет первоначальный сбой, вызвавший задержки.

**Способы решения:**

1. \*\*Увеличение таймаута между кэшем и бэком
2. **Установка короткого таймаута между приложением и кэшем**
3. **Комбинирование двух таймаутов**
4. **Альтернативный подход — пропуск части данных:** в некоторых случаях лучше не отображать отдельные блоки информации (например, необязательные элементы интерфейса), чем замедлять загрузку всего приложения. Добавление дополнительного таймаута между приложением и кэшем помогает контролировать, какие данные критичны для быстрой загрузки, а какие могут быть загружены позже.
