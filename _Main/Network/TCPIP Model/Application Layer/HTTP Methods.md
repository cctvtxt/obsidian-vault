[[HTTP 1.1]]

REST (Representational State Transfer) использует стандартные HTTP‑методы для взаимодействия с ресурсами

| Method  | Idempotent | Safe |
| ------- | ---------- | ---- |
| OPTIONS | Да         | Да   |
| GET     | Да         | Да   |
| HEAD    | Да         | Да   |
| PUT     | Да         | Нет  |
| POST    | Нет        | Нет  |
| DELETE  | Да         | Нет  |
| PATCH   | Да         | Нет  |

**Идемпотентность** означает, что многократное повторение одного и того же HTTP‑запроса даёт **тот же эффект**, что и однократное выполнение.

**Безопасность** означает, что метод **не изменяет состояние сервера**. Такие методы предназначены только для чтения данных.

Метод **GET** предназначен для получения ресурса. Он считается безопасным (не меняет состояние сервера) и идемпотентным (повторный вызов не приводит к иным последствиям). Ключевая особенность — возможность кэширования ответов, что повышает производительность. Однако в GET‑запросах нельзя передавать большие объёмы данных (ограничение длины URL), а чувствительная информация (пароли, токены) в URL небезопасна — её могут увидеть в логах или истории браузера. Кроме того, избыточное использование GET для операций, изменяющих состояние (например, удаление через `/delete?id=123`), нарушает REST‑семантику и создаёт риски непреднамеренных действий при повторном открытии ссылки.

Метод **POST** служит для создания нового ресурса или выполнения действия, результат которого не поддаётся однозначной идентификации. В отличие от GET, POST передаёт данные в теле запроса, что снимает ограничения на объём и позволяет отправлять сложные структуры (JSON, формы). Однако POST не является идемпотентным: повторная отправка может привести к дублированию (например, два заказа вместо одного). Также POST‑запросы не кэшируются, что иногда снижает эффективность. Ошибки в обработке POST‑запросов могут оставлять «мусорные» записи на сервере, если логика не предусматривает проверку уникальности.

Метод **PUT** используется для полной замены ресурса по указанному URI. Он идемпотентен: многократный вызов с теми же данными не изменит результат после первого применения. Особенность PUT — требование передавать весь ресурс целиком, даже если нужно обновить лишь одно поле. Это может быть неэффективно для больших объектов. Если URI не существует, PUT может создать ресурс, но это зависит от реализации сервера. Проблемы возникают при конфликтах версий: если два клиента одновременно обновляют ресурс, одно из изменений может быть потеряно без механизма проверки версий (ETag, Last‑Modified).

Метод **PATCH** предназначен для частичного обновления ресурса. В отличие от PUT, он позволяет изменять только нужные поля, снижая нагрузку на сеть. Однако PATCH не является строго идемпотентным: результат может зависеть от порядка применения патчей (например, инкремент счётчика). Формат патча (JSON Patch, JSON Merge Patch) должен быть согласован между клиентом и сервером, иначе возможны ошибки интерпретации. Некорректные патчи могут привести к несогласованности данных, если сервер не проверяет целостность изменяемых полей.

Метод **DELETE** удаляет ресурс по указанному URI. Он идемпотентен: повторное удаление уже удалённого ресурса не вызывает ошибок (обычно возвращается код `204 No Content` или `404 Not Found`). Проблема возникает при «мягком удалении» (пометка как удалённого без физического удаления): если клиент считает ресурс удалённым, а сервер сохраняет его, это может привести к логическим ошибкам. Также DELETE‑запросы иногда блокируются прокси‑серверами или браузерами, что требует явного указания метода в настройках CORS.

Методы **HEAD** и **OPTIONS** выполняют вспомогательные функции. HEAD возвращает только заголовки ответа (без тела), что полезно для проверки существования ресурса или его метаданных (например, `Content-Length`). OPTIONS описывает поддерживаемые методы и параметры для ресурса, помогая клиенту согласовать взаимодействие. Оба метода безопасны и идемпотентны, но их некорректная реализация на сервере может раскрывать избыточную информацию о структуре API.

Ключевые общие проблемы REST‑методов:
**Неоднозначность семантики**: разработчики иногда нарушают принципы (например, используют POST для чтения), что усложняет поддержку API.
**Обработка ошибок**: отсутствие стандартизированных кодов ошибок для бизнес‑логики (например, «недостаточно средств») вынуждает использовать нестандартные подходы.
**Безопасность**: передача токенов в URL (GET), отсутствие CSRF‑защиты для POST/PUT/DELETE, недостаточная валидация входных данных.
**Кэширование**: неправильное управление заголовками `Cache-Control` и `ETag` может приводить к устаревшим данным на клиенте.
**Версионирование**: изменения в API без учёта обратной совместимости ломают клиентские приложения.

2. **_Метод_** _(_**_Method_**_). Стандартные методы_ **_API_** _позволяют выполнять_ **_CRUD_**_-операции: создавать, получать, изменять и удалять ресурсы._
   - **_GET_**_: просит сервер_ **_извлечь_** _ресурс_
   - **_POST_**_: просит сервер_ **_создать_** **_новый_** _ресурс_
   - **_PUT_**_: просит сервер_ **_отредактировать_**_/_**_обновить_** **_существующий_** _ресурс_
   - **_DELETE_**_: просит сервер_ **_удалить_** _ресурс_
   - **_PATCH_**_: просит сервер_ **_частично_** **_отредактировать_**_/_**_обновить_** **_существующий_** _ресурс_
   - **_HEAD_**_: Запрос заголовков_ **_ответа без тела_**_._
   - **_OPTIONS_**_: Запрос_ **_поддерживаемых методов_** _для ресурса._
   - **_CONNECT_**_: Установка туннеля к серверу._
   - **_TRACE_**_: Возвращает полученный запрос (для отладки)._
