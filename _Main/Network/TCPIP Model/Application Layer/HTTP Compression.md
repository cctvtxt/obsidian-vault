**Сжатие** — важный способ повысить производительность веб-сайта. Для некоторых документов уменьшение размера на 70 % снижает требования к пропускной способности. С годами алгоритмы сжатия становились все более эффективными, а новые алгоритмы поддерживаются клиентами и серверами.

На практике веб-разработчикам не нужно внедрять механизмы сжатия — они уже реализованы в браузерах и на серверах, но необходимо убедиться, что сервер настроен правильно. Сжатие происходит на трех разных уровнях:

1. сначала некоторые форматы файлов сжимаются с помощью специальных оптимизированных методов,
2. затем может применяться общее сжатие на уровне HTTP (ресурс передается в сжатом виде от начала до конца),
3. и, наконец, сжатие может быть настроено на уровне соединения между двумя узлами HTTP-соединения.

Все современные браузеры и серверы поддерживают эту функцию, и единственное, о чем нужно договориться, — это **алгоритм сжати**я. Эти алгоритмы оптимизированы для работы с текстом. В 1990-х годах технологии сжатия развивались стремительными темпами, и в набор возможных вариантов добавилось множество новых алгоритмов. Сегодня актуальны только два: `gzip` — самый распространенный, и `br` — новый перспективный алгоритм.

Браузер отправляет заголовок `Accept-Encoding` с поддерживаемым **алгоритмом и порядком его приоритетности, сервер выбирает один из них**, использует его для сжатия тела ответа и с помощью заголовка `Content-Encoding` сообщает браузеру, какой алгоритм он выбрал. Поскольку согласование контента использовалось для выбора представления на основе его кодирования, сервер должен отправить в ответе заголовок `Vary` с указанием как минимум `Accept-Encoding` вместе с этим заголовком. Таким образом, кэши смогут сохранять различные представления ресурса.

## Как работает сжатие в HTTP

- Клиент в запросе посылает заголовок:
    
    text
    
    `Accept-Encoding: gzip, deflate`
    
    Это говорит серверу: «я умею принимать сжатые ответы».
    
- Если сервер решает сжать, он отвечает:
    
    text
    
    `Content-Encoding: gzip Content-Type: text/html; charset=utf-8`
    
    и в `body` отправляет уже упакованные gzip‑данные.
    
- Современные HTTP‑клиенты (`curl`, `fetch`, `axios`, Go‑http, .NET HttpClient и т.п.) по умолчанию умеют распаковывать `gzip` и `deflate` автоматически, если видят `Content-Encoding: gzip`.

## Как «указать» клиенту распаковывать gzip

Практически везде **не нужно явно «указывать»** клиенту распаковывать — это поведение по умолчанию. Но важно, чтобы:

1. **Клиент корректно отправлял `Accept-Encoding`**  
    Обычно:
    
    text
    
    `Accept-Encoding: gzip, deflate`
    
    или просто:
    
    text
    
    `Accept-Encoding: gzip`
    
    У большинства библиотек это включено «из коробки» (браузеры, `curl`, `fetch`, `requests` и т.п.).[](https://ruweb.net/articles/kak-uskorit-rabotu-sajta-s-pomoshhyu-gzip-szhatiya)
    
2. **Сервер отвечал `Content-Encoding: gzip`**  
    Пример Nginx:
    
    text
    
    `gzip on; gzip_vary on; gzip_types text/plain text/css application/json application/javascript text/xml;`
