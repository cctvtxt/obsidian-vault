[[HTTP 1.1]] [[TCP]]

Так как HTTP — это клиент-серверный протокол, HTTP сессия состоит из трёх фаз:

1. Клиент устанавливает TCP соединения (или другое соединение, если не используется TCP транспорт).
2. Клиент отправляет запрос и ждёт ответа.
3. Сервер обрабатывает запрос и посылает ответ, в котором содержится код статуса и соответствующие данные.

Начиная с версии HTTP/1.1, **после третьей фазы соединение не закрывается**, так как клиенту позволяется инициировать другой запрос. 2-я и 3-я фазы повторяются

## Установка соединения

Так как HTTP это клиент-серверный протокол, соединение всегда устанавливается клиентом. Открыть соединение в HTTP — **значит установить соединение через соответствующий транспорт, обычно TCP**.

В случае с TCP, в качестве порта HTTP сервера по умолчанию на компьютере используется порт 80, хотя другие также часто используются, например 8000 или 8080. URL загружаемой страницы содержит доменное имя и порт, который можно и не указывать если он соответствует порту по умолчанию.

---

Управление соединением является ключевой темой HTTP: открытие и поддержка соединения оказывает значительное влияние на производительность веб-приложений. В HTTP/1.x имеются следующие модели: _краткосрочные соединения_, _постоянные соединения_ и _конвейерная обработка HTTP_ (_HTTP pipelining_).

В качестве транспортного протокола, обеспечивающего связь между клиентом и сервером, HTTP по большей части использует TCP. Это **краткосрочные (short-lived) соединения**: при каждой отправке запроса открывается новое соединение, которое закрывается после того, как ответ получен.

Такой модели присущи проблемы в отношении производительности: **ресурсы приходится затрачивать на открытие каждого соединения TCP**. Клиенту и сервером необходимо обмениваться несколькими сообщениями. При отправке каждого запроса приходится считаться с запаздыванием и пропускной способностью сети. Современным веб-страницам требуется выполнять множество (десятки) запросов для передачи необходимой информации, что делает данную модель неэффективной.

## В HTTP/1.1 были созданы две новые модели

**Модель постоянного соединения** оставляет соединение открытым между последовательными запросами, экономя время, требуемое для открытия новых соединений. **Модель конвейерной обработки** HTTP делает следующий шаг - она позволяет отсылать несколько запросов подряд, не дожидаясь ответа, что существенно сокращает время ожидания в сети.

Заголовок `Connection` управляет поведением сетевого соединения между клиентом и сервером в HTTP‑сессиях. Ключевая задача — определить, будет ли соединение **закрыто** после текущего запроса/ответа или **переиспользовано** для последующих запросов.

### Значения заголовка `Connection`

Соединение _остаётся открытым_ для последующих запросов/ответов в рамках той же TCP‑сессии.

```http
Connection: keep-alive
```

Соединение _обязательно закрывается_ после завершения текущей транзакции (запроса + ответа).

```http
Connection: close
```

В HTTP/1.1 по умолчанию подразумевается `keep-alive` (если не указано иное). В HTTP/1.0 — `close`.

---

Управление соединениями через `Connection` — это **низкоуровневый механизм HTTP**, который в JS‑клиентах обычно скрыт за абстракциями (Fetch API, `http`, `axios`). Переиспользование соединений реализуется через:

- Пул TCP‑соединений.
- Таймауты неактивности.
- Ограничения на количество соединений.

Для большинства приложений достаточно полагаться на дефолтное поведение браузера/Node.js. Ручное управление требуется только в специфических сценариях (например, при работе с legacy‑серверами или тонкой оптимизации производительности). В браузерах и Node.js переиспользование соединений реализуется на уровне **HTTP‑клиента** (не напрямую через JS‑код). Разработчик лишь косвенно влияет на это через настройки запросов.

**Пул соединений**  
Браузер/Node.js поддерживает _пул активных TCP‑соединений_ к каждому хосту. При новом запросе клиент пытается:
**Ограничения пула**

- Найти _свободное активное соединение_ к нужному хосту.
- Если найдено — использовать его.
- Если нет — создать новое TCP‑соединение.
  **Ограничения пула**
- В браузерах обычно **6–8 соединений на домен** (зависит от браузера и версии HTTP)
- В Node.js лимит настраивается через `agent.maxSockets` (по умолчанию — `Infinity`).
  **Таймауты.** Соединения закрываются:
- По таймауту без активности (например, 5–60 сек).
- При явном указании `Connection: close`.
- При достижении лимита параллельных запросов.
  **HTTP/2 и HTTP/3.** В этих версиях протокола:
- Используется **одно соединение** на домен для всех запросов (мультиплексирование).
- Заголовок `Connection` игнорируется (управление соединением на уровне протокола).
