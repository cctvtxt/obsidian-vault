Rогда речь идёт о **защищённых соединениях (HTTPS)** или современных версиях протокола (HTTP/2, HTTP/3), появляется понятие **handshake** — это фаза установки защищённого канала перед передачей данных.

## HTTPS (HTTP over TLS/SSL)

Здесь handshake — это **TLS-рукопожатие**, цель которого:

- аутентифицировать сервер (и опционально клиента);
- согласовать алгоритмы шифрования;
- обменяться ключами для защищённой передачи данных.

**Основные этапы TLS-рукопожатия (на примере TLS 1.2/1.3):**

1. **Client Hello** — клиент отправляет:

- версию TLS;
- список поддерживаемых шифронаборов;
- случайное число (client random).

2. **Server Hello** — сервер отвечает:

- выбранной версией TLS;
- выбранным шифронабором;
- случайным числом (server random);
- своим сертификатом.

3. **Проверка сертификата** — клиент проверяет валидность сертификата сервера.
4. **Обмен ключами** (зависит от алгоритма):

- **RSA**: клиент шифрует pre‑master secret открытым ключом сервера.
- **DH/ECDH**: стороны генерируют общий секрет через обмен открытыми значениями

1. **Генерация сеансовых ключей**: на основе client random, server random и pre‑master secret создаются ключи для шифрования.
2. **Финализация** — стороны отправляют сообщения `Finished`, подтверждая успешное рукопожатие.

## HTTP/2

HTTP/2 **не вводит новый handshake** — он работает поверх TLS (как HTTPS).

## HTTP/3

HTTP/3 кардинально меняет транспортный уровень — вместо TCP используется **QUIC** (работающий поверх UDP).
