Это классическое противоречие, и у него есть чёткий ответ.

---

## Два полюса и где правда

**Полюс A — "умный gateway":** маппинг, трансформация, фильтрация полей, версионирование. Гибко, безопасно, но код трогаешь при каждом изменении API. Здесь ответы для клиента формируются на стороне gateway 

**Полюс B — "тупой gateway":** только роутинг, аутентификация, rate limit. Код не трогаешь, но контракт внутренних сервисов = публичный контракт.

Твоя цель — полюс B. Это называется **passthrough / thin gateway**, и это _валидный_ архитектурный выбор при одном условии:

> Ответственность за форму публичного ответа переезжает с gateway на микросервис.

То есть не gateway адаптирует ответ под клиента — а каждый микросервис сам отдаёт ровно то, что нужно клиенту, не больше и не меньше.

---

## Что меняется в микросервисах

Каждый микросервис должен иметь **два слоя контрактов**:

```
internal/          ← gRPC между сервисами (account-vault ↔ auth-orchestrator)
│   accounts.proto      богатая модель, все поля, внутренние статусы
│
public/            ← то что gateway пробрасывает клиенту напрямую
    accounts.proto      только публичные поля, client-safe нейминг, stable schema
```

Gateway подключается только к `public/` proto. Внутренние сервисы общаются через `internal/`. `account-vault` сам решает что попадает в публичный ответ — и никогда не кладёт туда `passwordHash`, `internalFlags` и т.д.

## Как выглядит config-driven gateway

```typescript
// routes.config.ts — единственное что меняется при новом эндпоинте
export const routes: Route[] = [
  {
    path: '/auth/register',
    method: 'POST',
    grpcService: 'AuthOrchestrator',
    grpcMethod: 'Register',
    auth: false,
    rateLimit: 'strict',
  },
  {
    path: '/accounts/me',
    method: 'GET',
    grpcService: 'AccountVaultAccounts',
    grpcMethod: 'GetAccountById',
    auth: true,           // guard инжектит accountId из JWT
    rateLimit: 'default',
  },
  {
    path: '/accounts/api-keys',
    method: 'POST',
    grpcService: 'AccountVaultApiKeys',
    grpcMethod: 'CreateApiKey',
    auth: true,
    rateLimit: 'default',
  },
];
```

Gateway читает этот конфиг, динамически регистрирует маршруты, прокидывает HTTP body → gRPC request, gRPC response → HTTP response. Код gateway не трогается.

---

## Что gateway всё равно делает (и это нормально)

Это инфраструктурный код, написанный один раз:

```
JWT verify → accountId в контекст
Rate limiting по IP / accountId
CORS, cookies, security headers
gRPC error codes → HTTP status codes  (NOT_FOUND → 404, UNAUTHENTICATED → 401)
Request ID / tracing headers
```

Маппинг gRPC статусов в HTTP — это тоже конфиг, не бизнес-логика:

```typescript
const grpcToHttp: Record<number, number> = {
  [status.OK]:                200,
  [status.NOT_FOUND]:         404,
  [status.ALREADY_EXISTS]:    409,
  [status.UNAUTHENTICATED]:   401,
  [status.PERMISSION_DENIED]: 403,
  [status.INVALID_ARGUMENT]:  400,
  [status.INTERNAL]:          500,
};
```

## Граница которую нельзя нарушать

|Можно в gateway|Нельзя в gateway|
|---|---|
|Проверить JWT подпись|Знать что значит `accountStatus: BANNED`|
|Добавить `X-Request-Id`|Переименовывать поля ответа|
|Выбрать HTTP статус по gRPC коду|Фильтровать поля из ответа|
|Rate limit по `accountId` из токена|Объединять ответы двух сервисов|
|Валидировать Content-Type|Добавлять бизнес-поля в ответ|

Агрегация ответов нескольких сервисов (BFF-паттерн) — это уже не thin gateway, это отдельный сервис. Для MVP не нужен.

---

## Итого

Твой подход правильный. Стремись к **thin/passthrough gateway + умные публичные контракты на стороне микросервисов**. Платишь за это дисциплиной: каждый микросервис сам отвечает за то, что видит клиент, и никогда не кладёт в публичный proto внутренние детали. Gateway трогаешь только чтобы добавить строку в `routes.config.ts`.
