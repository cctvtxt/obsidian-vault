Ключевая роль API Gateway — представлять собой единую точку входа для всех клиентов, скрывая сложность распределенной системы микросервисов.

**Главные задачи и обязанности:**

1. **Оболочка и маршрутизация (Routing):** Определение, какой из внутренних микросервисов должен обработать запрос, на основе пути (URL), заголовков или других параметров. Это скрывает внутреннюю архитектуру и расположение сервисов.
2. **Сокращение сложности конфигурации:** Обеспечение единого места для настройки, а не распределенной конфигурации каждого отдельного сервиса.

**Нефункциональные требования и сквозные кросс- cutting concerns (применялись к любому сервису):**

- **SSL/TLS Termination:** Дешифрирование внешних запросов, освобождая внутренние сервисы от заботы о безопасности.
- **Логирование (Logging):** Централизованный сбор информации о всех запросах для аудита и диагностики.
- **Метрики приложения (Application Metrics):** Сбор данных о производительности,latency, ошибках для мониторинга.
- <mark style="background:#ff4d4f">**Нагрузочное распределение (Load Balancing):** Равномерное распределение запросов между экземплярами одного сервиса.</mark> Данная зона ответственности передана внешнему балансировку (envoy настраивается в k8s), благодаря чему может быть создано несколько подов api gateway. 
- **Аутентификация и Авторизация (Authentication & Authorization):** Проверка подлинности пользователя/клиента и подтверждение прав доступа к конкретным ресурсам.
- **Согласование контента (Content Negotiation):** Определение формата данных (JSON, XML) для запроса и ответа.
- **Кэширование запросов (Caching):** Хранение часто запрашиваемых ответов для снижения нагрузки на бэкенд и ускорения ответов клиентам.
- **Сжатие контента (Content Compression):** Сокращение размера передаваемых данных (например, gzip) для экономии трафика.
- **Лимитирование скорости (Rate Limiting):** Ограничение количества запросов от клиента в единицу времени для защиты сервисов от перегрузок и DDoS-атак.

**Дополнительные ключевые обязанности:**

3. **Агрегация данных (Data Aggregation):** Выполнение сложных сценариев, когда для формирования одного ответа клиенту требуется вызвать несколько различных микросервисов. Это значительно упрощает работу фронтенда и уменьшает количество последовательных запросов.
4. **Трансформация данных (Data Transformation):** Изменение формата или структуры данных "на лету" (например, смена ключей в JSON, объединение полей), чтобы клиент не должен был этого делать.
5. **Обработка ошибок и отказоустойчивость (Error Handling & Resilience):** Централизованная обработка ошибок от бэкенд-сервисов, реализация механизмов таймаутов, повторов (retries) и отказоустойчивости, чтобы один сбой сервиса не приводил к падению всего приложения.
6. **Мониторинг и трассировка (Monitoring & Tracing):** Сбор информации о времени запроса, статусах ответов и возможность слежения за одним запросом по всей цепочке микросервисов (часто с помощью заголовков корреляции).
